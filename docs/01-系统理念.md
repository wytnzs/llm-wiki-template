# LLM Wiki 知识资产生产系统——核心理念

## 这不是什么

- **不是笔记本**——不是为了记下来然后忘掉
- **不是数据库**——不是为了精确检索每一条原始数据
- **不是RAG 知识库**——不是为了给 AI 当外部检索器
- **不是个人 wiki**——不是为了静态展示知识

## 这是什么

**一条知识资产生产线。**

```text
输入 → 精炼 → 连接 → 产出 → 复盘 → 反馈
```

把碎片信息加工成可复用的知识资产，再通过内容生产输出价值，最终反馈回系统形成闭环。

## 三条思想源头

### 1. LLM Wiki — 生产线骨架

- 知识是**加工出来**的，不是收集来的
- 每个环节有明确分工：收件箱只收不加工，知识卡片只解释不存事实，判例库只存事实不写情绪
- 目录结构就是生产线布局

### 2. GBrain — 长期记忆系统

- AI 的记忆不能靠会话上下文，要靠结构化文件
- Markdown 本身是 AI 友好的存储格式
- 分类存储 + YAML 元数据 = AI 可扫描的知识索引
- 详见 `06-GBrain思想吸收说明.md`

### 3. OKF — 标准格式

- 每份资产有固定格式：YAML 头部 + Markdown 正文 + 结构化引用
- 格式统一，AI 和人类都能高效阅读
- 详见 `07-Gemini-OKF-like格式说明.md`

## 技术栈

| 工具 | 用途 |
|------|------|
| **Markdown** | 知识资产唯一存储格式 |
| **Obsidian** | 人类读写界面（双向链接、图谱、标签） |
| **Claude Code** | AI 读写执行界面（知识管理、内容生产自动化） |
| **Git** | 版本管理、回滚、备份 |
| **YAML** | 元数据标准化 |

**不要什么：**

- ❌ 不需要数据库——Markdown 就是存储
- ❌ 不需要 RAG 系统——AI 直接读写文件
- ❌ 不需要 MCP server——CLI + 文件系统就够了
- ❌ 不需要向量库——目录结构和标签就是索引
- ❌ 不需要 Web UI——Obsidian + Claude Code 双界面

**个人使用独白场景下，一台电脑 + 一个 Obsidian + 一个 Claude Code 就够了。**

## 核心原则

### 一张卡片 = 一个概念

每张知识卡片只解释一个概念。概念是知识的最小可复用单元。

- 好的卡片：**"健康告知"**——解释告知义务的范围、判断标准、法律依据
- 不好的卡片：**"张三健康告知纠纷案"**——这是判例，不是概念

### 知识卡片回答"这是什么"，不回答"哪个案子"

- 知识卡片 = **解释层**：定义、判断维度、方法论
- 判例库 = **证据层**：事实、裁判观点、法律依据
- 两者互补，但不混淆

### 主题聚合回答"关于 X，我现在相信什么"

主题聚合（Theme Map）不是目录，而是一个专题的**观点陈述**：

```
主题：关于"带病投保"的法律风险
├── 我认为：2年内被发现，保险公司有权解除合同
├── 我认为：2年后不可抗辩条款保护消费者
├── 但我还认为：不可抗辩不保护恶意欺诈
└── 所以我的建议是：如实告知是底线
```

这是内容生产的认知基站——先知道信什么，再写什么。

### 产出反哺系统

内容生产不是终点，生产中的新认知、新角度、新案例应该回流到知识库：

```text
写文章 → 发现新角度 → 更新卡片 / 新建卡片 → 更新知识地图
写文章 → 遇到新判例 → 更新判例库 → 下次调用
写文章 → 验证/修正已有判断 → 更新主题聚合
```

**系统不为存储而存在，它为生产而存在。**
