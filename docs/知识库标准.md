# 最小知识库标准

## 1. 标准解决什么问题

本标准让人和 Claude Code 可以共同理解同一批知识文件，同时保留用户调整目录、字段和工作流的自由。

标准只规定三件事：入口、文件格式和处理原则。

## 2. 唯一必需入口

根目录 `README.md` 是唯一必需入口。它说明：

- 知识库目的；
- 主要区域分别做什么；
- 内容如何进入、整理、使用和复盘；
- 已启用哪些可选模块。

不强制创建 `index.md`。当某个目录确实需要逐层导航时，可以临时生成，但它不是核心组件。

知识库地图也不是通用必需文件。文件较多、需要统计和健康检查时再启用。

## 3. 目录是建议，不是约束

推荐用以下角色组织内容：

| 角色 | 作用 | 推荐目录 |
|---|---|---|
| 输入 | 尚未处理的材料 | `00-Inbox/` |
| 项目 | 当前要完成的事情 | `01-Projects/` |
| 资产 | 可以长期复用的知识 | `02-Areas/` |
| 资源 | 外部参考材料 | `03-Resources/` |
| 归档 | 已完成或暂时不用 | `04-Archive/` |

用户可以改名、合并或新增目录。README 中写清楚角色映射即可。

## 4. 最小元数据

正式知识文件至少包含：

```yaml
---
title: 标题
type: knowledge_card
---
```

推荐字段：

```yaml
description: 一句话说明文件解决什么问题
status: active
tags: []
source: 来源
created: YYYY-MM-DD
updated: YYYY-MM-DD
```

规则：

- `title` 和 `type` 必需；
- `description` 对入口、模板、主题地图和对外交换文件优先补充；
- 其他字段按任务需要添加；
- 消费端应容忍不认识的扩展字段；
- 使用 UTF-8 Markdown；
- 新建可迁移链接时优先使用标准 Markdown 链接，已有 Obsidian 双向链接可以保留。

## 5. 内容处理原则

```text
输入 -> 整理 -> 连接 -> 使用 -> 复盘
```

- 原始材料与加工后的知识分开；
- 知识文件应能独立说明一个概念、判断或方法；
- 事实、观点和操作流程不要混成一份文件；
- 输出失败时追溯上游知识、输入或 Skill，而不是只修改一次结果；
- 不强制每份知识都进入内容生产。

## 6. 可选模块

| 模块 | 启用条件 |
|---|---|
| 知识库地图 | 文件较多，需要统计、健康检查或项目概览 |
| 案例库 | 工作依赖大量独立案例或证据 |
| 选题库 | 用户明确需要持续内容生产和选题管理 |

安装前必须询问用户。未启用模块时，Skill 跳过相应目录和检查。

## 7. 文件说明规则

- 系统或项目使用 `README.md` 说明；
- 单个知识文件通过标题、元数据和正文自我说明；
- Skill 使用 `SKILL.md`；
- 不要求每个小文件夹都有 README 或 index；
- 只有打开目录后无法判断用途时，才增加说明文件。

## 8. 关系和索引

- 标签负责稳定分类；
- 双向链接表达两份具体内容之间的明确关系；
- 主题页保存围绕一个持续问题的当前判断、证据和边界；
- 聚合是筛选资料的动作，不单独形成知识类型；
- 知识库地图只记录动态状态，不替代主题页和检索入口；
- 日常优先维护来源、主题问题和真实使用项目，不要求填满所有关系字段。

详细准入标准见[知识关系与索引规则](知识关系与索引规则.md)。

## 9. AI 和公开边界

- 原始材料默认只读；
- 用户判断、外部资料和案例事实分开记录；
- 来源不确定时保持空缺或标记待核验，不编造来源；
- 删除、覆盖、批量迁移和高风险结论必须经过确认；
- 公开仓库只包含规则、模板、示例、Skill 和通用脚本，不包含真实业务或个人数据。

完整说明见[知识库运作系统与协作边界](知识库运作系统与协作边界.md)。
