# 最小知识库标准

## 1. 标准解决什么问题

本标准让人和 Claude Code 可以共同理解同一批知识文件，同时保留用户调整目录、字段和工作流的自由。

标准只规定三件事：入口、文件格式和处理原则。

## 2. 唯一必需入口

根目录 `README.md` 是唯一必需入口。它说明：

- 知识库目的；
- 主要区域分别做什么；
- 内容如何进入、整理、使用和复盘；
- 已启用哪些可选模块。

不强制创建 `index.md`。当某个目录确实需要逐层导航时，可以临时生成，但它不是核心组件。

知识库地图也不是通用必需文件。文件较多、需要统计和健康检查时再启用。

## 3. 目录是建议，不是约束

推荐用以下角色组织内容：

| 角色 | 作用 | 推荐目录 |
|---|---|---|
| 输入 | 尚未处理的材料 | `00-Inbox/` |
| 项目 | 当前要完成的事情 | `01-Projects/` |
| 资产 | 可以长期复用的知识 | `02-Areas/` |
| 资源 | 外部参考材料 | `03-Resources/` |
| 归档 | 已完成或暂时不用 | `04-Archive/` |

用户可以改名、合并或新增目录。README 中写清楚角色映射即可。

目录只表达内容角色，不承担全部来源、主题和项目关系。不要因为一份知识涉及多个主题就复制文件或新增一级目录；保留一个稳定位置，通过字段和链接调用。

## 4. 最小元数据

正式知识文件至少包含：

```yaml
---
title: 标题
type: knowledge_card
---
```

推荐字段：

```yaml
description: 一句话说明文件解决什么问题
status: active
tags: []
source: 来源
created: YYYY-MM-DD
updated: YYYY-MM-DD
```

规则：

- `title` 和 `type` 必需；
- `description` 对入口、模板、主题地图和对外交换文件优先补充；
- 其他字段按任务需要添加；
- 消费端应容忍不认识的扩展字段；
- 使用 UTF-8 Markdown；
- 新建可迁移链接时优先使用标准 Markdown 链接，已有 Obsidian 双向链接可以保留。

## 5. 内容处理原则

```text
输入 -> 整理 -> 连接 -> 使用 -> 复盘
```

- 原始材料与加工后的知识分开；
- 知识文件应能独立说明一个概念、判断或方法；
- 事实、观点和操作流程不要混成一份文件；
- 输出失败时追溯上游知识、输入或 Skill，而不是只修改一次结果；
- 不强制每份知识都进入内容生产。

### 原始材料流转

```text
输入 -> 原始资料收容 -> 知识提炼 -> 主题综合 -> 项目使用 -> 反馈
```

- `00-Inbox/` 接收尚未处理的输入；
- 处理后的外部原文进入资源或归档区域，正文保持不变；
- 知识卡片保存可复用的概念、判断和方法，并记录可核验来源；
- 主题页综合多份卡片、案例和原始证据，不用单份材料证明普遍结论。

### 主题中心

- 主题中心统一承接自然语言索引和主题页入口；
- 主题是需要持续回答的问题；
- 聚合是围绕问题筛选资料的动作，不单独形成目录或知识类型；
- 主题页保存当前综合判断、证据、方法、反例和边界；
- 知识库地图只负责展示系统动态状态。

默认以 `02-Areas/主题中心/README.md` 作为唯一入口，以 `02-Areas/主题中心/主题页/` 保存独立主题页。主题页中的 `core_question` 必须是可以回答的具体问题，不能只写一个领域名称。

### 最小关系

知识资产日常只优先维护：

1. `source`：从哪里来；
2. `related_theme_maps`：直接帮助回答哪个主题问题；
3. `related_projects`：在哪个项目中真实使用。

`supports`、`contrasts`、`examples` 用于明确的支持、反例和例子关系。只是同领域、关键词相同或以后可能使用时，保留标签即可，不建立主题关系。

已有知识库中的旧关系字段可以保留兼容，不做机械改名。批量添加或清理关系前必须先干跑、抽查正文并验证链接和双向一致性。

### AI 修改边界

- 原始材料只读，AI 可以分类和建立来源链接，不改写正文；
- 案例事实、合同条款、病历以及法律、医疗等关键结论，先进入待确认队列；
- 知识卡片和主题页可以由 AI 起草，但改变核心判断时需要人工确认；
- 模板、索引、状态统计、断链检查和变更日志可以按规则自动维护。

## 6. 可选模块

| 模块 | 启用条件 |
|---|---|
| 知识库地图 | 文件较多，需要统计、健康检查或项目概览 |
| 案例库 | 工作依赖大量独立案例或证据 |
| 选题库 | 用户明确需要持续内容生产和选题管理 |

安装前必须询问用户。未启用模块时，Skill 跳过相应目录和检查。

## 7. 文件说明规则

- 系统或项目使用 `README.md` 说明；
- 单个知识文件通过标题、元数据和正文自我说明；
- Skill 使用 `SKILL.md`；
- 不要求每个小文件夹都有 README 或 index；
- 只有打开目录后无法判断用途时，才增加说明文件。
