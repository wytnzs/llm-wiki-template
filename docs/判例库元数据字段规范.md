# 知识库与判例库优化建议

> 适用范围：知识库根目录，重点覆盖 `02-Areas/判例库/`、`02-Areas/知识卡片/`、`00-Inbox/`、`01-Projects/`。  
> 目标：降低维护成本，避免索引漂移，让知识库更稳定地服务判例检索、内容生产、业务复用和 AI 工作流。

---

## 一、总体判断

当前知识库已经不是普通资料夹，而是一套正在运转的工作系统。

它的主线大致是：

```text
00-Inbox / 02-Areas / 03-Resources
        ↓
02-Areas/知识卡片 知识卡片
        ↓
判例库 / 方法论 / 个人IP
        ↓
01-Projects 自媒体运营
        ↓
复盘沉淀，再反哺 02-Areas/知识卡片 和判例库
```

最有价值的地方不在于资料数量，而在于已经形成了“输入 - 提炼 - 输出 - 反哺”的闭环。

后续优化重点不是继续盲目增加资料，而是解决四类问题：

1. 索引和说明文件容易过期。
2. Inbox 原始文件和已处理文件存在重复。
3. 判例库字段还不够统一，影响自动检索和专题汇总。
4. 内容产出后的反馈数据还没有稳定反哺到知识库。

---

## 二、优化优先级总览

| 优先级 | 事项 | 核心目标 |
|------|------|---------|
| 第一优先级 | 解决文档漂移 | 让知识地图、判例索引、统计数据尽量自动生成或半自动生成 |
| 第二优先级 | 重整 Inbox 结构 | 区分原始输入、处理中材料、已处理归档，避免重复干扰 |
| 第三优先级 | 统一判例库元数据字段 | 让每个判例可被稳定检索、筛选、聚类和复用 |
| 第四优先级 | 拆分 02-Areas/知识卡片 稳定层与生长层 | 避免知识卡片越写越散，明确哪些可直接调用 |
| 第五优先级 | 增加内容复盘字段 | 让公众号、小红书、短视频的真实反馈反哺知识库 |

---

## 三、第一优先级：解决文档漂移

### 1. 当前问题

知识库里已经出现轻微“文档漂移”：

- `知识库地图.md` 里的文件数量和实际扫描结果存在差异。
- `判例库/README.md` 的统计仍是旧状态，例如“当前覆盖类型数：0”“当前判例总数：0”。
- `判例库元数据.md`、`类型索引.md`、`知识库地图.md` 都在承担索引功能，但同步方式不完全统一。

这类问题现在不严重，但随着内容继续增长，会越来越影响 AI 调用和人工判断。

### 2. 建议规则

全库索引类文件分成三类：

| 文件类型 | 作用 | 维护方式 |
|---------|------|---------|
| 地图类 | 全库导航，如 `知识库地图.md` | 尽量自动扫描生成 |
| 元数据类 | 规则、字段、统计口径，如 `判例库元数据.md` | 人工维护规则，统计尽量自动生成 |
| 索引类 | 按类型、年份、标签列出条目，如 `类型索引.md` | 从 YAML 或目录自动汇总 |

### 3. 推荐改法

保留这些文件，但明确分工：

```text
知识库地图.md
  负责全库入口、目录解释、最近更新摘要。

判例库/README.md
  负责说明判例库结构、使用方式、四件套规则。

判例库/判例库元数据.md
  负责版本、字段标准、统计口径、全库统计。

判例库/类型索引.md
  负责按争议类型自动汇总判例列表。
```

### 4. 执行建议

后续说“更新知识地图”时，建议同步做三件事：

1. 扫描全库文件数量、更新时间、大小。
2. 扫描判例库 YAML，刷新判例数量、产品线数量、争议类型数量。
3. 检查 `README.md`、`类型索引.md`、`判例库元数据.md` 是否和扫描结果一致。

## 四、第二优先级：重整 Inbox 结构

### 1. 当前问题

`00-Inbox` 中同时存在原始导入和已处理归档，例如：

```text
00-Inbox/得到大脑/...
00-Inbox/已处理/得到大脑_20260703/...
```

这会造成两个问题：

- 同一批材料被重复统计。
- AI 检索时可能同时读到原始版和处理版，增加噪音。

### 2. 推荐目录结构

建议把 `00-Inbox` 固定成三层：

```text
00-Inbox/
  raw/          原始输入，只进不改
  processing/   正在处理的材料
  processed/    已处理完成，可追溯来源
  archive/      长期归档，不参与日常检索
```

如果希望继续保留中文目录，也可以用：

```text
00-Inbox/
  00-原始输入/
  01-处理中/
  02-已处理/
  99-归档/
```

### 3. 操作规则

| 状态 | 放哪里 | 是否参与提炼 |
|------|------|-------------|
| 新导入资料 | `raw/` | 是 |
| 正在拆解的资料 | `processing/` | 是 |
| 已提炼成卡片或项目素材 | `processed/` | 一般否，只作追溯 |
| 大体量旧资料或重复来源 | `archive/` | 默认否 |

### 4. 关键原则

**Inbox 只负责承接输入，不负责长期保存价值内容。**

真正可复用的内容应流向：

- `02-Areas/知识卡片/`：概念、方法论、判断标准。
- `判例库/`：结构化判例。
- `01-Projects/`：项目产出和工作流材料。
- `03-Resources/`：外部参考资料归档。

## 五、第三优先级：统一判例库元数据字段

### 1. 核心原则

判例库字段应遵循一个基本原则：

**具体判例文件是唯一事实源；索引文件只负责汇总，说明文件只负责定义规则。**

也就是说：

- 单个判例的事实、标签、判断维度，放在该判例文件顶部的 YAML 元数据区。
- `类型索引.md` 只读取判例元数据后做汇总，不手工维护事实细节。
- `判例库元数据.md` 只维护字段规范、统计口径、版本记录和全库概览。
- `概览.md`、`法条依据.md`、`维权路径.md` 属于类型层总结，不放单案结构化字段。

### 2. 字段放在哪里

字段应放在每一个具体判例文件的开头 YAML 区。

推荐路径结构：

```text
判例库/
  01-重疾险/
    01-未如实告知拒赔/
      判例/
        某某判例.md   ← 字段放在这里
```

示例：

```markdown
---
案号: (2025)浙0402民初14539号
产品线: 重疾险
争议类型: 未如实告知拒赔
判决结果: 胜诉
---

## 一句话梗概
...
```

### 3. 三层文件分工

#### 单个判例文件：完整事实源

每个判例文件顶部 YAML 应保存该案的结构化信息，包括产品线、争议类型、结果、核心法条、攻击角度、证据、关键词等。

这是后续所有检索、统计、自动索引、专题文章生成的唯一可信来源。

#### `类型索引.md`：自动汇总层

`类型索引.md` 适合按争议类型、产品线、判决结果、年份等维度生成列表。

它不应该承担事实源功能，避免出现“判例文件写一套、索引文件写一套”的漂移问题。

#### `判例库元数据.md`：规则与统计层

`判例库元数据.md` 适合维护：

- 当前版本
- 字段标准
- 产品线统计
- 争议类型统计
- 最近更新日志
- 未完成录入清单
- 字段取值规范

```markdown
争议类型:

审理法院:

  -

```

### 5. 字段说明

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| `案号` | 必填 | 法院案号；没有案号时可写“待确认-法院/来源”。 |
| `产品线` | 必填 | 如：重疾险、医疗险、意外险、寿险、责任保险、待分类。 |
| `争议类型` | 必填 | 如：未如实告知拒赔、格式条款免责争议、等待期出险争议。 |
| `产品` | 建议填 | 涉案产品名称，能具体则具体。 |
| `保险公司` | 建议填 | 涉案保险公司名称。 |
| `审理法院` | 必填 | 具体法院名称。 |
| `审级` | 必填 | 一审、二审、再审、调解等。 |
| `判决年份` | 必填 | 用四位年份，如 2025。 |
| `判决结果` | 必填 | 建议使用统一取值：胜诉、败诉、部分胜诉、调解、驳回、维持、改判。 |
| `核心法条` | 必填 | 与裁判逻辑直接相关的法律条文。 |
| `攻击角度` | 必填 | 用户/被保险人一方可使用的维权切入点。 |
| `关键证据` | 必填 | 支撑胜败判断的核心证据。 |
| `保险公司抗辩要点` | 建议填 | 保险公司的主要拒赔理由或诉讼抗辩。 |
| `是否互联网投保` | 建议填 | 统一写：是、否、不明。 |
| `是否涉及提示说明义务` | 建议填 | 统一写：是、否、不明。 |
| `是否涉及因果关系` | 建议填 | 统一写：是、否、不明。 |
| `是否涉及30日解除权` | 建议填 | 统一写：是、否、不明。 |
| `是否涉及两年不可抗辩` | 建议填 | 统一写：是、否、不明。 |
| `是否涉及格式条款` | 建议填 | 统一写：是、否、不明。 |
| `是否涉及举证责任` | 建议填 | 统一写：是、否、不明。 |
| `是否改判` | 建议填 | 统一写：是、否、N/A、不明。 |
| `典型意义` | 必填 | 该案最值得复用的裁判规则或实务启发。 |
| `关键词` | 必填 | 便于检索和专题聚合。 |
| `录入日期` | 必填 | 建议格式：YYYY-MM-DD。 |
| `来源/链接` | 必填 | 裁判文书网、法院公众号、课程、书籍等。 |

### 6. 统一取值建议

#### 产品线

```text
重疾险
医疗险
意外险
寿险
责任保险
待分类
```

#### 判决结果

```text
胜诉
败诉
部分胜诉
调解
驳回
维持
改判
```

如果一个案件同时包含多个结果，可写成：

```yaml
判决结果: 二审改判胜诉
```

但建议在后续统计时增加一个标准化字段，例如：

```yaml
结果分类: 胜诉
```

#### 布尔类字段

统一使用：

```text
是
否
不明
```

不建议混用 `true/false`、`有/无`、`涉及/不涉及`，否则后续自动统计会变麻烦。

### 8. 为什么不放在其他文件里

#### 不放在 `概览.md`

`概览.md` 是某一类争议的整体说明，适合写定义、法院认定标准、维权趋势，不适合放单个案件字段。

#### 不放在 `法条依据.md`

`法条依据.md` 是法条和裁判规则层，不负责承载单案事实。

#### 不放在 `维权路径.md`

`维权路径.md` 是策略层，适合总结攻击角度和证据准备，不适合保存每个案件的结构化信息。

#### 不手工放在 `类型索引.md`

`类型索引.md` 应该从判例文件自动汇总。手工维护会导致索引和原始判例不一致。

## 六、第四优先级：拆分 02-Areas/知识卡片 稳定层与生长层

### 1. 当前问题

`02-Areas/知识卡片` 已经比较成熟，包含保险、认知、创作、个人IP、方法论等模块。

问题在于，随着卡片继续增加，容易出现三种情况：

- 已成熟卡片和试写卡片混在一起。
- AI 调用时不知道哪些卡片是稳定规则，哪些只是临时想法。
- 同一概念可能被不同角度重复写入，后期合并成本变高。

### 2. 推荐分层

建议给 `02-Areas/知识卡片` 增加 `status` 字段，而不是立刻大规模移动目录。

每张卡片 YAML 增加：

```yaml
status: stable
```

或：

```yaml
status: growing
```

推荐取值：

```text
seed
growing
stable
merge_candidate
needs_verification
archived
```

### 3. 各状态含义

| status | 含义 | AI 调用方式 |
|------|------|------------|
| stable | 已验证、可直接作为工作流规则 | 优先调用 |
| growing | 有价值但尚未完全定型 | 可参考，需谨慎 |
| merge_candidate | 与其他卡片重复或相近 | 调用前先查重 |
| needs_verification | 来源或结论需要再确认 | 不作为强规则 |
| archived | 暂时不用，但保留历史价值 | 默认不调用 |

### 4. 推荐卡片模板

```markdown
---
title: 概念名
type: knowledge_card
asset_type: concept / cognition / principle / method / action / evidence / expression
domain:
status: stable
tags: [知识卡片, 领域标签]
related_topics:
  - [[关联卡片]]
related_projects: []
links:
  supports: []
  contrasts: []
  examples: []
  applies_to: []
  derived_methods: []
---

## 定义

## 关键要点

## 判断维度

## 案例

## 来源

## 可调用场景
```

### 5. 执行建议

优先给以下卡片标记为 `稳定`：

- 个人IP 4 张卡片
- 保险维权系列卡片
- 内容生产核心方法卡片
- 直接被 Skill 调用的卡片

其他读书笔记、临时感悟、新提炼概念，可以先标为 `生长中`。

## 七、第五优先级：增加内容复盘字段

### 1. 当前问题

`01-Projects/自媒体运营/` 已经承担公众号、小红书、短视频等内容产出，但产出后的反馈信息还可以更结构化。

如果只保存文章本身，知识库只能回答“写了什么”。

如果增加复盘字段，知识库就能回答：

- 为什么这篇有效？
- 哪个钩子有效？
- 哪类判例更容易转化为内容？
- 哪类选题更适合公众号、小红书或短视频？
- 哪些卡片真正被高频调用？

### 2. 推荐在内容文件顶部增加复盘 YAML

适用于公众号、小红书、短视频脚本等内容文件。

```yaml
---
标题: 
平台: 公众号
内容类型: 案例文
发布状态: 草稿
发布日期: 
选题来源: 
调用卡片:
  - 
调用判例:
  - 
目标人群: 
核心钩子: 
核心观点: 
CTA: 
反馈数据:
  阅读: 
  点赞: 
  收藏: 
  转发: 
  评论: 
  私信: 
复盘结论: 
可沉淀卡片:
  - 
---
```

### 3. 字段说明

| 字段 | 说明 |
|------|------|
| `平台` | 公众号、小红书、短视频、朋友圈、直播等。 |
| `内容类型` | 案例文、方法论、观点文、产品解释、选题测试等。 |
| `发布状态` | 草稿、已发布、废弃、待改。 |
| `选题来源` | 来自判例、客户问题、课程、读书笔记、热点等。 |
| `调用卡片` | 本文使用了哪些 `02-Areas/知识卡片` 卡片。 |
| `调用判例` | 本文使用了哪些判例文件。 |
| `目标人群` | 代理人、客户、内勤、新人、老代理人等。 |
| `核心钩子` | 开头吸引用户继续看的关键句或冲突。 |
| `核心观点` | 文章真正想让读者记住的话。 |
| `CTA` | 最后的行动引导。 |
| `反馈数据` | 发布后的客观数据。 |
| `复盘结论` | 这篇内容为什么有效或无效。 |
| `可沉淀卡片` | 是否应反哺到 `02-Areas/知识卡片`。 |

### 4. 内容反哺规则

每篇已发布内容复盘后，判断是否需要反哺：

```text
如果只是表达优化 → 更新风格规律或项目日志
如果出现新判断标准 → 写入 02-Areas/知识卡片
如果出现新案例规律 → 补充判例库或维权路径
如果出现新选题规律 → 补充选题方法论
如果出现新转化反馈 → 补充内容复盘或成交漏斗
```

## 八、建议的执行顺序

不要一次性重构全库，建议按低风险顺序推进。

### 第一步：修正索引漂移

- 更新 `判例库/README.md` 的过期统计。
- 明确 `知识库地图.md` 的统计来源。
- 检查 `类型索引.md` 和 `判例库元数据.md` 是否冲突。

### 第二步：给新判例执行新 YAML 标准

不要一开始批量改 280 个旧判例。

先规定：从下一条新判例开始，必须使用新版 YAML 字段。

### 第三步：抽样补齐旧判例

先选 20 个高价值判例补齐字段：

- 未如实告知拒赔
- 格式条款免责争议
- 重疾定义争议
- 互联网投保争议
- 雇主责任险争议

### 第四步：清理 Inbox 重复源

先不删除资料，只移动或标记。

建议先处理：

```text
00-Inbox/得到大脑
00-Inbox/已处理/得到大脑_20260703
```

确认哪一份作为原始源，哪一份作为处理后归档。

### 第五步：给 02-Areas/知识卡片 增加状态字段

先给被 Skill 高频调用的卡片加 `status`，不必全库一次性完成。

### 第六步：给新内容增加复盘字段

从下一篇公众号或小红书开始，在文件顶部加入复盘 YAML。

这套优化完成后，知识库应达到以下状态：
