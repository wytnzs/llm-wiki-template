# 数据库与内容管线 Skill 关系设计

> 适用范围：`02-Areas/判例库/`、`02-Areas/知识卡片/`、`02-Areas/知识卡片/个人IP/`、`.claude/skills/` 下的内容管线 skill。
> 核心目标：让数据库负责沉淀事实和认知资产，让内容管线 skill 负责生产内容，中间通过"内容素材包"完成交接。

---

## 一、核心判断

数据库和内容管线 skill 不应该互相替代。

最合理的关系是：

```text
判例库 / 知识卡片 / 个人IP卡片
         → 内容素材包 (content-brief)
         → gzh-pipeline / xhs-pipeline / short-video-script
         → 公众号 / 小红书 / 短视频
         → 效果复盘 / 新卡片 / 判例补充
```

一句话：
**数据库是事实和认知资产，内容管线 skill 是生产线，中间需要一份内容素材包作为交接单。**

---

## 二、各层分工

### 1. 判例库：证据层

判例库负责保存单个案件的事实、裁判观点、攻击角度、关键证据。

它不负责写文章，也不负责选题包装。

内容管线调用判例库时，只拿三类东西：
- 这个案子讲了什么。
- 它能证明什么。
- 它适合支撑哪个内容观点。

判例库越冷静、越结构化，越有价值。

### 2. 知识卡片：解释层

`02-Areas/知识卡片/` 负责提供解释框架和判断标准。

比如：
- `拒赔六大维度`
- `未如实告知拒赔维权`
- `格式条款免责争议维权`
- `五层抽象金字塔`
- `爆款内容传播框架`

判例库给事实，知识卡片给解释框架。

### 3. 个人IP卡片：表达约束层

`02-Areas/知识卡片/个人IP/` 不负责提供事实，而是决定表达方式。

它回答这些问题：
- 这件事用什么立场讲？
- 哪些话不能说？
- 语气是冷静、简洁，还是更有情绪？
- 面向哪个具体的人？
- CTA 应该自然到什么程度？

也就是说，个人IP卡片负责"怎么像这个人一样说话"。

### 4. 内容管线 skill：生产层

`gzh-pipeline`、`xhs-pipeline`、`short-video-script` 负责把素材加工成平台内容。

它们不应该自己变成资料库。

内容管线 skill 的职责是：

```text
检索相关材料 → 生成内容素材包 → 写作/改写/排版/配图
→ 记录调用了什么 → 发布后反哺知识库
```

---

## 三、为什么需要内容素材包

如果内容管线 skill 直接面对整个数据库，会出现几个问题：

- 每次都要全库搜索，成本高。
- AI 容易拿错材料，或者把不相关卡片混进来。
- 文章用了哪些判例、哪些卡片，事后难追踪。
- 判例结论容易被放大成普遍规则。
- 公众号、小红书、短视频之间重复理解同一批素材。

所以中间应该增加一层：

```text
.claude/drafts/content-brief.json
```

它是数据库和内容管线之间的"交接单"。

> content-brief.json 是唯一核心生产交接单。不再使用 delivery-brief.json。

## 五、典型工作流

### 1. 写公众号

```text
判例库 + 知识卡片 + 个人IP卡片
→ 生成 content-brief
→ gzh-pipeline
→ 公众号文章
→ 发布后复盘 → 更新知识卡片 / 判例标签 / 风格规律
```

公众号适合承载完整逻辑。它应该优先使用：
- 判例事实
- 裁判观点
- 维权路径
- 多个卡片共同支撑的深度解释

### 2. 转小红书

```text
公众号文章 + content-brief
→ xhs-pipeline
→ 小红书轮播图 + 发布文案
```

小红书不应该重新理解全部判例。它应该从 `content-brief` 中抽取：
- 一个核心观点
- 三个可视化要素
- 一两个最有冲突感的案例细节
- 明确的评论区互动 CTA

### 3. 写短视频

```text
content-brief
→ short-video-script
→ 口播脚本
```

短视频不要承载太多判例。它适合只讲一个点：
- 一个反常识结论
- 一个真实场景
- 一个代理人能立刻用的动作

### 4. 选题挖掘

```text
topic-selection
→ 扫描 知识卡片、判例库、近期内容复盘
→ 生成候选选题
→ 选定后生成 content-brief
→ 进入对应平台管线
```

选题 skill 不应该直接写稿。它负责发现"哪些知识资产值得被内容化"。

## 六、内容管线调用数据库的规则

### 1. 先定问题，再查资料

不要先全库乱搜。

推荐顺序：
```text
确定目标人群和核心问题 → 查知识卡片找解释框架
→ 查判例库找支撑案例
→ 查个人IP卡片确定表达边界
→ 写入 content-brief
```

### 2. 判例只作支撑，不作万能结论

每个判例进入内容前，都要回答：
- 这个案子的适用边界是什么？
- 它支持的是哪个具体观点？
- 有没有可能被读者误解成"所有案件都这样"？

如果有误解风险，必须写入 `forbidden`。

### 3. 卡片不是用来引用的，是用来重构素材的

内容里不一定要出现卡片名称。真正重要的是：这张卡片有没有改变文章的分析方式。

比如：
- 用 `五层抽象金字塔` 从判例中提炼方法。
- 用 `拒赔六大维度` 给案例定位。
- 用 `注意力是最宝贵的财富` 设计开头。

### 4. 每篇内容都要记录调用来源

文章文件顶部建议增加：

```yaml
调用卡片:
  - 02-Areas/知识卡片/保险/未如实告知拒赔维权.md
调用判例:
  - 02-Areas/判例库/按险种/01-重疾险/01-未如实告知拒赔/判例/xxx.md
内容素材包: .claude/drafts/content-brief.json
```

这样后续才能知道哪些知识资产真的在生产内容价值。

---

## 七、发布后的反哺规则

内容发布后，不只是记录阅读量。要判断它反哺到哪里：

| 复盘发现 | 回写位置 |
|---------|---------|
| 出现新的判断标准 | `02-Areas/知识卡片/` |
| 某个判例特别适合传播 | 判例 YAML 增加内容标签 |
| 某种开头效果好 | `表达资产库.md` |
| 某个选题树表现好 | 选题卡或选题复盘 |
| 某个平台转化好 | 对应平台管线经验 |
| 读者提出新问题 | `00-Inbox/` 或新选题池 |

## 八、不要做的事

### 1. 不要把判例库写成文章库

判例库应该保持结构化、冷静、可检索。情绪、标题、传播包装，不应该写进判例文件本体。

### 2. 不要让 skill 变成资料库

Skill 里可以写流程、规则、检查清单。不要在 skill 里长期堆判例、观点、素材。否则后面会出现两套知识源：

```text
数据库里一套，skill 里又一套。

这会让你维护成本变高。
```

### 3. 不要跳过内容素材包

直接从数据库到文章，看似快，但长期会造成不可追溯。`content-brief` 是控制质量和减少幻觉的关键。

### 4. 不要把个案结论写成普遍承诺

尤其是保险理赔和诉讼内容，必须保留边界。

可以这样说：
```text
这个案子说明，保险公司的某种拒赔理由并非当然成立。
```

不要这样说：
```text
遇到这种情况一定能赔。
```

---

## 九、最终关系

最终这套系统应该变成：

```text
数据库负责可靠
知识卡片负责解释
IP负责边界和口吻
content-brief 负责交接
skill 负责生产
复盘负责进化
```

一句话总规则：

**判例库不要被写成文章库，skill 不要变成资料库；中间由 content-brief 把"讲什么"和"怎么讲"分开。**
