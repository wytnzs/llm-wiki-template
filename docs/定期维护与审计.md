# 定期维护与审计

## 1. 维护目标

定期维护负责发现、分流和记录问题，不以“所有指标变绿”为目标。不能用猜测来源、虚假双链、空卡片或批量改写原始资料消灭告警。

| 频率 | 目的 | 默认动作 |
|---|---|---|
| 每周 | 发现新故障 | 乱码、链接、新文件元数据和 Inbox 时效检查 |
| 每两周 | 完成维护闭环 | 来源复核、主题证据抽查、Inbox 路由、地图和日志更新 |
| 每月 | 检查是否真的有用 | 抽查检索下钻、项目调用、长期未用资产和公开边界 |

批量建链、目录调整、模板升级、Skill 改动或大量导入后，应立即运行一次完整检查。

## 2. 执行前边界

1. 先读 README 和本次任务对应的 Skill；
2. 查看工作区已有修改，不回滚或覆盖其他协作者的内容；
3. 不让两个 Agent 同时修改同一成品；
4. 原始输入和03来源层正文默认只读；已有04被标记为退役时，只检查其是否仍仅保留说明；
5. 定期维护不自动删除、提交、推送或对外发送；
6. 来源冲突、隐私、高风险结论和主题核心判断变化进入人工确认。

## 3. 每周快速检查

公开模板仓库可运行：

```powershell
./scripts/check-mojibake.ps1
./scripts/check-links.ps1
```

个人知识库还应检查：

- 新增正式知识是否包含 `title` 和 `type`；
- `tags` 和关系字段是否保持列表形态；
- 最近使用的路径和双链是否能找到唯一目标；
- Inbox 是否有超过两周仍未路由的内容；
- 是否出现没有依据却被标为稳定结论的卡片。

没有结构变化时，不为了留下维护痕迹改写地图和日志。

## 4. 每两周完整维护

### 来源和状态

- 回查缺少 `source` 的知识卡片；
- 找不到来源时保持空缺，并让 `status`、`evidence_status` 和正文提示与证据强度一致；
- 未核验的数字、阈值和案例不进入稳定结论或主题强证据。

### 主题页

- 抽查近期实际调用的 1 至 2 张主题页；
- 核心判断必须回答 `core_question`；
- 证据优先使用具体卡片、案例、原始文件、章节或项目记录；
- 目录只能作为导航；
- 新增或移除显式主题关系时保持两侧一致；
- 主题页中的内容方向不能自动升级为正式选题。

### Inbox

- 每批处理不超过 10 条；
- 先给出“继续保留、进入项目、提炼资产、进入来源层（外部资料、个人输入或系统历史）、待清理隔离”的路由建议；
- 未确认时不移动、不覆盖、不删除；
- 图片与对应 Markdown 作为同一组处理，避免附件断链。

### 地图和日志

- 知识库地图只保存当前规模、项目状态、健康问题和 3 至 5 项下一步；
- 变更日志只记录影响结构、规则、正式判断或能力的重要变化；
- README 解释系统，地图展示当前状态，日志保存重要变化，三者不重复维护同一内容。

## 5. 每月有效性抽查

从真实任务中选择 3 个问题，实际走完：

```text
问题 -> 主题页 -> 卡片或案例 -> 原始依据 -> 输出或决策
```

记录：

- 是否命中正确主题；
- 是否能在 1 至 2 跳内找到可用内容；
- 是否能回到来源；
- 是否出现明显误连；
- 最终是否帮助判断或行动。

链接数量、卡片数量和标签数量都不是质量目标。

## 6. GitHub 发布检查

公开模板发布前运行：

```powershell
./scripts/check-mojibake.ps1
./scripts/check-links.ps1
./scripts/check-public-safety.ps1
git diff --check
git diff --cached
```

只逐项暂存规则、模板、示例、Skill 和通用脚本。真实数据与公开代码处于同一工作区时，不使用 `git add -A`。

## 7. 可复用维护提示词

```text
请执行一次知识库定期维护。

先读取根目录 README.md、协作规则和任务匹配的 Skill，并查看工作区已有修改，不回滚或覆盖他人内容。

依次完成：
1. 检查乱码、链接、正式文件最小元数据和关系字段；
2. 复核缺来源卡片与证据状态，不猜来源；
3. 抽查近期使用的主题页，把模糊证据收敛到具体文件；
4. 只列出最多 10 条 Inbox 路由建议，未经确认不移动、不删除；
5. 更新待确认队列、知识库地图和重要变更日志；
6. 重新运行检查并报告验收结果。

请输出：本次发现、已安全修复、需要确认、未处理原因和验证结果。不要自动提交或推送 GitHub。
```

## 8. 完成标准

- 检查结果与实际文件一致，没有为归零制造假数据；
- 原始输入仍可追溯；
- 关键判断能够回到依据；
- 主题关系明确、有用、可追溯；
- 待确认、地图和日志状态一致；
- 未处理事项说明原因、影响和下一步。
