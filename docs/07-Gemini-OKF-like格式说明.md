# OKF（Obsidian KMG File）格式说明

## 什么是 OKF

OKF（Obsidian KMG File）是 Gemini 团队提出的 Obsidian 知识库标准化格式规范。核心思想是：

**让 AI 能像读数据库一样读 Markdown 文件——通过 YAML 头部理解元数据，通过正文理解内容，通过引用理解关系。**

## OKF 的三段式结构

每个 OKF 文件包含三个部分：

```markdown
─── 第一部分 ───
YAML 头部：元数据声明

─── 第二部分 ───
Markdown 正文：人类可读的内容

─── 第三部分 ───
结构化引用：关联和来源
```

### 第一部分：YAML 头部

YAML 头部放在文件最顶部，用 `---` 包裹：

```yaml
---
title: 文件标题
type: 文件类型
tags: [标签1, 标签2]
links: [[关联文件1]], [[关联文件2]]
created: 2026-01-01
updated: 2026-06-15
---
```

YAML 头部的作用：

| 作用 | 说明 |
|------|------|
| **标识** | 让 AI 一眼知道这是什么类型的内容 |
| **检索** | 通过标签快速筛选相关文件 |
| **链接** | 明确声明关联关系（比埋在正文里的链接更好解析） |
| **溯源** | 记录创建和更新时间，判断内容时效性 |

### 第二部分：Markdown 正文

按语义分章节，每章用 `##` 区分。不同文件类型有不同的章节结构（见下文示例）。

### 第三部分：结构化引用

文件末尾统一记录来源：

```markdown
## 来源

- 来源 1
- 来源 2
## 相关文件

- [[文件A]]
- [[文件B]]
```

## 本系统适配的 OKF-like 格式

### 知识卡片（KNOWLEDGE_CARD）

```markdown
---
title: 概念名称
type: KNOWLEDGE_CARD
tags: [领域, 知识卡片, 具体标签A, 具体标签B]
links: [[关联卡片1]], [[关联卡片2]]
created: 2026-01-01
updated: 2026-06-15
source_type: book | article | course | experience
confidence: high | medium | low
---

## 定义

一句话说清楚这个概念是什么。

## 关键要点

- 要点 1
- 要点 2
- 要点 3

## 判断维度

用在什么场景？帮我做什么决策？

## 案例

一个通用示例（非真实数据），说明概念如何应用。

## 来源

- 来源 1
- 来源 2

## 相关卡片

- [[关联卡片1]]
- [[关联卡片2]]
```

**type 必须为 KNOWLEDGE_CARD**，这是文件类型标记，方便 AI 做类型过滤。

### 主题聚合（THEME_MAP）

```markdown
---
title: 关于议题 X 的当前认知
type: THEME_MAP
tags: [主题聚合, 议题X, 标签A]
links: [[相关卡片1]], [[相关卡片2]]
created: 2026-01-01
updated: 2026-06-15
status: 活跃 | 待更新 | 已存档
---

## 当前观点

### 子议题 A
- 我认为：...
- 依据：[[相关卡片]]
- 置信度：高/中/低

### 子议题 B
- 我认为：...
- 依据：[[相关卡片]]
- 置信度：高/中/低

## 待验证

- [ ] 待验证的问题

## 近期变化

- 2026-06-15：修正子议题 A 的观点
- 2026-01-10：新建此主题聚合
```

### 选题卡片（TOPIC_CARD）

```markdown
---
title: 选题：一句话标题
type: TOPIC_CARD
tags: [选题, 领域, 内容形式, 优先级]
links: [[相关卡片1]], [[相关主题聚合]]
created: 2026-01-01
status: 未开始 | 进行中 | 已发布
priority: high | medium | low
---

## 选题描述

一句话描述：这个选题要说什么？

## 目标受众

写给谁看？他们的痛点是什么？

## 核心信息

用 3-5 点列出要传达的核心信息。

## 素材准备

- [ ] 知识卡片：[[卡片A]]、[[卡片B]]
- [ ] 判例 / 案例：[[判例X]]
- [ ] 主题聚合：[[主题聚合Y]]

## 分发建议

| 平台 | 形式 | 要点 |
|------|------|------|
| 公众号 | 长文 | ... |
| 小红书 | 图文 | ... |
| 短视频 | 脚本 | ... |
```

### 判例条目（CASE_ENTRY）

```markdown
---
title: 案件关键词
type: CASE_ENTRY
tags: [判例, 类型, 原因, 领域分类]
links: [[相关卡片1]], [[相关卡片2]]
created: 2026-01-01
---

## 事实摘要

（简述案件事实，不写情绪和立场）

## 争议焦点

- 焦点 1
- 焦点 2

## 裁判观点

- 法院认为：...
- 裁判要旨：...

## 攻击/防御角度

- 从主张方角度：...
- 从相对方角度：...

## 来源

- 案号 / 链接
```

## 为什么 YAML 帮助 AI 扫描和理解

```text
没有 YAML：
  文件里写满了文字，AI 需要从头读到尾才能判断：
  "这个文件讲什么的？"
  "这是知识卡片还是主题聚合？"
  "它和哪些其他文件相关？"

有 YAML：
  title: "健康告知义务"
  type: KNOWLEDGE_CARD
  tags: [保险, 法律, 知识卡片]
  links: [[不可抗辩条款]], [[如实告知]]
  
  AI 一行 YAML 就判断出：
  "这是一个概念卡片，讲健康告知义务的，
   关联到不可抗辩条款和如实告知。"

  然后决定是否还需要继续读正文。
  如果不相关，读完 YAML 就可以跳过。
```

**对 AI 而言，YAML 头部是元数据索引，正文是内容深度，引用是关系网络。三部分各司其职。**

## 迁移策略

### 新内容 → 强制标准

所有新创建的知识资产必须遵守 OKF-like 格式：

```text
✅ 知识卡片必须包含：title + type(KNOWLEDGE_CARD) + tags + links + created
✅ 主题聚合必须包含：title + type(THEME_MAP) + tags + links + created + status
✅ 选题卡片必须包含：title + type(TOPIC_CARD) + tags + links + created + status
✅ 判例条目必须包含：title + type(CASE_ENTRY) + tags + links + created
```

### 旧内容 → 逐步迁移

按优先级逐步为旧文件添加 YAML 头部：

```text
第一优先级：活跃文件（经常被引用的）
  → 经常在内容生产中调用的卡片
  → 经常在主题聚合中链接的判例

第二优先级：索引文件（知识地图、选题总索引）
  → 这些文件是 AI 扫描的第一入口

第三优先级：低频文件（很少调用的）
  → 有机会读到就顺手更新，不主动推动
```

迁移不必追求 100% 覆盖。**核心目标：新生产的资产高标准，被频繁调用的资产持续优化。** 低频文件保持原样也没问题。
