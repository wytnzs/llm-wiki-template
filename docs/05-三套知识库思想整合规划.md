# 三套知识库思想整合规划

本系统整合了三套知识管理思想：

| 思想 | 来源 | 核心 | 在本系统中的作用 |
|------|------|------|----------------|
| **LLM Wiki** | Karpathy 三层架构实践 | 知识生产管线，四层分工 | 生产线骨架和目录结构 |
| **GBrain** | Garry Tan 的 LLM 长期记忆系统 | Markdown 存储，类型化分类，持续维护 | 长期记忆存储和管理策略 |
| **OKF** | Gemini OKF（Obsidian KMG File）格式 | YAML 元数据 + Markdown 正文 + 结构化引用 | 知识资产标准化格式 |

---

## 1. 当前状态评估

### 已吸收完成的部分

| 要素 | 来自 | 状态 |
|------|------|------|
| 四层分工结构 | LLM Wiki | 已实现 |
| 目录即管线布局 | LLM Wiki | 已实现 |
| Markdown 作为唯一存储格式 | GBrain | 已实现 |
| 类型化分类 | GBrain / OKF | 已实现 |
| Obsidian 作为人类读写界面 | GBrain / OKF | 已使用 |
| YAML 元数据 | OKF | 已定义格式 |
| 知识卡片、主题聚合、判例库三元资产 | 三者融合 | 已实现 |
| 输出反哺闭环 | LLM Wiki | 已建立 |
| AI 可读写文件系统 | GBrain | 已实现（Claude Code） |

### 需持续优化的部分

| 要素 | 来自 | 问题 | 优先级 |
|------|------|------|--------|
| 格式规范执行一致性 | OKF | 旧文件可能缺少 YAML 或格式不一 | **高** |
| 知识卡片密度 | GBrain | 卡片应该更小更原子 | 中 |
| 主题聚合活跃度 | GBrain | 部分主题聚合可能过时未更新 | **高** |
| 双向链接覆盖率 | OKF | 卡片之间的链接不够密集 | 中 |
| 索引自动化 | LLM Wiki | 知识地图需手动或 skill 更新 | 中 |
| 检索引擎 | GBrain | 缺少多维度标签分析 | 低 |

### 明确不采用的部分

| 要素 | 来自 | 不采用的原因 |
|------|------|-------------|
| Postgres/SQLite 数据库 | GBrain | 个人场景无需数据库，Markdown + 目录结构即可 |
| 向量搜索 | GBrain | 单用户知识库文件数有限，Grep + 目录导航更快 |
| MCP Server | GBrain | CLI + 文件系统对个人开发场景足够 |
| 图数据库 | GBrain | Obsidian 图谱可视化已够用 |
| Web UI | GBrain | Obsidian + Claude Code 双界面已覆盖 |
| 多用户 / 权限管理 | GBrain | 个人场景不需要 |
| 消息队列 / 任务编排 | GBrain | 个人场景无并发需求 |

## 2. 优化阶段

### 第一阶段：格式规范统一（立即执行）

目标：所有知识资产文件符合 OKF-like 格式

- [ ] 新文件严格执行 YAML + Markdown 结构
- [ ] 旧文件逐步添加 YAML 头部（优先处理活跃文件）
- [ ] 每个文件至少有 `title`、`type`、`tags`、`created` 四个字段
- [ ] 知识卡片 type 统一为 `KNOWLEDGE_CARD`
- [ ] 主题聚合 type 统一为 `THEME_MAP`
- [ ] 判例 type 统一为 `CASE_ENTRY`
- [ ] 选题卡片 type 统一为 `TOPIC_CARD`

### 第二阶段：资产质量提升（持续进行）

目标：知识资产从"有"到"好"

- [ ] 拆分过大的卡片（一张卡片只讲一个概念）
- [ ] 补充缺失的判断维度（光有定义不够）
- [ ] 增加卡片间双向链接
- [ ] 每张卡片至少有一个通用案例
- [ ] 主题聚合更新日期不超过 90 天
- [ ] 无引用来源的卡片补充来源

### 第三阶段：生产闭环优化（月度迭代）

目标：产出更多回流到知识库

- [ ] 每次内容生产后整理新认知到卡片
- [ ] 每次服务交付后提炼可复用框架
- [ ] 定期（月度）审查知识库活跃度
- [ ] 根据内容生产数据调整知识资产结构
- [ ] 用 Git commit 历史分析知识库建设进度

## 3. 未来规划

### 短期（1-3 个月）

1. **格式扫描工具**：写一个简单脚本，检查所有知识卡片是否包含 YAML 头、必备字段
2. **知识地图自动化**：Claude Code skill 自动扫描全库更新索引
3. **卡片密度提升**：每月目标新增 10-20 张高质量卡片

### 中期（3-6 个月）

1. **模板库建设**：为五类用户提供开箱即用的模板
2. **多语言支持**：知识卡片内容支持中英双语
3. **社区贡献指南**：如果是开源模板，提供贡献规范

### 长期（6-12 个月）

1. **模板与真实项目分离**：模板仓库只存结构、格式、示例文件
2. **全自动化维护**：大部分日常知识库维护由 AI Skill 自动完成
3. **导出工具**：知识资产可导出为其他格式（PDF、HTML、JSON）

---

## 4. 整合后的统一原则

```
1. 知识是生产出来的，不是收集来的
2. 一张卡片 = 一个概念
3. 每份资产有标准格式（YAML + Markdown）
4. 产出必须回流
5. AI 可读写是默认要求
6. 个人使用场景不需要数据库/向量/RAG/MCP
7. 目录结构就是管线布局
8. 四层不越界（证据/解释/表达/流程）
```
