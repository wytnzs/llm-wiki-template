# 知识关系与索引规则

## 1. 最小原则

知识关系的目标是提高检索和判断质量，不是让图看起来更密。

新增或保留一条正式关系前，只问三个问题：

1. **关系明确吗？** 能否用一句话说明目标提供了什么证据、方法、例子、反例、边界或实际应用？
2. **沿过去有用吗？** 人或 AI 进入目标后，是否能更好地回答问题、作出判断或采取行动？
3. **能够回到依据吗？** 是否能够继续回到原文、数据、案例、合同、项目记录或其他可靠来源？

三项都是“是”才建立关系。只是同领域、关键词相同、以后可能有用时，只保留标签或待补说明。

## 2. 元数据、标签、双链和主题页

| 工具 | 回答的问题 | 示例 |
|---|---|---|
| `type` | 这份内容是什么 | `knowledge_card`、`theme_map` |
| `tags` | 它属于哪些稳定类别 | AI、保险、知识管理 |
| 双向链接 | 它具体关联哪一份内容，为什么 | 某案例是某项原则的应用实例 |
| 主题页 | 综合资料后，当前判断和边界是什么 | 如何处理保险拒赔争议 |
| `source` | 这项判断从哪里来 | 原始文章、项目记录、案例或用户确认 |
| `related_projects` | 在哪里真实使用过 | 某个当前项目 |

标签不承担推理，双链不承担全文检索，主题页不承担全库统计。

## 3. 日常优先维护的关系

```yaml
source: 03-Resources/某份原始资料.md
related_theme_maps:
  - "[[某主题地图]]"
related_projects:
  - "[[某项目]]"
```

- `source`：必须是真实依据，不能用另一张卡片的互相引用冒充来源；
- `related_theme_maps`：内容必须直接回答主题问题，或提供证据、方法、反例、边界；
- `related_projects`：只有项目实际调用过才填写。

支持、反例和例子在需要机器筛选或跨页导航时再增加：

```yaml
links:
  supports: []
  contrasts: []
  examples: []
```

如果只能写“相关”，先不要建立正式关系。

## 4. 主题页关系准入

主题页的 `related_cards` 和 `related_cases` 是精选入口，至少满足一项：

1. 直接回答核心问题；
2. 提供关键证据或方法；
3. 提供反例或适用边界；
4. 是需要再次调用的代表性验证案例。

主题页不追求覆盖全部同领域文件。新增或移除关系时，若知识库使用显式双向字段，应同步维护卡片一侧；Obsidian 的自动反向链接不能替代元数据一致性。

## 5. 自然语言索引和检索顺序

推荐检索顺序：

```text
自然语言问题
-> 主题中心：定位持续问题
-> 主题页：读取当前判断和边界
-> 知识卡片或案例：读取方法与证据
-> 原始材料：核验正式引用
-> 项目或输出
```

索引只负责“先找到哪里”，双链负责“下一步为什么去那里”。二者应当配合，而不是互相替代。

## 6. 不应建立关系的情况

- 两份内容只共享一个宽泛领域；
- 内容生产过程中使用了 AI，但正文并不讨论 AI；
- 某知识未来也许能写成内容，但尚未真实使用；
- 单个案例被用来证明普遍规律；
- 只是为了消除孤立页或提高关系数量；
- 来源缺失，无法说明连接依据。

## 7. 批量关系变更

批量添加或清理关系时：

1. 先输出预计新增、保留和移除数量；
2. 抽查高相关、边界相关和明显不相关三类样本；
3. 阅读正文，不只看标题、标签和关键词；
4. 检查 YAML、断链、同名歧义和双向一致性；
5. 一旦出现系统性误连，立即收紧规则并重新干跑；
6. 链接数量只作观察，不作为质量目标。

法律、医疗、保险和客户材料中的关键结论链必须逐条核对证据和适用边界。

## 8. 关系复查触发条件

只在以下情况复查，不做无意义的周期打分：

- AI 沿关系连续返回不相关内容；
- 来源、正文或适用边界发生实质变化；
- 文件移动、合并、替代或出现同名歧义；
- 新证据推翻原来的连接依据。

移除关系不等于删除原始内容；无法确认的关系可以退回标签或待补项。
