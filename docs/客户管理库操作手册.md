# 客户管理库操作手册

> 适用版本：customer-solution-library v3  
> 适用对象：个人使用者、团队负责人、Claude Code / Codex / Agent 使用方  
> 核心目标：把客户信息从“能记录”升级为“能分析、能筛选、能生成方案、能安排跟进”。

---

## 1. 这套客户管理库是什么

客户管理库不是传统 CRM，也不是通讯录、Excel 表格或普通知识库。

它是一套面向长期客户服务的「客户信息管理与解决方案系统」。

一句话说：客户管理库用 Markdown 保存长期档案，用结构化索引支持快速查询和筛选，用 Skill 约束 Agent 的录入、分析、匹配、方案和复盘动作。

它要解决的问题不是：

```text
我有没有把客户信息记下来？
```

而是：

```text
我是否真正理解了这个客户？
下一步该怎么服务？
哪些客户适合某个产品/服务？
哪些客户还缺关键信息？
哪些客户本周最值得跟进？
```

这套系统围绕下面这条链路工作：

```text
客户信息
→ 入口判断
→ 身份匹配
→ 档案沉淀
→ 客户洞察
→ 信息缺口
→ 产品/服务匹配
→ 专属方案
→ 跟进复盘
```

最终沉淀的不只是客户资料，而是可持续使用的客户经营资产。

### 1.1 为什么这样构建

这套系统选择“知识库 + Skill + 私有客户库”的三层结构，是为了解决三个矛盾：

1. 客户资料需要长期沉淀，但不能暴露在公开仓库或知识库正文里；
2. 知识库需要持续积累方法论，但不能被真实客户细节拖得越来越臃肿；
3. Agent 需要知道如何操作客户资料，但不能每次都靠临时口头说明。

所以分工是：

```text
知识库：保存可复用的方法、产品知识、话术、模板、脱敏案例
Skill：保存客户管理规则、字段规范、脚本和 Agent 操作协议
私有客户库：保存真实客户档案、事件流、分析、方案、跟进记录
```

这样既能让 AI 借用知识库能力，又能把真实客户资料隔离在私有客户库中。

### 1.2 与传统方案的区别

| 方案 | 主要价值 | 主要限制 |
|---|---|---|
| 通讯录 | 找到人 | 缺少过程、分析和方案 |
| Excel/表格 | 批量管理字段 | 长文本、聊天记录、方案复盘很难沉淀 |
| 传统 CRM | 销售流程、商机阶段 | 对个人知识库、私域服务、复杂客户洞察不够灵活 |
| 普通知识库 | 沉淀经验 | 不适合保存真实客户隐私和动态跟进 |
| 客户管理库 | 录入、分析、筛选、方案、跟进、复盘一体化 | 需要遵守路径、隐私、身份匹配和人工复核规则 |

## 2. 适合什么场景

适合：

- 保险客户管理、保单整理、保障规划、理赔协助；
- 咨询顾问管理客户需求、方案、交付和复盘；
- AI / 知识库 / 自动化服务客户管理；
- 私域客户、内容用户、转介绍客户管理；
- 团队协同处理客户资料、生成分析和跟进清单；
- 根据某个产品或服务反向筛选潜在客户。

不适合：

- 只想保存姓名电话的简单通讯录；
- 完全不需要分析和跟进的静态客户表；
- 希望系统替代专业判断、核保、理赔、法律或财务承诺的场景。

## 3. 核心概念

### 3.1 知识库项目

如果你已经有个人知识库，建议把本 Skill 安装到知识库项目内。

知识库负责保存：

- 方法论；
- 产品知识；
- 标准话术；
- 操作模板；
- 脱敏案例；
- 复盘经验；
- 可公开分享的内容。

知识库不应该保存真实客户资料。

### 3.2 真实客户库

真实客户库是你保存客户资料的地方。路径必须由使用者自己指定，不应使用本仓库作者的电脑路径作为默认值。

```text
<你的私有客户库路径>
```

这里会保存客户档案、事件流、分析、方案、跟进、产品机会清单等内容。

真实客户资料不要放进 Skill 仓库，也不要提交到公开 GitHub。

### 3.3 客户管理 Skill

本仓库就是客户管理 Skill。

它负责给 Agent 提供客户管理规则、字段规范、分析流程和脚本工具。

推荐安装位置：

```text
知识库项目/.claude/skills/customer-solution-library/
```

这样 AI 在处理知识库项目时，可以同时读取知识库内容和客户管理规则。

### 3.4 客户信息事件

系统的最小计量单位不是“客户”，而是“客户信息事件”。

客户信息事件可以是：

- 一段聊天记录；
- 一段录音转写；
- 一次面谈纪要；
- 一条客户口述；
- 一份保单或资料补充；
- 一个健康异常信息；
- 一次方案沟通；
- 一个跟进动作；
- 一次团队反馈。

客户主档案只保存稳定摘要；历史细节放进事件流。

### 3.5 客户洞察卡

客户信息事件回答“发生了什么”。

客户洞察卡回答“这意味着什么”。

洞察类型包括：

- 需求洞察；
- 风险洞察；
- 机会洞察；
- 顾虑洞察；
- 信息缺口；
- 跟进建议。

### 3.6 产品/服务画像

如果你想问：

```text
某产品有哪些潜在客户？
```

就需要先为产品或服务建立画像。

产品画像说明：

- 适合谁；
- 不适合谁；
- 匹配信号是什么；
- 风险信号是什么；
- 形成方案前必须具备哪些信息。

### 3.7 当前已经进化到什么程度

这套客户管理库已经不只是“录入系统”，而是一个面向 Agent 的客户经营操作系统雏形。

目前已经覆盖：

- 信息入口：判断一段信息是否有价值，是否应该入库；
- 身份匹配：防止同名客户、不同业务线客户被误合并；
- 分批补全：客户信息不完整时，先事件化保存，再逐步补齐；
- 深度分析：识别需求、风险、机会、顾虑和下一步动作；
- 信息缺口：判断形成方案前还缺什么；
- 产品匹配：根据产品/服务反向筛选潜在客户；
- 经营看板：按周查看优先跟进、缺动作、可复核客户；
- 知识库联动：用知识库的方法论和产品知识辅助生成方案；
- 团队协同：支持脱敏摘要、协同任务、复核清单和分享边界。

它现在最重要的价值不是“存资料”，而是让客户信息进入一条可以持续经营的链路。

### 3.8 如何避免知识库越来越臃肿

知识库联动不是把客户库内容全部塞进知识库。

正确做法是：

- 真实客户事实留在私有客户库；
- 知识库只保存可复用内容；
- 只有脱敏后的典型案例、方法论、话术、复盘经验才能回流知识库；
- Agent 处理客户任务时，按需读取知识库，而不是每次读取全部内容；
- 输出客户方案时，要标明哪些来自客户事实，哪些来自知识库参考。

这能降低两个成本：

1. 隐私成本：真实客户资料不会被错误扩散；
2. token 成本：Agent 不必每次加载大量无关知识库内容。

## 4. 安装方式：优先项目级安装

不建议默认全局安装。

更推荐把 Skill 安装到当前项目的 `.claude/skills` 目录中，尤其是你的知识库项目。

这样 Claude Code / Agent 在打开该项目时，更容易稳定识别并使用客户管理规则。

### 4.1 Windows：安装到当前项目

进入你的知识库或工作项目目录，例如：

```powershell
cd "<你的知识库项目路径>"
```

运行：

```powershell
New-Item -ItemType Directory -Force ".claude\skills" | Out-Null; git clone https://github.com/wytnzs/customer-solution-library.git ".claude\skills\customer-solution-library"
```

安装后，对 Agent 说：

```text
使用项目内 .claude/skills/customer-solution-library，帮我初始化并管理客户库。
```

如果已经安装过，更新：

```powershell
git -C ".claude\skills\customer-solution-library" pull
```

如果更新时报错，或者这个目录不是独立 git 仓库，建议先备份旧目录，再重新安装：

```powershell
Rename-Item ".claude\skills\customer-solution-library" "customer-solution-library.backup"
git clone https://github.com/wytnzs/customer-solution-library.git ".claude\skills\customer-solution-library"
```

### 4.2 macOS / Linux：安装到当前项目

在终端中运行：

```bash
mkdir -p .claude/skills && git clone https://github.com/wytnzs/customer-solution-library.git .claude/skills/customer-solution-library
```

安装后，对 Agent 说：

```text
使用项目内 .claude/skills/customer-solution-library，帮我初始化并管理客户库。
```

如果已经安装过，更新：

```bash
git -C .claude/skills/customer-solution-library pull
```

### 4.3 可选：全局安装

只有当你希望所有 Codex 任务都默认可用时，才建议安装到全局：

```powershell
New-Item -ItemType Directory -Force "$env:USERPROFILE\.codex\skills" | Out-Null; git clone https://github.com/wytnzs/customer-solution-library.git "$env:USERPROFILE\.codex\skills\customer-solution-library"
```

但如果你的主要使用场景是“知识库项目 + 客户管理库”，请优先使用项目级安装。

如果之前已经全局安装过，也不影响项目级安装。项目内 `.claude/skills/customer-solution-library` 应作为当前项目的优先使用版本。

## 5. 第一次使用：初始化客户库

建议把真实客户库放到单独的私有目录。不要默认放到 Skill 仓库或知识库正文目录中，除非你明确知道自己在做什么。

示例路径仅供参考，不是默认值：

```text
Windows: %USERPROFILE%\CustomerSolutionLibrary-Private
macOS/Linux: ~/CustomerSolutionLibrary-Private
云盘/团队目录: <你的私有同步目录>/CustomerSolutionLibrary-Private
```

可以让 Agent 初始化：

```text
使用项目内 .claude/skills/customer-solution-library。
请先询问我希望把私有客户库存放在哪里，然后在我确认的路径初始化客户库。
```

也可以手动运行：

```powershell
python scripts/customer_library.py init --base "<你的私有客户库路径>"
```

也可以设置环境变量，后续命令就不用每次写 `--base`：

Windows PowerShell：

```powershell
$env:CUSTOMER_LIBRARY_BASE="<你的私有客户库路径>"
```

macOS / Linux：

```bash
export CUSTOMER_LIBRARY_BASE="<你的私有客户库路径>"
```

初始化后会生成：

```text
00-入口池/
01-客户档案/
02-客户事件流/
03-业务模块/
04-方案与分析/
05-团队协同/
06-服务跟进/
07-客户分层/
08-复盘案例/
09-产品与机会/
10-客户洞察/
90-索引与看板/
99-模板与规则/
```

## 6. 日常用法一：录入客户信息

大多数时候，你只需要把客户信息交给 Agent。

示例：

```text
使用项目内 .claude/skills/customer-solution-library。
客户库路径是 <你的私有客户库路径>。

请处理这条客户信息：
张先生，杭州，李姐转介绍，已婚一孩，想了解重疾险，最近体检发现甲状腺结节，周五晚上有空。
```

Agent 应完成：

1. 判断信息价值；
2. 搜索是否已有客户；
3. 检查同名客户风险；
4. 确认后写入客户事件流；
5. 更新客户主档案摘要；
6. 更新相关业务模块；
7. 设置下一步动作；
8. 刷新索引。

### 6.1 一步操作、三步操作和自动操作

日常低风险场景，可以使用一步操作：

```text
使用项目内 .claude/skills/customer-solution-library。
客户库路径是 <你的私有客户库路径>。

请处理这条客户信息，并在适合入库时完成入库、更新索引和下一步建议：
……
```

一步操作适合：

- 客户身份明确；
- 信息敏感度低；
- 只是补充普通沟通记录；
- 不涉及承诺、健康、理赔、法律或财务结论。

如果信息很敏感或客户身份不确定，使用三步操作。

```text
第一步：先只判断这段客户信息的价值，不要入库：……
第二步：根据上面的信息匹配可能客户，但先不要写入。
第三步：确认是 CUST-0001，请正式入库。
```

适合三步操作的情况：

- 同名客户；
- 健康告知；
- 理赔、投诉、纠纷；
- 保单变更；
- 重要承诺；
- 隐私信息；
- 团队交接。

如果想让各种 Agent 都尽量实现“一步自动操作”，建议在项目说明或启动语中固定写明：

```text
处理客户信息时，请默认按客户管理库流程执行：
入口价值判断 → 身份匹配 → 写入事件流 → 更新主档案 → 生成洞察/缺口 → 设置下一步动作 → 刷新索引。

但如果出现同名客户、身份不确定、敏感信息、重大承诺、健康/保险/理赔/法律/财务风险，请先停止在确认点，不要直接写入或对外输出。
```

## 7. 日常用法二：查询客户

用客户编号最稳：

```text
查询 CUST-0001 的客户速览。
```

如果只知道姓名或称呼：

```text
查一下张先生，先列出可能客户，不要直接合并。
```

手动搜索命令：

```powershell
python scripts/customer_library.py find --base "<你的私有客户库路径>" --query 张先生
```

注意：

只有姓名相同不能自动并档。必须结合来源、城市、手机号后四位、历史事件、业务线等身份锚点确认。

## 8. 日常用法三：生成客户深度分析

录入只是基础，真正有价值的是分析。

你可以对 Agent 说：

```text
使用项目内 .claude/skills/customer-solution-library。
客户库路径是 <你的私有客户库路径>。

请分析 CUST-0001，输出客户深度分析、信息缺口和下一步跟进建议。
```

手动命令：

```powershell
python scripts/customer_library.py analyze-customer --base "<你的私有客户库路径>" --customer-id CUST-0001
```

输出位置：

```text
04-方案与分析/CUST-0001/客户深度分析.md
```

深度分析应重点回答：

- 当前客户的关键需求是什么；
- 关键风险是什么；
- 潜在机会是什么；
- 客户顾虑是什么；
- 还缺哪些信息；
- 下一步最小动作是什么。

注意：

深度分析默认是内部版，不要直接发给客户。发给客户前，应让 Agent 转成客户可读版话术。

## 9. 日常用法四：生成信息缺口清单

很多时候不能直接出方案，不是客户没价值，而是关键信息还不够。

你可以说：

```text
请检查 CUST-0001 还缺哪些信息，才能形成稳妥方案。
```

手动命令：

```powershell
python scripts/customer_library.py gaps --base "<你的私有客户库路径>" --customer-id CUST-0001
```

输出位置：

```text
04-方案与分析/CUST-0001/信息缺口清单.md
```

常见信息缺口：

| 客户类型 | 常见缺口 |
|---|---|
| 保险客户 | 已有保单、健康异常详情、预算/缴费能力、家庭责任 |
| 咨询客户 | 明确目标、当前卡点、预算范围、决策链条 |
| 合作客户 | 合作目标、资源边界、决策人、推进时间 |
| AI/知识库客户 | 当前流程、资料范围、使用人、交付目标 |

## 10. 日常用法五：根据产品/服务筛选潜在客户

这是 v3 的核心能力之一。

你可以问：

```text
我准备推广 PROD-001，请筛选潜在客户，并说明匹配原因、风险点、缺口和建议动作。
```

手动命令：

```powershell
python scripts/customer_library.py product-match --base "<你的私有客户库路径>" --product-id PROD-001
```

输出位置：

```text
09-产品与机会/matching/PROD-001-潜在客户清单.md
```

### 10.1 先建立产品/服务画像

产品画像建议放在：

```text
09-产品与机会/products/
```

示例：

```markdown
---
type: product_profile
product_id: PROD-001
product_name: 家庭重疾险
business_line: insurance
status: active
suitable_for:
  - 有家庭责任
  - 有保障缺口
not_suitable_for:
  - 明确拒绝长期缴费
matching_signals:
  - 重疾险
  - 家庭保障
  - 孩子
risk_signals:
  - 甲状腺结节
  - 预算压力
required_information:
  - 已有保单
  - 体检报告
---

# 家庭重疾险

适合有家庭责任、保障缺口、主动咨询重疾险的客户。
```

### 10.2 怎么理解匹配结果

产品潜在客户清单不是“直接推荐名单”，而是“内部经营初筛名单”。

你需要重点看：

- 匹配度；
- 匹配原因；
- 风险点；
- 缺口；
- 建议动作。

尤其是保险、健康、理赔、财务承诺相关场景，高匹配也不代表可以直接推荐，必须人工复核。

## 11. 日常用法六：生成本周经营看板

经营看板用于回答：

```text
这周我应该重点跟谁？
哪些客户缺下一步动作？
哪些客户信息不足但仍值得推进？
哪些客户可以进入方案复核？
```

你可以说：

```text
使用 $customer-solution-library。
请生成本周经营看板。
```

手动命令：

```powershell
python scripts/customer_library.py dashboard --base "<你的私有客户库路径>"
```

输出位置：

```text
90-索引与看板/本周经营看板.md
```

建议每周至少生成一次。

## 12. 日常用法七：刷新索引和质检

### 12.1 刷新索引

客户批量更新后，刷新索引：

每周或每月运行：

```powershell
python scripts/customer_library.py audit --base "<你的私有客户库路径>"
```

生成：

```text
90-索引与看板/数据质量报告.md
```

质检会检查：

- 活跃客户是否缺下一步动作；
- 方案/服务阶段是否缺跟进日期；
- 信息完整度是否偏低；
- 是否有待处理入口；
- 是否有待匹配入口；
- 是否有疑似重复客户；
- 是否有隐私风险；
- 是否有待复核协同任务。

## 15. 分享与团队协同

客户管理库可以支持团队协作，但协作的默认原则是“最小必要共享”。

常见协同方式：

- 给团队成员：分享脱敏客户摘要、跟进任务、待确认问题；
- 给上级或合伙人：分享阶段性经营看板、产品机会清单、复核事项；
- 给执行同事：分享明确任务卡，不分享不必要的完整客户隐私；
- 给外部客户：只输出客户可读版话术，不输出内部推断和评分。

建议使用三类输出：

| 输出类型 | 用途 | 是否包含真实客户隐私 |
|---|---|---|
| 协同任务卡 | 分派下一步动作 | 尽量少 |
| 脱敏摘要 | 团队讨论、复盘、培训 | 不应包含 |
| 内部复核清单 | 保险、理赔、法律、财务等高风险场景 | 可包含必要信息，但应限制权限 |

团队共享前，先让 Agent 做一次脱敏检查：

```text
请把 CUST-0001 的内部分析整理成团队协同版，只保留完成任务所需的信息，移除不必要的隐私细节，并列出仍需人工复核的内容。
```

## 16. 隐私和边界

不要把真实客户资料提交到公开仓库。

建议：

- 真实客户库单独保存；
- 高敏信息尽量脱敏；
- 文件名不要直接写完整真实身份；
- 团队协同时只共享必要摘要；
- 对客户可见内容和内部判断分开；
- 不确定的信息标记为待确认，不要写成事实；
- 涉及健康、法律、理赔、财务承诺时保持保守。

特别提醒：

产品匹配清单、客户深度分析、信息缺口清单都属于内部经营辅助材料，不应未经处理直接发给客户。

## 17. 一句话总结

这套客户管理库的价值，不是把客户资料存起来，而是把客户信息变成可理解、可检索、可分析、可筛选、可跟进、可复盘的长期服务资产。

最重要的使用原则是：

```text
先接住信息，再判断价值；
先匹配身份，再沉淀档案；
先识别缺口，再生成方案；
先人工复核，再对外沟通。
```
