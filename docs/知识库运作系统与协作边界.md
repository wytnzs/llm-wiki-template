# 知识库运作系统与协作边界

## 1. 这套系统解决什么问题

知识库不是“把文件存起来”，而是让原始输入经过整理、连接和使用，逐步变成可追溯、可复用、可维护的知识资产。

系统同时运行两条链路：

1. **知识生产链**：输入如何变成资产并服务真实工作；
2. **检索验证链**：问题如何找到当前判断，并回到原始依据核验。

```mermaid
flowchart LR
    I["输入：原始表达与外部材料"] --> R["整理：判断内容角色"]
    R --> S["统一来源层：外部资料、个人输入、系统历史"]
    R --> A["资产：卡片、案例、方法"]
    A --> T["主题页：当前综合判断"]
    T --> P["项目与输出"]
    P --> F["反馈与复盘"]
    F --> A
    F --> T
    S -. "来源核验" .-> A
```

目录只说明内容当前承担的角色，不负责表达所有关系。一份知识保留一个稳定位置，通过来源、主题页和项目关系进入不同使用场景。

## 2. 四种内容角色

| 角色 | 回答的问题 | 推荐位置 | 处理原则 |
|---|---|---|---|
| 输入 | 这是什么新材料 | `00-Inbox/` | 保留原貌，先判断再加工 |
| 项目 | 当前要完成什么 | `01-Projects/` | 只保存服务当前目标的状态和材料 |
| 资产 | 什么能跨项目复用 | `02-Areas/` | 保存概念、判断、方法、案例和主题认知 |
| 来源 | 原始依据在哪里 | `03-Resources/` | 区分外部资料、个人输入和系统历史，保留原貌和来源 |

已有知识库可以使用其他目录名，只需在根目录 README 中写清实际映射。不要为了符合模板批量迁移文件。归档不是必需的顶层目录：仍需追溯的材料进入来源层，处理状态由集合导读、标签和使用关系表达；确认重复或不再保留的内容另行执行可恢复清理。

## 3. 基本运作闭环

```text
输入 -> 整理 -> 连接 -> 使用 -> 检查 -> 复盘
```

### 输入

- 保存用户原始表达和外部材料；
- 不急着把每条输入都变成卡片；
- 标明来源状态：用户观察、外部资料、案例事实或待核验。

### 整理

- 只给一个主要去向：继续保留、进入项目、提炼资产、进入来源层（外部资料、个人输入或系统历史），或在确认后进入可恢复清理；
- 只有能跨项目复用并帮助判断、决策或行动的内容才进入资产层；
- 外部观点不能悄悄改写成用户自己的稳定判断。

### 连接

一份资产日常先回答三个问题：

1. 从哪里来：`source`；
2. 直接帮助回答哪个持续问题：`related_theme_maps`；
3. 在哪个项目中真实使用过：`related_projects`。

支持、反例、例子等语义关系按需使用，不是必填项。

### 使用

- 项目和内容生产前先明确目标、受众、当前状态、证据与风险；
- 不从单张卡片直接跳到最终结论；
- 高风险结论必须回到合同、案例、原文或其他可靠证据。

### 检查与复盘

- 检查编码、YAML、链接、来源、边界和实际效果；
- 失败时判断问题来自输入、知识、项目定义、模板还是 Skill；
- 可复用修正写回上游，不只修改一次输出。

## 4. 主题、聚合、标签、双链和地图的分工

| 组件 | 作用 | 不承担什么 |
|---|---|---|
| `type` | 说明文件是什么 | 不负责主题检索 |
| `tags` | 稳定分类和筛选 | 不表达因果、支持或反例 |
| 双向链接 | 连接两份有明确语义关系的内容 | 不为增加网络密度而批量互链 |
| 主题 | 一个需要持续回答的问题 | 不是普通领域名称 |
| 聚合 | 围绕问题寻找、筛选资料的动作 | 不单独形成知识类型或一级目录 |
| 主题页 | 保存当前判断、证据、方法、反例和边界 | 不是同领域文件清单 |
| 主题中心 | 自然语言入口和主题页的统一导航 | 不保存全库运行统计 |
| 知识库地图 | 数量、项目状态、健康问题和下一步 | 不替代主题页或内容索引 |

完整关系规则见[知识关系与索引规则](知识关系与索引规则.md)。

## 5. AI 检索顺序

```mermaid
flowchart LR
    Q["任务或自然语言问题"] --> T["主题中心与主题页"]
    T --> K["知识卡片或案例"]
    K --> O["原始资料、合同、数据或项目记录"]
    O --> D["回答、决策或交付"]
    D --> W["反馈写回"]
```

双链不是搜索引擎。AI 通常先结合 README、文件搜索、标题、描述、标签和主题入口定位内容，再沿双链下钻。双链的价值是缩小下一跳范围并说明“为什么相关”，而不是替代索引、全文搜索或来源核验。

## 6. 最小文件和证据规范

正式知识文件至少包含：

```yaml
---
title: 文件标题
type: knowledge_card
---
```

按需要增加：

```yaml
description: 一句话说明解决什么问题
status: seed | growing | stable
tags: []
source: 可核验来源
related_theme_maps: []
related_projects: []
created: YYYY-MM-DD
updated: YYYY-MM-DD
```

证据规则：

- 没有可靠来源时保持空缺或标记 `needs_review`，不写“综合研究”等模糊来源；
- 具体数字、比例、市场案例和法律、医疗结论必须能够回到原始依据；
- 单个案例只能说明该案或提供一种可能，不能直接证明普遍结论；
- 主题页中的目录可以作为导航，但不能冒充具体证据。

## 7. AI 修改权限

| 内容 | 可以自动做 | 必须确认 |
|---|---|---|
| 原始输入与03来源材料 | 读取、分类建议、建立来源 | 改写正文、移动、覆盖、删除 |
| 知识卡片 | 补格式、起草摘要、修复明确链接 | 改变用户判断、补猜测来源、升级证据状态 |
| 主题页 | 更新结构、整理已确认依据 | 改写核心判断、加入高风险结论、批量增删关系 |
| 案例、合同、病历 | 检索、提取、比对、生成草案 | 修改事实、形成最终法律或医疗结论 |
| 索引、地图、日志 | 按既定口径统计和维护 | 用旧数字或估算替代实际扫描 |
| Git 与外部系统 | 查看状态、生成发布清单 | 提交、推送、发消息、删除远程数据 |

## 8. 协作边界

1. README 是系统职责和真实目录的唯一入口；
2. 不让两个 Agent 同时修改同一成品；
3. 工作区已有修改默认属于用户或其他协作者，不擅自回滚；
4. 批量操作先干跑、抽查，再写入；
5. 删除、覆盖、移动、对外发送和高风险判断必须有明确授权；
6. 不把 Skill 变成业务资料库，不把个人内容写进通用模板；
7. 用户判断、外部资料和案例事实必须分开记录。

## 9. GitHub 公开边界

公开仓库可以包含：

- 系统规则和说明；
- 空白模板；
- 脱敏示例；
- 通用 Skill；
- 不含业务数据的检查脚本和初始化工具。

公开仓库不得包含：

- 真实知识卡片、案例和客户资料；
- 合同、病历、沟通记录和个人信息；
- 项目内部材料、内容成品和个人表达配置；
- API Key、访问令牌、本地路径和私有环境配置。

混合工作区发布时必须逐项暂存，禁止默认使用 `git add -A`。推送前运行公开安全检查，并查看最终暂存差异。

## 10. 什么时候才需要数据库、RAG 或知识图谱

文件系统、全文搜索、主题入口和高质量双链能够覆盖大多数个人知识工作。只有同时出现以下需求时，才值得评估更复杂的基础设施：

- 数据规模已经让文件检索明显失效；
- 需要多用户权限、并发写入或审计日志；
- 需要自动实体抽取、向量召回或跨库联合查询；
- 有稳定工程资源维护数据库、索引和评估体系。

复杂技术不能替代来源、边界、更新和质量标准。没有稳定内容治理时，引入 RAG 只会更快地检索到混乱内容。

## 11. 系统完成标准

- 人能从 README 判断每个区域的职责；
- AI 能在 1 至 2 跳内从问题进入可用资产；
- 关键判断能够回到依据；
- 标签、双链、主题页和地图没有重复承担同一职责；
- 原始材料仍可追溯；
- 待确认事项、当前地图和变更日志状态一致；
- 公开仓库不包含私人或业务数据。
