# GBrain 思想吸收说明

## GBrain 是什么

GBrain 是 Garry Tan（Y Combinator 前 CEO）提出的 LLM 长期记忆系统。核心思路是：

**给 AI 一个可读写的文件系统作为长期记忆，而不是依靠会话上下文或 RAG。**

当 AI 需要记住什么，就写到文件里；当 AI 需要回忆什么，就去文件里读。

---

## 我们采纳了什么

### 1. Markdown 作为长期记忆的存储格式

**GBrain 的选择：** Markdown 够简单、够通用、AI 最易读。

**我方采纳：** 全库使用 Markdown 作为唯一存储格式，不引入数据库、二进制文件、富文本格式。

### 2. 类型化分类

**GBrain 的选择：** 不同内容类型用不同存储策略。

**我方采纳：**

| 类型 | 存储位置 |
|------|---------|
| 概念卡片 | 02-Areas/知识卡片/ |
| 主题聚合 | 02-Areas/主题聚合/ |
| 判例 | 02-Areas/判例库/ |
| 选题卡片 | 06-选题库/cards/ |
| 项目笔记 | 01-Projects/ |
| 原始素材 | 00-Inbox/ |

### 3. 信号捕获（Capture Signal）

**GBrain 理念：** 有价值的信息捕获要即时，格式不重要，先存下来再说。

**我方采纳：** `00-Inbox/` 专门做信号捕获。不管是灵感、链接、对话片段，先记下来。定期加工提炼。

### 4. 大脑优先检索（Brain-First Retrieval）

**GBrain 理念：** 先读目录结构和索引文件，必要时再读具体内容。

**我方采纳：** `知识库地图.md` 作为第一入口，配合 `06-选题库/选题总索引.md` 等索引文件。AI 执行任何任务前先读索引，再决定读哪些文件。

### 5. 持续维护（Continuous Maintenance）

**GBrain 理念：** 长期记忆不维护会腐烂。

**我方采纳：**

- 定期加工收件箱（清理 00-Inbox）
- 主题聚合更新（反映观点变化）
- 卡片质量提升（补充判断维度、案例）
- 知识地图更新（反映新增内容）
- Obsidian 图谱维护（确保链接可达）

### 6. 厚技能（Thick Skills）

**GBrain 理念：** Skill 应该包含足够的上下文和示例，让 AI 读一个文件就知道怎么做。

**我方采纳：** Skill 不仅写步骤，还要写格式模板、示例输出、常见错误、边界条件。

## 我们没采纳什么（及原因）

| GBrain 要素 | 不采纳原因 | 替代方案 |
|------------|-----------|---------|
| **Postgres 数据库** | 个人使用场景下，Markdown 文件存取比数据库更简单，不需要 SQL、不需要连接管理 | 文件系统 + Grep |
| **向量搜索（pgvector）** | 单用户知识库文件数量有限，Grep + 目录结构导航更直接 | `grep -r` / 标签过滤 |
| **MCP Server** | 需要额外维护一个服务进程，增加复杂度和故障点。个人命令行场景 CLI 直接操作文件更可靠 | Claude Code 直接读写文件 |
| **图数据库** | 知识图谱用 Obsidian 可视化就够了，不需要额外的数据库维护 | Obsidian 图谱视图 |
| **多用户 / 权限管理** | 个人场景不需要 | 不需要 |
| **Web UI** | 个人场景下 Obsidian 作为桌面端 + Claude Code 作为命令行端已经覆盖全部需求 | Obsidian + Claude Code |
| **消息队列 / 任务编排** | 个人工作流无并发需求 | 串行执行 / 简单的命令行脚本 |

---

## 这个轻量替代方案适合谁

**目标用户画像：**

- 个人使用者（非团队）
- 有一定的编程或命令行基础（至少会用终端）
- 使用 Obsidian 做笔记或想开始做笔记
- 使用 Claude Code 或类似 AI 编程工具
- 想建立一个可持续积累、AI 可读写的知识系统
- **不需要**数据库、服务器、Web UI 等基础设施

**不适合谁：**

- 需要团队协作的（本系统未设计多用户权限）
- 不想用命令行的（需要 Obsidian 和基本终端操作）
- 需要移动端深度使用的（Obsidian 移动版可用但体验有限）
- 需要完整的 REST API 的（本系统不提供 API）

## 总结

```text
GBrain 的理念                         我方吸收
──────────────────────────────────────────────
Markdown 存储           →  全库 Markdown
类型化分类              →  五类资产各归其位
信号捕获               →  00-Inbox 收件箱
大脑优先检索            →  知识库地图 + 索引文件
持续维护               →  定期加工和更新
厚技能                 →  Skill 含模板和示例

Postgres               →  不需要（文件系统就够了）
向量搜索               →  不需要（Grep 就够了）
MCP Server             →  不需要（CLI 就够了）
图数据库               →  不需要（Obsidian 图谱够了）
Web UI                 →  不需要（Obsidian 够了）
多用户                 →  不需要（个人场景）
```

**轻量化的代价：** 没有 Web 界面远程访问、没有 API 供其他系统调用、检索效率在大规模知识库（>10 万文件）会下降。但这些代价对个人用户来说可以接受。
