# 说明书类文章框架

> 基于「佛教数字文献整合平台 fojin」和「史记知识库 shiji-kb」两篇范例提炼。
> 适用于：开源项目介绍、工具/产品说明书、平台使用指南、知识库/数据集发布说明等"让人理解并用起来"的内容。

---

## 与其他文章类型的区别

| 维度 | 深度长文（writing-style.md） | 教程类（tutorial-framework.md） | 说明书类（本文件） |
|------|---------------------------|-------------------------------|-----------------|
| 目标 | 10w+ 传播、思想输出 | 让读者"学会并动手" | 让读者**理解全貌并选择最适合自己的用法** |
| 字数 | 8000-12000 字 | 2000-4000 字 | 3000-6000 字 |
| 开头 | 故事先行，700 字不出论点 | 先看结果，10 秒展示成品 | **一句话定义 + 核心数据冲击** |
| 结构 | 四幕叙事弧 | 六段式（结果→概念→操作→实战→交付→升华） | **概览→成果→特性→用法→价值→尾声** |
| 情感弧线 | 好奇→认同→焦虑→释然→升华 | 惊艳→理解→跟做→成就→想试 | **震撼→逐一认知→"我该从哪入手"→行动** |
| 语气 | 叙事+观点输出 | 口语化，手把手 | **客观、清晰、实用主义，不讲故事** |
| 表格密度 | 每 500-800 字一个 | 每 300-500 字一个 | **每 200-400 字一个，表格是主力** |
| 核心武器 | 金句、跨文化引用、隐喻 | 代码块、截图标记、分阶段叙事 | **数据表格、功能清单、分层用户路径** |

## 框架：六段式结构

### 0. 项目一句话定义 + 核心数据（开头即高潮）

**铁律：标题和第一段就是全文最密的信息。不铺垫、不讲故事、不绕弯。**

标题模式：
```
[一个开源的] + [一句话说清楚这是什么] + [核心数据点]
```

范例标题：
- 「一个开源的AI《史记》知识库与26个可复用Skill，57万字史记，130篇的10万个实体、3185个事件、7652条关系全部结构化」
- 「一个开源的佛教数字文献整合平台，它将全球440余个佛教文献数据库汇聚于一处」

第一段格式：
```markdown
[项目名]：[一句话说清楚做了什么]。做了两件事：第一，[核心产出A——附带数据]；
第二，[核心产出B——附带数据]。[一句话说可复用性]。

[为什么需要这个项目——一段话说痛点]。
这个项目要回答一个问题：**[核心问题]？** 答案是[结论]。

项目创作者为[名字]，[一句话背景]。

**Github地址**
[链接]

**在线体验**
[链接]
```

**关键原则：**
- 标题即摘要——读者看完标题就知道这个项目做了什么、规模多大
- 第一段不超过 150 字，必须包含核心数据
- Github 和在线体验链接紧跟其后，不要藏在文末
- 痛点一段话说完，不展开讲故事

---

### 一、核心成果（数据总览表）

**任务：用一张表让读者在 10 秒内了解项目的完整产出。**

```markdown
## 核心成果

| 类别 | 数量 | 说明 |
|------|------|------|
| [产出A] | [具体数字] | [一句话解释] |
| [产出B] | [具体数字] | [一句话解释] |
| [产出C] | [具体数字] | [一句话解释] |
| ... | ... | ... |
| [方法论/工具] | [数量] | **可复用，适用于 [更大范围]** |
```

**关键原则：**
- 表格行数 5-8 行为佳，不要超过 10 行
- 每行的"数量"列必须有具体数字（不要写"大量""丰富"）
- 最后一行突出可复用性/可扩展性（加粗）
- 这张表配一张配图（数据总览信息图）

---

### 二、功能特性（逐一展开，每个特性一小节）

**任务：每个特性用一个独立小节介绍，标题即功能名+一句话卖点。**

标题格式：
```markdown
### [功能名]：[一句话说清楚这个功能干什么]
```

范例：
- 「语法高亮：18类实体，一眼看懂谁在哪里做了什么」
- 「三家注弹窗：一千年前的注释即点即现」
- 「多维检索：一搜即达的海量文献库」

每个特性的内容结构：
```markdown
### [功能名]：[卖点]

[1-2 段文字描述，包含具体数据]。
[如有具体清单，用列表或表格呈现]。

[配图：该功能的效果截图/信息图]
```

**关键原则：**
- 每个特性独立成节，互不依赖，读者可跳读
- 数据具体化：不写"支持多种实体"，写"18类实体、99,214次标注"
- 标题要有信息量：不写"功能一"，写"事件地铁图：用地铁线路的方式看历史"
- 特性数量 5-8 个为佳，太少显得单薄，太多读者疲劳
- 每个特性配一张配图（功能效果图）

---

### 三、怎么用（分层用户路径）

**任务：按用户类型/使用深度分层，每层给出最简行动路径。**

**这是说明书类文章区别于其他类型的核心特征——它必须按用户分层。**

层级设计模板：
```markdown
## 怎么用

### 一、[最低门槛的用法]（[目标用户]）

[一句话说怎么开始]。

**推荐体验路线：**

| 顺序 | 看什么 | 链接 |
|------|--------|------|
| 1 | [入门级内容] | [链接] |
| 2 | [进阶内容] | [链接] |
| ... | ... | ... |

### 二、[中级用法]（[目标用户]）

[代码/命令]

[用途表格]

| 数据/模块 | 内容 | 适合做什么 |
|----------|------|-----------|
| ... | ... | ... |

### 三、[高级用法]（[目标用户]）

[最简路径 + 代码块]

### 四、[创意用法]（[目标用户]）

| 场景 | 怎么用 |
|------|-------|
| ... | ... |
```

范例分层（shiji-kb）：
1. **直接在线阅读**（零门槛）— 打开链接选章节
2. **搭配AI使用**（研究/创作）— git clone + 喂给大模型
3. **用Skill处理其他书籍**（开发者）— 复用26个方法论
4. **做内容创作**（内容创作者）— 事件/关系/洞见作为素材

范例分层（fojin）：
1. **在线使用**（零门槛）— 打开 fojin.app
2. **Docker 部署**（自托管）— 4 行命令跑起来
3. **开发环境搭建**（贡献者）— 前后端分别启动

**关键原则：**
- 从零门槛到专家，层次分明
- 每层的第一步必须极简（一句话/一行命令）
- 表格展示"能做什么"比长段文字有效得多
- 最低门槛的用法放最前面

---

### 四、价值/洞见（超出功能本身的意义）

**任务：说清楚"结构化/数字化之后会发生什么"——这是把说明书从工具文档提升到内容的关键。**

```markdown
## [标题：暗示超越工具本身的价值]

项目在 [过程] 中意外发现了 [数量] 个 [洞见类型]——
这些不是预设的 [假设]，而是 [方法] 后自然涌现的模式：

- **[洞见1名称]**：[一句话解释]
- **[洞见2名称]**：[一句话解释]
- **[洞见3名称]**：[一句话解释]
- ...

**[一句话总结——当 [条件] 后，[结论]。**
```

范例：
```markdown
## 知识结构化之后，会发生什么

项目在知识提取过程中意外发现了20多个跨章洞察——
这些不是预设的研究假设，而是知识图谱结构化后自然涌现的模式：

- **征服-治理倒转**：打天下的手段恰恰是治天下的障碍
- **推恩令模式**：用分封瓦解诸侯，表面慷慨，实质自动碎裂
- **沉默证据**：秦始皇在位37年，《史记》完全没有记载他的皇后

**当知识被充分结构化后，机器辅助的推理能发现人类难以通过线性阅读察觉的深层模式。**
```

**关键原则：**
- 如果项目没有这类洞见，可以用"扩展路线图"代替
- 洞见用列表呈现，每条一行，不展开
- 最后一句话必须是加粗的总结性判断

---

### 五、扩展路线（可选，如有）

**任务：用表格展示项目未来的规模化路径，制造想象空间。**

```markdown
| 目标 | 规模 | 现状 |
|------|------|------|
| [已完成] | [数据] | ✅ |
| [近期] | [数据] | 近期 |
| [中期] | [数据] | 中期 |
| [远期] | [数据] | 远期 |
```

---

### 六、写在最后（收束）

**不要总结全文。用一句话说核心价值，一句话说愿景，一句引文收尾。**

```markdown
## 写在最后

这个项目的核心价值不是"[表面成果]"——是把"[真正价值]"的完整方法论开源了。

[1-2 句展开]。

最终愿景：**[一句话愿景]。**

> [一句引文/金句]

在线体验：[链接]
GitHub：[链接]

[社群引导（可选）]

---

**调研 & 撰写**：AI（Claude）
**主导 & 审校**：[用户名]
```

## 配图规划

说明书类文章的配图密度高于其他类型——**每个功能特性都应有一张配图**。

| 位置 | 配图内容 | 风格 |
|------|---------|------|
| 核心成果之后 | 数据总览信息图（数字矩阵） | 暗底 |
| 每个功能特性之后 | 功能效果展示（模拟截图/数据可视化） | 暗底/浅底交替 |
| 方法论/管线之后 | 流程图/阶段图 | 暗底 |
| 使用方式表格之后 | 数据目录/使用路径图 | 浅底 |
| 价值/洞见之后 | 洞见摘要信息图 | 暗底 |
| 扩展路线之后 | 路线图 | 浅底 |

配图总数预期：**10-15 张**，比教程类（4-6 张）和深度长文（3-5 张）更多。

## 视觉节奏规范

| 元素 | 频率 | 作用 |
|------|------|------|
| 数据表格 | 每 200-400 字至少一个 | 说明书的主力表达方式 |
| 配图 | 每个功能特性一张 | 功能可视化 |
| 代码块 | 仅出现在"怎么用"部分 | 可复制的命令 |
| 列表 | 洞见/特性/路线部分 | 并列信息 |
| 加粗 | 核心判断、数据点 | 扫读时的视觉锚点 |

**说明书类文章的信息密度 > 教程类 > 深度长文**——读者不是来"阅读"的，是来"查阅"的。表格是最高效的查阅单位。

## 语言风格适配

继承 `writing-style.md` 的基础风格（真实、有态度、不端着），但做以下调整：

| 维度 | 深度长文 | 教程类 | 说明书类 |
|------|---------|--------|---------|
| 书面/口语比 | 70/25 | 50/45 | **80/15** |
| 叙事性 | 高（四幕结构） | 中（阶段叙事） | **低（客观陈述）** |
| 引用密度 | 5-8 个 | 0-2 个 | **0-1 个（仅用于收尾引文）** |
| 金句 | 3-5 句 | 1-2 句 | **1 句（写在最后）** |
| 段落长度 | 2-4 句 ≤150 字 | 1-3 句 ≤100 字 | **1-2 句 ≤80 字** |
| emoji | 0-2 个 | 0-3 个 | **0 个** |
| 人称 | 第一人称叙事 | 第一人称分享 | **第三人称客观** |

**说明书类的口语化最低**——它是工具文档，不是个人分享。语气像产品白皮书，不像朋友聊天。

## 标题技巧

说明书类标题的核心公式：

```
[项目性质] + [一句话定义] + [核心数据]
```

**好标题的特征：**
- 读完标题就知道这个东西是什么、规模多大
- 有具体数字（不写"大量"，写"10万次标注"）
- 不用疑问句或悬念句——说明书不需要勾引

**范例：**
- ✅ 「一个开源的AI《史记》知识库与26个可复用Skill，57万字史记，130篇的10万个实体、3185个事件、7652条关系全部结构化」
- ✅ 「一个开源的佛教数字文献整合平台，它将全球440余个佛教文献数据库汇聚于一处」
- ❌ 「你不知道的史记阅读新方式」（太虚、没数据）
- ❌ 「AI+古籍=？一个让人震惊的实验」（标题党、没信息量）

## 写作清单（出稿前自检）

- [ ] 标题是否包含项目定义 + 核心数据？
- [ ] 第一段是否 150 字内说清楚"做了什么"？
- [ ] Github 和在线体验链接是否在前 300 字内出现？
- [ ] 是否有核心成果总览表（5-8 行）？
- [ ] 每个功能特性是否独立成节、标题有信息量？
- [ ] 使用方式是否按用户分层（从零门槛到专家）？
- [ ] 每层是否有表格展示"能做什么"？
- [ ] 是否有超出工具本身的价值/洞见段落？
- [ ] 表格总数是否 ≥ 6 个？
- [ ] 配图数量是否与功能特性数量匹配？
- [ ] 语气是否客观清晰（第三人称、不讲故事）？
- [ ] 结尾是否简洁（核心价值 + 愿景 + 链接）？
- [ ] 字数是否在 3000-6000 字之间？
