# 公众号内容管线 GZH-Pipeline

> 全流程：素材→选题→写作→审校→排版→配图→朋友圈文案→效果追踪
>
> 核心目标：每篇文章让读者觉得"这文章没白看"

---

## 〇、心法层（每次写之前，先做这件事）

**写的不是文章，是在跟一个具体的人说话。**

放下"我要写一篇好文章"的念头。先想：

1. **想一个具体的人**——他不是"目标用户画像"，是一个你认识的人。
   他可能是：那个被拒赔后不知道怎么办的代理人，那个买了保险但从来不敢看条款的朋友，那个刚入行被客户问住的新人。

2. **问自己：这件事我为什么想讲给他听？**
   不是"因为我需要发一篇文章"。
   是"他真的需要知道这个"。
   是"我知道这件事会帮他少走弯路"。
   是"我想让他看到我没说清楚的那一面"。

3. **写完一段，检查一下**：
   - 这句话他会听进去，还是划走？
   - 我是在跟他说话，还是在"输出内容"？
   - 如果他就坐在我对面，我会这么说话吗？

这不是流程里的一个步骤。是**整个管线的底色**——选题、写作、审校、标题、CTA，每一步都往回调。

## 一、认知引擎（每次生产第一步，硬执行）

写作不是从空白页开始。知识库里已有的东西，应该自动喂到写作现场。

### 1.1 加载个人IP

读取 `02-Areas/知识卡片/个人IP/` 目录下所有卡片，全量加载为风格约束：
- `王永涛-口音与表达.md` → 说话方式、情绪底色、视角
- `王永涛-价值观锚点.md` → 核心信念、立场、刻意不碰的话题
- `王永涛-客群画像.md` → 从哪个具体人群里选"那个朋友"
- `王永涛-风格约束.md` → 硬约束：语气、句长、禁忌、节奏、CTA格式

**如果目录为空**（尚未建立个人IP卡片），使用本文件定义的默认自然风格。

### 1.2 检索思维框架

根据本次选题方向，在 `02-Areas/知识卡片/` 中检索能"重新拆解素材"的卡片：

| 选题类型 | 常搜索的目录 | 卡片示例 |
|---------|------------|---------|
| 保险理赔/条款 | 02-Areas/知识卡片/保险/ | 拒赔六大维度、理赔调查流程 |
| 行业趋势/认知 | 02-Areas/知识卡片/商业/、02-Areas/知识卡片/概念/ | 叙事经济学、复利效应、定位理论 |
| 方法/工具 | 02-Areas/知识卡片/方法论/ | 五层抽象金字塔、万能钥匙思维、Bootstrap自举法 |
| 写作技巧 | 02-Areas/知识卡片/创作/ | 素材六种信号法、爆款内容传播框架、刘润写作心法 |
| 思维/认知 | 02-Areas/知识卡片/认知/ | 镜像神经元、元认知能力、注意力是最宝贵的财富 |

**关键不是提一嘴概念名字，是用这个思维框架去重新拆解素材**。比如找到了「复利效应」——不要写"保险理赔也有复利效应"，而是用它重新组织整篇文章的逻辑：理赔能力的积累如何产生复利。

### 1.3 卡片驱动检查

选题确定后，问自己：
- 这篇文章最深的那句话，能被哪张卡片支撑？
- 素材里的表面信息，用卡片框架重新看，能看出什么不一样的？
- 写完每节，检查：这节有没有被某张卡片"点亮"？（不是生硬套用，是用框架重新理解素材）

## 三、触发规则

| 触发词/场景 | 行为 |
|------------|------|
| `出稿` / `写公众号` / `写文章` / `帮我写文章` | 执行认知引擎 → 出 content-brief → 进入写作 |
| 用户直接发来资料/链接/话题"写成文章" | 自动识别为素材，执行认知引擎 → 出 content-brief → 进入写作 |
| 用户发来 PDF/Word 文件"看看这个" | **先格式归一化**（读 `references/format-normalize.md` 转 Markdown），再执行认知引擎 → 出 content-brief → 进入写作 |
| 用户发来纯文本/无格式文字"写成文章" | 自动执行格式归一化（语义推断结构），再执行认知引擎 → 出 content-brief → 进入写作 |
| `排版` + 文章路径 | Markdown → 公众号 HTML → 配图（连续执行，不中断） |
| `配图` | 仅重新生成配图卡片 + 封面（不重新排版，不重新审校） |
| `做头图` / `做封面` | 单独生成公众号组合封面 |
| `排版+配图` / `全套排版` | 排版 + 插图 + 封面全流程 |
| 用户发来一段文字说"帮我改改" / "润色" | 不触发完整管线，只做文字修改 |

### 素材加载规则

当用户直接发链接、文件或文字要求"写成文章"时：

1. **先读取/抓取内容**作为素材保存到 `drafts/current.json`
2. **格式归一化**：判断输入类型（Markdown/docx/PDF/txt/HTML）：
   - Markdown → 直接下一步
   - docx → 运行 `scripts/extract_docx.py` 转 Markdown
   - PDF → Read 分页读取 + 清噪（页眉页脚/硬断行/标题识别）
   - 纯文本 → 语义推断标题结构
   - 详规见 `references/format-normalize.md`
3. **结构确认**：展示归一化后的结构和标题给用户确认（全自动模式跳过）
4. **执行认知引擎**（第一步）
5. **进入策划阶段**（定方向 → 出 content-brief → 用户确认）
6. **不要问"要不要先看看素材"**——直接干

## 四、完整工作流

```
📌 【心法层】→ 想一个具体的人 + 这件事为什么想讲给他听
     ↓
📄 【格式归一化】→ docx/PDF/txt/HTML 统一转 Markdown（结构确认）
     ↓
🔍 【认知引擎】→ IP卡片 + 知识卡片框架 + 判例库（涉理赔时）写入来源记录 YAML
     ↓
🤖 【策划阶段】→ 确定选题方向 + 生成 content-brief → 用户确认
     ↓
🤖 【调研阶段】→ WebFetch 搜集最新素材（增量保存）
     ↓
🤖 【写作阶段】→ 按选定结构边写边存，每节用卡片重拆
     ↓
👤 【确认阶段】→ 用户审阅
     ↓
🤖 【审校阶段】→ 结构一致 + 事实核查 + 语言自然度 + 冷读测试
     ↓
🤖 【排版+配图阶段】→ 内联 HTML + 插图卡片 + 封面（连续执行，不停顿）
     ↓
🤖 【朋友圈阶段】→ 朋友圈文案
     ↓
🤖 【发布后阶段】→ 提炼知识卡片 → 记录效果 → 更新经验
```

**关键原则**：

- 👤 标记的步骤必须等待用户确认，不能跳过
- 💾 每个阶段完成后必须存档到文件
- 🤖 执行中如遇不确定情况，主动询问用户

## 阶段一：内容策划（一个用户决策点）

### 核心原则

策划不是想标题，是想清楚**这篇文章对谁、有什么用**。

好选题 = 对的人 × 对的时机 × 对的内容

### 第一步：吃透素材

素材里真正有价值的东西，往往不是信息本身，而是信息背后的逻辑和对读者的实际用处。

问自己三个问题，**把答案写下来**：

| 问题 | 说明 |
|------|------|
| **这篇文章到底在说什么？** | 用一句话概括核心信息。不要复述，要提炼。 |
| **这个信息对那个具体的人有什么用？** | 是帮他避坑？提升效率？理解规则？看清趋势？ |
| **我有什么角度能让这个信息更好懂？** | 能用到哪张知识库卡片？能用什么例子让它不"干"？ |

确定后，直接进入下一步——生成 content-brief。

**文章类型说明**：

| 类型 | 字数 | 结构倾向 | 配图数 | 适合什么 |
|------|------|---------|-------|---------|
| **短文** | <1500字 | 并列式为主 | 2-3张 | 讲一个具体方法/观点/案例 |
| **中长文** | 1500-2500字 | 递进式或并列式 | 4-6张 | 讲一个完整的分析/对比/趋势 |
| **深度长文** | >2500字 | 递进式为主，可嵌套 | 6-10张 | 多维深度分析/多案例串联 |

在策划阶段就要确定文章类型，提前让用户知道工作量。

---

### 标题怎么写

好标题的标准：**准确 > 吸引**。先保证读者看完知道文章在说什么，再考虑吸引他点进来。

写标题时自查：
- 读者看完标题，能猜出文章大概在说什么？（猜不出，改）
- 文章的内容对得起标题的承诺吗？（对不上，要么改标题要么改内容）
- 用具体代替抽象——有数字吗？有名字吗？有结果吗？

不要用夸张、恐吓、悬念制造虚假吸引力。让事实自己说话。

**5 种基础标题结构**（选最自然的一种）：

| 结构 | 说明 | 例 |
|------|------|-----|
| **陈述结论型** | 直接说核心结论 | "经济越不好，保险代理人越该换个活法" |
| **指出问题型** | 说清楚文章在讲什么事 | "被叫了这么多年黄牛，原来我们搞混了三件事" |
| **具体数字型** | 数据让标题更可信 | "6个AI工具，让保险代理人的工作效率翻倍" |
| **对比并列型** | 两件事放一起自然产生张力 | "职工医保 vs 居民医保，待遇差距有多大" |
| **场景切入型** | 用一句话把读者拉进画面 | "客户跟我说：买保险的时候你可不是这么说的" |

**8 种爆款标题结构**（当选题本身有爆款潜质时，用以下结构放大）：

| 结构 | 公式 | 例 |
|------|------|-----|
| **结果前置** | 数字/结局甩前面，让人想知道怎么做到的 | "被拒赔15万，凭一张挂号单拿回来了——保险公司沉默了一星期" |
| **颠覆认知** | "你以为X，其实Y"——打破读者的固有认知 | "你以为客户在跟医生聊天，其实在帮保险公司写拒赔理由" |
| **对比反差** | 两件事放一起，短句对短句，制造张力 | "合同里最短的一句，拒赔时最好用的一句" |
| **代入视角** | 第一人称说话，让读者觉得"这说的是我" | "卖了5年医疗险，我从来没认真看过这4个字" |
| **反常识+损失** | 告诉读者做错一步会损失什么 | "客户一出险，先别急着报案——做错这一步，理赔少一半" |
| **情绪击中** | 直接打到读者最无力的那个时刻 | "保险人最无力的时刻：你讲的每一句都对，但客户听不进去" |
| **对话开头** | 用一句对话把读者拉进冲突现场 | "保险说一分不赔。我问他：挂号等于确诊吗？" |
| **递进恐惧** | 一件小事→被忽略→严重后果，层层递进 | "体检单上一项指标高了一点点，代理人没当回事，客户被拒赔了" |

**爆款标题 ≠ 标题党。** 以上结构只是让好内容更容易被点开——正文必须兑现标题的承诺。

---

### 选题质量检查

出完选题后，逐项检查：

- [ ] 每个标题让人一眼看懂文章在说什么？
- [ ] 每个选题标注了"他在愁什么"？（具体的人 + 具体的困境）
- [ ] 每个选题的"错误认知"真实存在？（不是编的、不是夸大的）
- [ ] 素材真的能撑起这些选题？
- [ ] 每个选题标明了文章类型（短文/中长文/深度长文）？
- [ ] 标注的卡片真的能用来重新拆解素材？还是只是名字好听？

---

### 用户决策方式

- **选择**：说"1"、"2"、"3"
- **修改**：说"把2的标题再想想"、"1换个角度"
- **重新生成**：说"再来几个"、"换一批"

### 第三步：确定文章类型和骨架（用户选择后执行）

用户选择选题后，**明确这篇文章的定位**，写入 `drafts/文章定位.json`：

```json
{
  "selected_topic": "选题一/二/三",
  "article_type": "短文 | 中长文 | 深度长文",
  "target_word_count": "<1500 | 1500-2500 | >2500",
  "suggested_structure": "递进式 | 并列式 | 清单式 | 故事型",
  "key_sections": ["节1主题", "节2主题", "节3主题"],
  "required_insight": "这篇文章的核心洞见是什么（一句话）",
  "activated_cards": ["用到的知识库卡片列表"],
  "target_person": "这篇文章写给哪个具体的人（他在愁什么）"
}
```

**结构选择树**：

```
Q1：文章核心是什么？
  ├─ 讲一个方法/操作 → 清单式/并列式
  ├─ 讲一个观点/认知 → 递进式
  ├─ 讲一个故事/案例 → 故事型
  └─ 做对比分析 → 并列式

Q2：文章类型？
  ├─ 短文 → 递进式（2-3节）或并列式（3-4个并列点）
  ├─ 中长文 → 递进式（3-4节，层层深入）
  └─ 深度长文 → 递进式嵌套（大节套小节）

Q3：要不要打破常规结构？
  ├─ 清单型 → 开头说明为什么需要，每个工具独立成段
  ├─ 纯故事型 → 全程叙事，分析和钉子融入叙事
  └─ 政策解读 → 先给政策核心，再逐条展开
```

确认后将定位写入文件，进入调研阶段。

## 阶段二：调研（🤖 AI 执行）

用户选择方向后，进行网络调研。**不要等所有信息搜完再整理——边搜边存**。

### 调研前先定向

开始搜索前想清楚：这篇文章最需要什么素材？数据？案例？法规原文？操作步骤？

带着明确问题去搜，不是泛泛浏览。

**参考文章类型调整调研深度**：
- **短文**：1-2 个核心数据/案例即可，不要过度搜索
- **中长文**：每个大纲节点至少 1 个硬素材
- **深度长文**：需要多源交叉验证，数据/案例/法规均需覆盖

### 调研方法（按优先级）

#### 优先级 1：WebFetch 直抓目标网站

根据主题类型，直接去对应网站获取内容（无需通过搜索引擎）：

| 主题类型 | 目标网站 |
|---------|---------|
| 科技趋势/AI | 36kr、虎嗅、钛媒体 |
| 保险/金融 | 中国保险行业协会网站、财新保险频道 |
| 政策解读 | 政府官网、国家部委网站 |
| 综合热点 | 财新、澎湃、凤凰网 |

#### 优先级 2：WebSearch

使用 WebSearch 工具搜索关键词补充信息。

#### 优先级 3：CDP 浏览器（fallback）

当 WebFetch 和 WebSearch 都拿不到干净内容时，使用 web-access skill 的 CDP 浏览器。详见 `references/web-research.md`。

### 调研要求

- **增量保存**：每获取一批信息，立即追加写入 `drafts/调研简报.md`
- 不要等全部完成才写文件，防止中断丢失
- 输出包含：关键事实、数据来源链接、待确认问题

### 调研停止判定（绑定大纲，不计数）

**每次搜完一轮后，问自己三个问题：**

| 问题 | 继续还是停 |
|------|-----------|
| **这份材料能撑起这篇文章吗？** → 翻开大纲，检查每个小节是否至少有一个硬素材（数据/案例/法规/引述） | 全部有 → **停**。有的节点缺 → 只针对缺的那个节点再搜一轮 |
| **已有的素材够具体吗？** → 有具体的数字、名字、判例号、政策名称？ | 够具体 → **停**。不够 → 再搜一轮补齐细节 |
| **还缺什么新的视角？** → 现有素材是不是都是一个来源同一个角度？ | 角度已丰富 → **停**。需要另一个视角佐证 → 搜 |

**注意区分"想要"和"需要"**。一般公众号文章，2-5条有效数据/案例即可支撑。如果搜了4轮以上还有缺口，进入写作阶段，让用户在确认阶段补充。

### 调研深度检查（素材质量确认）

- **够新鲜吗？** 数据/案例是近期的？
- **够具体吗？** 有具体的数字、名字、判例号、政策名称？
- **缺什么？** 明确列出来——用户可能有补充。

### 输出格式

```markdown
# 调研简报

## 关键事实
- [事实1]
- [事实2]

## 数据来源
- [来源1]：[链接]
- [来源2]：[链接]

## 待确认问题
- [问题1]
- [问题2]
```

## 阶段三：写作

### 核心原则

**一篇文章的价值 = 信息量 × 洞见深度 × 阅读体验**

- **信息量**：新事实、新数据、新案例
- **洞见深度**：不是"信息集装"，而是让读者读完有"原来是这样"的恍然大悟
- **阅读体验**：读起来顺、不累、有节奏

### 风格规范

**已加载个人IP卡片时**，逐条执行卡片中的硬约束（见 `02-Areas/知识卡片/个人IP/王永涛-风格约束.md`）：
- 核心语气：简洁、直指要害、不装
- 句长：15-25字为主，最长40字
- 禁忌：说教感、销售感、学术感
- 节奏：信息段→钉子段→新信息→钉子段，节间无过渡
- Emoji：极少用
- CTA：自然嵌入内容逻辑

**无个人IP卡片时**，使用以下默认风格：

- **语气**：像朋友聊天，不是新闻联播
  - 口语化到可以直接念出来
  - 用"我"、"你"建立对话感，不用"笔者"、"本文"
  - 能说"干"就不说"执行"，能说"搞定"就不说"完成实施"
- **段落**：极短段落，1-3 句，大量单句成段制造节奏
- **开头**：第一句话必须有画面感或具体场景，零铺垫
  - ✅ "上个月一个代理人找我，说客户被拒赔了，保险公司说他买的重疾险不属于恶性肿瘤。"
  - ✅ "春节不想让孩子刷短视频，就说：来，咱自己做个植物大战僵尸。"
  - ❌ "在数字化浪潮席卷全球的今天…"
- **金句**：用判断句收束观点，句式简洁有力
- **表格**：能用表格就不用段落，表格是核心表达工具
- **示例 > 概念**：先给例子，再总结规律

**禁用词组**（不论有无IP卡片，均适用）：
- "在...的背景下"、"随着...的发展"
- "不可否认"、"值得注意的是"
- "本文将介绍"、"笔者认为"、"众所周知"
- 超过 3 个"的"连用的句子 → 拆短

### 卡片驱动写作

写作不只是组织信息，是用思维框架重新理解素材。

**每写一个小节，问自己**：
1. 这个节的核心信息是什么？
2. 有没有一张知识库卡片能帮我换一个角度重新看这个信息？
   - 比如「五层抽象金字塔」→ 能否从这个案例中提炼一个原则？
   - 比如「复利效应」→ 这个做法长期积累的复利是什么？
   - 比如「叙事经济学」→ 这个观点能否用一个故事来传播？
3. 卡片不是用来"引用"的，是用来"重新拆解"的。如果只是提一嘴概念名字——删掉。

**卡片融入深度检查**（写完后自查）：
- 核心洞见是不是被卡片"点亮"了？
- 有没有至少一处，用卡片框架让读者产生"原来可以这么看"的恍然大悟？

### 洞见深度要求

- **每篇文章至少有一个让读者"没想到"的观点**
  - 不是重复常识，而是对常识的再审视
  - "X 不一定是 Y 的原因，真正的原因是 Z"
  - 把表面现象归类，提炼出规律
- **融合多篇资料精华，不是堆砌信息**
  - 看了 A、B、C 三篇资料，总结出 D——这才是你的价值
- **吸引力三选一（最好占两个）**：
  - **有用**：读者看完能做什么
  - **有共鸣**：读者觉得"说的就是我"
  - **有趣**：让人想看完

### 文章结构：按类型匹配骨架

#### 通用呼吸节奏（所有文章）

```
钩子（5-10%）
  ↓  一个具体场景/对话/数据，零铺垫直入
框架（10-15%）
  ↓  一句话点题——为什么这事值得说、和读者有什么关系
展开（65-75%）
  ↓  2-4个小节，每节一个核心点
落地（5-10%）
  ↓  一句话收束全文
```

#### 按文章类型选骨架

**📄 短文 (<1500字) — 并列式 或 精简递进式**

适合：讲一个具体方法、一个观点、一个案例

```
钩子：一个具体场景/问题（1-2段）
↓
框架：一句话点题（1段）
↓
小节1：核心内容（3-5段）
小节2：为什么有效/延伸分析（2-4段）
小节3：怎么做/对你有什么用（2-4段）
↓
落地：金句收束（1段）
```

**📄 中长文 (1500-2500字) — 递进式 或 并列式**

适合：完整分析、对比解剖、趋势判断

```
递进式（观点层层深入）：
  钩子：让人停下来想想的场景
  ↓
  框架：一句话亮观点
  ↓
  小节1：现象/数据——问题有多严重
  小节2：深层原因——为什么会出现
  小节3：影响/启示——对读者意味着什么
  小节4：怎么做——可操作的建议
  ↓
  落地：收束到行动或判断

并列式（多件事逐一拆解）：
  钩子：一个具体的问题或困惑
  ↓
  框架："不止一件事"
  ↓
  小节1：第一件事（具体+分析+钉子）
  小节2：第二件事（具体+分析+钉子）
  小节3：第三件事（具体+分析+钉子）
  ↓
  落地：串联三件事，收束到一个判断
```

**📄 深度长文 (>2500字) — 递进式嵌套**

适合：多维深度分析、多案例串联

```
开场（大问题或电影场景）
↓
第一部分：框架搭建——概念命名/定义问题
  小节1.1  场景/引子
  小节1.2  核心概念
  小节1.3  为什么重要
↓
第二部分：案例/论证展开
  小节2.1  案例A（故事→数据→判断）
  小节2.2  案例B（故事→数据→判断）
  小节2.3  规律提炼
↓
第三部分：方法论/原则
  小节3.1  原则一
  小节3.2  原则二
  小节3.3  原则三
↓
落地：回到开头问题，闭合环形 + 金句终钉
```

#### 每个小节内部的微节奏（所有文章通用）

| 拍子 | 做什么 |
|------|--------|
| **切入** | 一个具体场景/数据/引述，让读者一秒知道这节在说什么 |
| **分析** | 展开逻辑。为什么是这样？背后原因/数据/法规/案例 |
| **钉子** | 一句判断句收束，让这节的观点被记住 |

每个小节自己就是一次吸气（切入）→ 屏息（分析）→ 呼气（钉子）。

节和节之间**不用过渡句**。一节钉完，下一节直接切新场景。节奏靠观点推进。

#### 特殊类型

| 类型 | 结构调整 |
|------|---------|
| **清单型**（"6个AI工具"） | 开头说明为什么需要，每个工具独立成段，工具间不需要过渡 |
| **纯故事型** | 全程叙事，分析和钉子融在叙事里，不单独拆节 |
| **政策解读型** | 先给政策核心一段，再逐条展开。每条的切入是"这条在说什么" |

### 标题规范

写作阶段执行的标题标准——与策划阶段的标题规范一致。写正文标题时用同样的公式和自我审查标准。

**写作阶段额外强调**：如果你写出来的标题和策划阶段的大纲标题差异很大，先审视内容是否跑偏了。标题始终是内容的承诺。

### 边写边存

每个小节写完后，追加写入文章 `.md` 文件，不要等全文写完才存。

### 写作节奏

1. **信息/叙事段落**（中长）→ **钉子段**（单句成段）→ **新信息** → **钉子段**
2. 每 300-500 字至少一个表格或数据点
3. 每节结尾有一句加粗判断句收束

## 阶段四：审校

初稿完成后，**必须经过审校才能进入排版**。

### 第一步：结构一致性检查

对照 `drafts/文章定位.json` 检查结构是否匹配：

- [ ] 实际字数是否匹配选定的文章类型范围？
- [ ] 小节数是否匹配选定的骨架（短文2-3节 / 中长文3-4节 / 深度长文3+部分）？
- [ ] 每节是否遵循"切入→分析→钉子"微节奏？
- [ ] 节与节之间是否有过渡词？（应该有，删掉）
- [ ] 开头是否为具体场景/数据/对话，而非背景铺垫？
- [ ] 结尾是否有加粗判断句收束？
- [ ] 核心洞见是否兑现了策划阶段的承诺？

不通过 → 标注问题位置，返回写作阶段修改。

### 第二步：事实核查 + 信息质量 + 洞见达标

- 数据是否准确（日期、数字、比例、人名）
- 逻辑是否有漏洞
- 信息是否新鲜、有据可查（不是道听途说）
- **洞见检查**：这篇文章有让人"没想到"的观点吗？如果没有，需要补一个
- **洞见密度**：中长文/深度长文至少3处"认知反转或深度判断"，短文至少1处
- **卡片融入检查**：标注的卡片真的用来重新拆解素材了吗？还是只提了一嘴名字？
- **标题检查**：标题是否兑现了选题承诺？标题和内容对得上吗？

### 第三步：语言自然度

**读一遍自己的稿子，读出声来**——哪里拗口改哪里。

**AI 腔检测清单**（逐句扫）：
- [ ] 是否出现"在...的背景下"、"随着...的发展"、"值得注意的是"？
- [ ] 是否出现"本文"、"笔者认为"、"总而言之"、"众所周知"？
- [ ] 是否有超过 3 个"的"连用的句子？（拆短）
- [ ] 是否有"应该说"、"事实上"等冗余引导词？
- [ ] 是否有"接下来让我们"、"下面我们来"等导航体？
- [ ] 是否有"你不妨"、"你或许会问"等假互动？
- [ ] 是否有"总之一句话"、"归根结底"、"说到底"？（删掉，直接说）
- [ ] 是否有对称对照句式如"不是X，是Y"/"不是X的差距，是Y的差距"？（打破对称——"别误会，我说的不是X。是用Y——..."）
- [ ] 是否有"你以为的X，实际上是Y"？（认知冲突本身没问题，但不能篇篇用同一句式）
- [ ] 排比句是否超过两个分句？（删到两个，或拆散）
- [ ] 段落开头是否堆砌"首先/其次/最后/同时/此外"？（直接删掉）
- [ ] 过渡句是否太多？（"说到这里""接下来我们看看""综上所述"——删掉，直接切下一节）
- [ ] 金句后面是否跟了解释？（金句说完就走，别解释）

**人话检测清单**（全文扫）：
- [ ] 开头第一句话有没有画面感或具体场景？还是从大背景起笔？
- [ ] 段落是否太长了？（超过 5 句的段落必须拆开）
- [ ] 有没有"说教感"？（"你应该"、"你必须" → 改成"我发现"、"我的做法是"）
- [ ] 有没有一句能让人记住或想截图的判断句？
- [ ] **心法检查**：读完全文，能感觉到是在跟一个具体的人说话吗？还是感觉在"发文章"？

### 第四步：冷读测试

抛开"作者"身份，假装自己是那个"具体的朋友"：

> 打开这篇文章，第一句能让我停下来吗？
> 读到中间有想划走的冲动吗？
> 读完最后一句，我会觉得"这人懂我"吗？

如果任何一个答案是"不确定"或"不会"——标注问题位置，回去改。

### 审校输出

```
✅ 结构一致性检查：通过（或：发现 X 处与选题定位不符，已修正）
✅ 事实核查：通过（发现 X 处问题，已修正）
✅ 洞见密度：达标（核心洞见 X 处，质量符合要求）
✅ 卡片融入：通过（X张卡片起了实际作用）
✅ 语言润色：通过（调整 X 处）
✅ 冷读测试：通过（如不通过，注明问题段落）
```

## 阶段五：排版 + 配图（连续执行）

> 排版和配图是两个连续步骤，排完配跟，中间不询问用户。"排版"触发即执行排版→配图全流程。

### 步骤 0：读取公众号长阅读尺度

排版前必须读取 `references/wechat-scale.md`，先判断本文使用哪条公众号 scale：

| 内容类型 | 默认 scale |
|---|---|
| 观点文、案例拆解、行业趋势、方法论、个人复盘 | 深度编辑风 |
| 教程、工具盘点、操作指南、AI 工作流、清单 | 信息密度风 |

公众号的核心是长阅读，不是海报流。正文要耐读，图片要解释关系，颜色要克制。

如果已经生成 `.claude/drafts/style-brief.json`，优先读取其中的 `route`、`visual_profile` 和 `image_plan`；没有则按 `wechat-scale.md` 自动决策。

### 步骤 0A：选择排版主题

排版前必须读取 `references/theme-index.md`，按以下流程选择主题：

1. **检查文章定位**：读取 `drafts/文章定位.json`（或 `drafts/current.json`），获取 `content_type`（内容类型）和文章情绪
2. **匹配主题**：按 `references/theme-index.md` 的"选择流程"表自动匹配建议主题
3. **用户确认**：展示推荐主题和备选，让用户选择（或说"1""2""3"确认）
4. **读取主题文件**：根据选定主题，读取对应组件库文件（如 `references/theme-deepblue.md`）

**主题速查：**

| 主题 | 气质 | 适用内容 |
|------|------|---------|
| **深蓝商务**（默认） | 理性、专业 | 案例文、解读文、保险条款、数据分析 |
| **暖橙故事** | 温暖、亲和 | 个人经历、客户故事、成长感悟 |
| **科技暗色** | 现代、冲击 | 观点文、工具文、方法论、评测 |

用户说"直接排""一键排版"时跳过选择，按内容类型自动匹配默认主题。

**自动匹配规则：** 案例文/解读文 → 深蓝商务；故事文 → 暖橙故事；观点文/工具文 → 科技暗色。

### 步骤 A：排版 → 内联生成 HTML

直接在当前对话中生成 HTML 排版文件，不需要调用外部脚本。

**关键规则：所有样式必须写成内联（`style="..."`），不能写在 `<style>` 块里。** 因为用户复制到公众号编辑器时，`<style>` 里的样式会丢失，只有内联样式能保留。配图卡片也要用内联样式表格嵌入在正文对应位置，而不是单独放个"配图建议"标注。

**排版规范** — 分三步完成：Markdown 预处理 → HTML 生成（含关键词标记）→ 排版校验

---

#### 第一步：Markdown 预处理

排版前先处理 Markdown 原文，做两件事：章节编号 + 目录提取。

**① 章节自动编号**

扫描 `##` 标题，按顺序添加编号 `01/02/03…`（上限 20 章）：

| 章节位置 | 编号格式 | 示例 |
|---------|---------|------|
| 第 1 节 | `01` | `## 01 为什么拒赔` |
| 第 2 节 | `02` | `## 02 保险公司的逻辑` |
| ... | ... | ... |
| 末章（结语类） | 特殊标记 | `## ∞ 写在最后` 或 `## 结语` |

**末章判定**：末章 `##` 标题包含「结语/总结/写在最后/尾巴/结尾/尾声/后记/END/写在后面」时，编号改为 `∞`。不包含则继续用数字编号。

处理后将编号后的标题清单暂存，供目录提取使用。

**② 目录提取**

从前 3 个已编号 `##` 标题生成导读目录 HTML，插入文章开头（标题之后、正文之前）：

```html
<section style="background: 主题引用背景色; border-radius: 8px; padding: 20px 24px; margin: 20px 0; font-family: 主题字体;">
  <p style="font-size: 14px; color: 主题次要色; letter-spacing: 2px; margin: 0 0 12px;">📖 本篇文章看点</p>
  <p style="font-size: 15px; color: 主题标题色; margin: 0 0 8px; font-weight: 600;">01 {章节1标题}</p>
  <p style="font-size: 15px; color: 主题标题色; margin: 0 0 8px; font-weight: 600;">02 {章节2标题}</p>
  <p style="font-size: 15px; color: 主题标题色; margin: 0; font-weight: 600;">03 {章节3标题}</p>
</section>
```

目录组件仅在前 3 个章节足够概括文章全貌时使用。如果文章不足 3 个 `##` 节，不生成目录。

**③ 配图插入点标记**

保留原文的 `[配图：描述]` 标记，供配图阶段定位插入点。

---

#### 第二步：生成 HTML（含主题样式 + 关键词下划线）

此时步骤 0A 已选定主题，已读取对应主题文件。按以下流程生成 HTML：

**① 读取主题文件的「正文排版 CSS」章节**，获取全部内联样式 CSS

**② 遍历正文段落 → 标注关键词下划线**

对每个 `<p>` 段落，自动识别 1-3 个关键短语，包裹下划线样式：

| 识别优先级 | 找什么 | 示例 |
|-----------|-------|------|
| 1（最高） | 核心判断句、结论性短语 | "拒赔的核心原因"、"最大的误区" |
| 2 | 关键数据、数字+单位 | "15万"、"83%"、"6个月" |
| 3 | 专有名词、概念名 | "如实告知义务"、"等待期条款" |

**下划线样式**：从主题文件的「下划线/标记风格」取色值，格式为：
```html
<span style="border-bottom: 2px solid {主题强调色}; font-weight: 600;">{关键短语}</span>
```

**规则**：
- 每段最多标 3 处，最少 1 处（实在没有可标点的段落除外）
- 同一短语不重复标记（第一次出现时标）
- 下划线短语 4-15 字为佳
- 不标整句话，只标核心词
- 代码块、引用块内部不标
- 标题内部不标（标题本身的加粗和编号已足够）

**③ 按主题 CSS 生成完整 HTML**

将主题 CSS 逐元素应用为内联样式，组装完整的 `<section>` 容器。使用 `wechat-scale.md` 的密度判定调整段落间距和组件密度。

**主题 CSS 速查（完整 CSS 在对应主题文件中）：**

| 元素 | 深蓝商务 | 暖橙故事 | 科技暗色 |
|------|---------|---------|---------|
| 背景色 | `#FFFFFF` | `#FAFAF7` 米白 | `#0D0D0D` 极黑 |
| 正文色 | `#1A1A1A` | `#3D2E1E` 深茶 | `#E0E0E0` 浅灰 |
| 标题色 | `#102542` | `#5C3D2E` 栗棕 | `#FFFFFF` |
| 强调色 | `#F77F00` 橙 | `#C4814A` 琥珀 | `#00D4FF` 青 |
| 正文字体 | Noto Sans SC | Noto Serif SC（衬线） | Noto Sans SC |
| 引用背景 | `#F7F9FC` | `#F5F0EB` | `#1A1A2E` |
| 表头背景 | `#102542` | `#5C3D2E` | `#00D4FF`（白字变黑字） |
| 下划线色 | `#F77F00` | `#C4814A` | `#00D4FF` |

#### 第三步：排版校验

HTML 生成完成后，运行校验脚本检查排版合规性：

```bash
PYTHONIOENCODING=utf-8 python "${SKILL_DIR}/scripts/validate_gzh_html.py" "公众号/{年份}/{月份}/{文章标题}_preview.html"
```

校验内容：
1. **平台禁用项检查**：`<style>` / `<script>` / `<div>` 不应出现，`position:fixed/absolute/sticky` 不应使用
2. **样式内联检查**：所有样式通过 `style=""` 属性应用
3. **半角标点警告**：正文中的半角逗号/句号/引号应改为全角（代码块和 URL 除外）
4. **配图卡片检查**：`_preview.html` 中不应有下载按钮

输出格式：

```
✅ 排版校验通过（0 ERROR, 0 WARNING）
或
❌ 发现 2 个 ERROR，请修复后重新校验
  ERROR 1: ...
  WARNING 1: ...
```

ERROR 清零后方可进入发布流程。先修 ERROR，再修 WARNING。

---

**正文字号**：默认 16px，标题 h2=19px / h3=17px / h1=24px

**文件名**：`{文章标题}_preview.html`，存放在 `公众号/{年份}/{月份}/` 目录。

### 发布流程

1. 浏览器打开 `_preview.html` → 确认配图位置和内容
2. 浏览器打开 `_配图下载.html` → 逐张点击下载 PNG
3. 将 PNG 上传到公众号图片库
4. 回到 `_preview.html` → 全选复制正文 → 粘贴到公众号编辑器
5. 在对应位置插入已上传的配图
6. 微调格式（如有必要）

### 步骤 B：配图 — 插图卡片 + 封面

配图分两个文件存放：

- **`_preview.html`**：配图卡片内嵌于正文中展示位置，**不带下载按钮**（保持复制干净）
- **`_配图下载.html`**：所有配图卡片纵向排列，每张带下载按钮，集中下载 PNG

### 配图生成流程

1. **读取尺度与模板库**：读取 `references/wechat-scale.md` 和 `references/illustration-guide.md` 获取长阅读尺度、配图数量、5 种卡片模板和下载机制
2. **确定插入点**：
   - 如果文章 markdown 中有 `[配图：描述]` 标记 → 在对应位置生成卡片
   - 如果没有标记 → 在每个 `<h2>` 小节后评估是否适合插入配图
3. **选择卡片类型**：根据该位置的内容特征，按以下指引选择：

| 内容特征 | 推荐卡片类型 |
|---------|------------|
| 有步骤/流程/时序 | 类型 1：流程图 |
| 多个并列概念/功能点 | 类型 2：网格卡片（2x2） |
| A vs B / 新旧 / 有无对比 | 类型 3：暗底对比卡 |
| 多维评分/权重/数据对比 | 类型 4：柱状图 |
| 两种视角/投入产出/方案对比 | 类型 5：分栏布局 |

4. **填充卡片内容**：根据上下文提取数据填入模板的 `{{变量}}`，ID 使用 `illu-N`（N 从 1 递增）
5. **生成两个文件**：
   - **预览版**：卡片 HTML 去掉下载按钮，插入到 `_preview.html` 对应位置
   - **下载版**：创建 `_配图下载.html`，包含 html2canvas CDN + `downloadCard()` 函数 + 所有卡片（带下载按钮）纵向堆叠，页脚注明"打开预览 HTML 确认位置，在本页下载 PNG"
6. 在 `_preview.html` 页脚加入指向下载页的提示

### 配图数量参考

| 文章长度 | 参考配图数 |
|---------|----------|
| 深度长文（>2500字） | 6-10 张 |
| 中等长度（1500-2500字） | 4-6 张 |
| 短文（<1500字） | 2-3 张 |

**原则：每张配图都传递独立信息，不为配而配。若图片只是装饰，宁可不插。**

### 头图（封面）——长方形 + 方形组合封面

封面单独生成 `_封面.html` 文件（不嵌入 preview）。

**组合尺寸**：左侧长方形 `900×383px`（16:9）+ 右侧方形 `383×383px`（1:1），总尺寸 `1283×383px`

**导出**：用 html2canvas scale=2 导出，`2566×766px` PNG

**设计规范**：

| 副标题 | 26-28px，rgba(255,255,255,0.6) | — |
| 标签 | 1-3个，圆角胶囊形 | 不要放摘要/数据/标签 |

### 配图 QA 检查

配图生成完成后、进入朋友圈阶段前，执行 `references/illustration-qa-checklist.md` 的三类检查：

**① 内容侧** — 逐张卡片核实：
- 卡片类型是否匹配内容特征？（流程内容→流程图，对比内容→对比卡）
- 卡片的数字/结论与正文一致？
- 金句卡提取的是核心判断句，不是过渡句？
- 没有"为配图而配图"的装饰卡？

**② 视觉侧** — 整体扫：
- 强调色与当前排版主题一致？
- 相邻卡片风格交替了？（不连续 3 张同底色）
- 全文配色统一？

**③ 格式侧** — 复用已有清单：
- 预览版无下载按钮
- 卡片 ID 唯一连续
- html2canvas CDN 已引入

**QA 输出格式：**

```
内容侧：✅ 通过（类型匹配 X/X · 信息准确 X/X）
视觉侧：✅ 通过
格式侧：✅ 通过
总体：✅ 配图 QA 通过 → 进入朋友圈阶段
```

**通过标准**：内容侧 + 格式侧必须全部 ✅。视觉侧 ⚠️ 可选修，不阻塞。任一内容/格式项 ❌ → 修复后重新 QA。

## 阶段六：朋友圈文案

根据文章核心内容，生成朋友圈文案。目的不是卖文章，是 **让人觉得"这家伙有东西，值得关注"**。

### 风格

- 像发给朋友看的状态，不是营销推文
- 第一句话要抓人（悬念/成果/反转/吐槽）
- 3-5行正文，不超过6行
- 可以带1-2个emoji，不堆砌
- **不要放全文链接**——引导私聊或评论区互动
- 最后一句暗示"文章里还有更多"

### 参考写法

**文章讲了一个规律或道理**
> 被叫了这么多年"黄牛"，我终于搞清楚了。
> 保险公司说的"黄牛"和真正的理赔协助，根本是三件事。
> 明天用一篇文章讲清楚，看完以后你再也不会被吓唬了。

**文章教了一个方法或操作**
> 理赔多拿30%，就靠一个动作。
> 6个AI工具+具体怎么用，写着写着发现我之前好几个工具都用错了。
> 明天推。

**文章讲了一个故事或案例**
> 今天有个客户跟我说："买保险的时候你可不是这么说的。"
> 我一愣，然后发现他说的对。
> 写完了，挺长的，但最后几段我自己写的时候都在想：原来我也一直这么想的。
> 明天发。

**文章分析了趋势或政策**
> 2026版人身险产品负面清单出来了。
> 看完在想——这跟代理人有直接关系。
> 整理了三个最相关的点，明天发。

### 检查

写完朋友圈后问自己：
- 如果自己刷到这条，会停下来看完吗？
- 有没有"发广告"的感觉？（有就重写）
- 最后一句有没有给人一个互动的理由？

### 输出格式

```markdown
📋 朋友圈文案

---
[文案内容]

---
```

## 故障处理

| 问题 | 处理 |
|------|------|
| 素材不足 | 出选题时注明"素材有限"，建议用户补充 |
| 搜索乱码 | 切换到 CDP 浏览器或用 WebFetch 直抓目标网站 |
| 排版脚本报错 | 改用内联 HTML 生成 |
| WebFetch 抓不到内容 | 用 CDP 浏览器 fallback |
| 导出图片模糊 | 检查 html2canvas scale=2，浏览器缩放 100% |
| 下载按钮无效 | CDN 可能被墙，配图仍可见——换网络或手动截图兜底 |
| 文章定位.json 缺失 | 重新根据选题内容确认文章类型和骨架 |
| 个人IP卡片为空 | 使用本文件默认自然风格，提示用户可建立个人IP卡片 |
