# Inbox 输入加工

把未处理输入转化为一项可确认的路由决定和资产草稿。先判断，后写入；保留原始来源，不静默删除、移动或扩大结论。

## 1. 准备上下文

1. 读取知识库根目录 `README.md`，确认使命、目录职责和真实路径。
2. 若存在 `.claude/skills/knowledge-base-collab/SKILL.md`，读取其中的协作与安全规则。
3. 读取用户指定的 Inbox 文件；未指定时列出候选，不自行处理整个目录。
4. 仅在需要判断调用场景时读取相关项目、主题地图或卡片，避免无目的扫描全库。
5. 处理知识卡片时读取 `references/knowledge-card-template.md`；判断去向时读取 `references/routing-rubric.md`。

## 2. 判断输入

依次回答：

1. 是否与当前知识库使命或一个真实项目有关？
2. 它是原始观察、外部资料、单案事实，还是本人判断？
3. 是否具备可追溯来源？哪些信息仍待核验？
4. 能否跨项目复用并帮助判断、决策或行动？
5. 当前是否有项目会直接使用它？
6. 是否含隐私、敏感业务材料、高风险结论或对外承诺？

只给一个主要去向。必要时补充一个次要动作，例如“先用于项目，项目结束后再提炼卡片”。

## 3. 生成加工建议

按 `references/output-format.md` 输出：

- 原始文件与一句话摘要；
- 内容角色与来源状态；
- 建议去向、目标路径和判断依据；
- 可选资产草稿；
- 需要人工确认的问题；
- 对原文件的建议处理方式。

若证据不足，只生成“待验证草稿”，不得写成稳定结论。若与使命无关，说明继续保留、进入待清理隔离区或不处理的理由，不为了清空 Inbox 强行制卡。`04-Archive/` 已退役，不作为路由目标。

## 4. 人工确认后写入

默认停在加工建议，不修改文件。只有用户明确确认或在请求中明确说“按建议直接写入”时才执行写入。

写入时：

1. 创建目标资产并保留原始来源链接；需要长期追溯的原始材料进入 `03-Resources/`，并区分外部资料、个人输入和系统历史；
2. 检查标题、`type`、状态、来源和适用边界；
3. 若需要移动原文件，先说明目标位置；只在用户已确认时移动；
4. 不覆盖同名文件，遇到冲突先比较并请求决定；
5. 写入后报告新文件、原文件状态和仍待处理事项。

即使用户允许直接写入，也不得自动删除原始输入、处理未脱敏隐私、伪造来源或替用户作高风险最终判断。

## 5. 批量模式

- 一次默认处理不超过 10 条输入。
- 先输出路由清单，再按用户确认的条目生成资产。
- 不把多条不同判断合并为一张卡片，除非它们共同支持同一个明确结论且来源分别保留。
- 已被资产承接的原始输入仍可进入来源层；“已处理”只更新集合状态和连接，不转入04。
- 批量过程中遇到敏感信息、来源冲突或目标文件冲突时，暂停该条，继续报告其余安全条目。

## 6. 安全闸门

遇到以下情况必须暂停写入：

- 可识别个人隐私、未脱敏客户资料、合同或医疗信息；
- 来源缺失，却要求写成确定事实；
- 要求删除、覆盖、批量迁移或对外发送；
- 单个案例被要求扩大为普遍规律；
- 目标路径或文件职责无法从 README 与现有结构判断。

暂停时只说明风险和所需确认，不复制扩散敏感内容。

## 7. 验收

完成一次处理必须满足：

- 原始输入仍可追溯；
- 去向与目录职责一致；
- 资产草稿脱离原文仍能读懂；
- 来源、边界、状态和下一次调用场景明确；
- 所有移动、覆盖、删除和高风险判断均经过人工确认。

三类标准测试位于 `evals/evals.json`，测试材料位于 `evals/fixtures/`。
