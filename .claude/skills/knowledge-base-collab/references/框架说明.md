# 知识库框架说明

## 理论来源

本库吸收四类思想，但不机械复制任何一套目录：

1. Zettelkasten：原子化、链接和自下而上生长；
2. LLM Wiki：先沉淀可读的主题知识，再按问题调用；
3. GBrain：持续维护、处理冲突、记录反馈和长期记忆；
4. OKF-like：用简单、可扩展的 Markdown 元数据提高机器可读性。

## 两个维度

### 内容角色

输入、项目、资产、来源回答“这份内容现在承担什么任务”。实际路径由根目录 README 定义。来源层同时容纳外部资料、个人原始输入和系统历史；“已处理”是状态，不再通过迁入独立归档目录表达。

### 资产层次

| 层次 | 回答的问题 | 典型内容 |
|---|---|---|
| 证据 | 发生了什么 | 判例、事实、数据、原文 |
| 解释 | 这意味着什么 | 概念、判断框架、方法 |
| 综合认知 | 当前相信什么及其边界 | 主题地图 |
| 表达约束 | 由谁、以什么立场表达 | 个人 IP 卡片 |
| 流程 | 如何稳定完成任务 | Claude Code Skill |

内容角色和资产层次不是两套目录，而是两个判断维度。先判断内容当前用途，再判断它属于事实、解释、综合认知、表达约束还是流程。

## 主题页与导航

主题是需要持续回答的问题，聚合是筛选资料的动作，主题地图是聚合后形成的主题页。主题页不复制全部材料，只保存当前综合判断、关键证据、方法、反例、边界和待补问题。

目录给内容一个稳定位置；来源、主题页和项目关系让同一份知识进入多个场景。关系增加时优先生成导航视图，不新增一级目录或复制文件。

标准调用顺序：

```text
问题 -> 主题页 -> 知识卡片/判例 -> 03-Resources 原始依据 -> 输出 -> 反馈写回
```

`04-Archive/` 已退役，仅保留迁移说明。旧系统方案、完成项目的历史记录等仍有追溯价值时，进入 `03-Resources/03-系统资料/系统资料/`；用户自己的原始表达进入 `03-Resources/02-个人输入/`。来源是否已筛选、提炼或使用，由集合导读、标签和具体连接表达，不反复搬动原文件。

## 判例库也是判断框架的来源

单个判例属于证据层；同类判例中反复出现的裁判逻辑、证据组合和维权路径，可以提炼为知识卡片或主题地图。判例库中的概览和维权路径因此具有“基于案例归纳的判断框架”性质，但仍需保留来源、样本边界和反例，不能把经验概率写成确定承诺。

## 内容生产关系

```text
项目状态 + 目标受众 + 知识资产 + 证据 + 表达约束
-> content brief
-> 平台 Skill
-> 内容成品
-> 复盘
-> 更新知识、项目定义或 Skill
```

选题库在本库中已启用，但只是可选的内容管理模块。候选方向需要用户确认，不是所有知识都必须转化为选题。
