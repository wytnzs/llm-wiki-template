# 知识库协作 Skill

## 入口

先读取根目录 `README.md`。目录、项目和可选模块以 README 与实际文件为准，不依赖本 Skill 中的固定清单。

开始写入前查看工作区已有修改。不得回滚、覆盖或全量暂存不属于当前任务的内容。当前磁盘文件、README、匹配的 Skill 与历史对话冲突时，按该顺序使用当前规则。

## 总控路由

| 任务 | 路由 |
|---|---|
| 记录、提炼、查询、内容角色判断 | 继续使用本 Skill |
| 处理具体 Inbox 文件或批次 | `inbox-processing` |
| 创建、更新、补链或审查主题页 | `theme-map-maintenance` |
| 运行全库体检、更新知识库地图 | `knowledge-map-update` |
| 录入或维护拒赔判例 | `case-law-entry` |
| 形成候选选题、保存确认选题 | `topic-selection`、`topic-library` |
| 进入内容生产 | 先用 `content-briefing`，再调用平台 Skill |
| 输出失败或用户否定结果 | `feedback-writeback` |
| 定期维护 | 读取 `05-Skills/知识库定期维护指令.md`，按其顺序组合以上 Skill |

专用 Skill 存在时不得在本 Skill 中另造一套流程。只读取当前任务需要的参考文件，不为了“全面理解”扫描全部私人资料。

## 内容角色

| 角色 | 判断标准 | 本库默认位置 |
|---|---|---|
| 输入 | 尚未处理，需要保留原貌 | `00-Inbox/` |
| 项目 | 只服务当前目标 | `01-Projects/` |
| 资产 | 可跨项目复用 | `02-Areas/` |
| 来源 | 需要保留、核验或追溯的原始材料 | `03-Resources/` |

`03-Resources/` 是统一来源层，包含外部资料、个人原始输入和系统历史。三类来源不能混为同一种证据：外部资料不自动代表用户判断，个人输入不自动证明外部事实，系统历史不替代当前规则。`04-Archive/` 已退役，只保留迁移说明，不再作为内容角色或路由目标。

目录只表达内容角色，不表达全部主题、来源和项目关系。不得因为一份知识同时涉及多个主题而复制文件或新增一级目录；保留一个稳定位置，通过来源、主题页和项目关系调用。

## 统一术语

- 主题：需要持续回答的问题；
- 聚合：围绕问题筛选资料的动作；
- 主题地图（主题页）：保存当前综合判断、证据、反例和边界；
- 主题中心：自然语言索引和主题页的统一入口；
- 知识库地图：动态状态报告。

## 触发流程

### 记一下

保存用户原始表达，不擅自扩写，不强制立即制成知识卡片。默认写入 `00-Inbox/灵感随手记/`。

### 整理收件箱

逐项判断：继续保留输入、进入项目、提炼资产，或进入统一来源层并标明“外部资料 / 个人输入 / 系统历史”。已经处理但仍需追溯的内容继续留在来源层，不因“完成”搬往另一目录。无法判断时列出建议，不静默删除或隔离。

需要处理具体文件、生成资产草稿或执行三类测试时，调用 `inbox-processing` Skill；本 Skill 只定义知识库层面的总规则。

### 提炼知识卡片

只有内容能跨项目复用，并帮助判断、决策或行动时才建卡。新卡至少包含 `title` 和 `type`；新建或实际更新的卡片优先补齐 `description`、可核验来源、状态和适用边界。来源不确定时先写入 `00-Inbox/AI待确认.md`，不得编造。

批量补齐现有卡片或选题的检索摘要、正文中已经明确写出的来源、基础类型标签，或修复 YAML 双链列表时，先干跑：

```text
python .claude/skills/knowledge-base-collab/scripts/enrich_metadata.py
```

逐项复核输出后才可加 `--apply` 写入。该工具不得推断新关系、猜测来源或改写正文观点。

### 查一下

先确定检索范围，再按“主题页 -> 知识卡片/判例 -> 原始材料”下钻。回答时区分用户判断、外部来源和单案事实，并说明边界。

### 项目与内容生产

先读取项目状态，形成目标、受众、核心问题、来源资产和风险边界，再交给对应生产 Skill。`02-Areas/自媒体运营/` 是已产出内容资产区，不替代项目定义。

### 复盘

判断问题来自输入、知识、项目定义、模板还是 Skill；把修正写回上游，不只修改一次结果。

### 定期维护

当用户要求“维护知识库”“执行体检”“定期更新”或类似综合操作时，完整读取 `../../../05-Skills/知识库定期维护指令.md`。先运行只读审计，再复核来源、主题、Inbox、地图和日志；未经确认不移动、不删除输入，不自动提交或推送。

### 接管自检

首次接管、修改总控规则、增删核心 Skill 或用户怀疑 Claude Code 没有正确加载时，运行：

```text
python .claude/skills/knowledge-base-collab/scripts/check_claude_code_setup.py
```

该脚本只检查入口、Skill、关键文件和运行脚本是否存在且结构有效，不读取业务正文，不自动修复。

## 本库的资产分工

- 判例库保存事实、法院认定、证据和攻击角度；
- 知识卡片保存跨案例、跨项目可复用的概念、判断和方法；
- 主题地图保存围绕问题的当前综合认知与边界；
- 个人 IP 卡片保存表达立场、口吻和禁忌；
- Skill 保存流程、规则和检查清单。

## 最小关系与 AI 权限

知识资产日常只优先维护：`source`、`related_theme_maps`、`related_projects`。支持、反例和例子按需使用；旧的 `applies_to`、`derived_methods` 字段继续兼容，但不要求新卡片默认填写。

现有 `related_topics` 是早期通用关联字段，不等于主题页关系，不批量迁移为 `related_theme_maps`。

- 原始材料只读，AI 可分类和建立来源链接，不改写正文；
- 判例事实、合同条款、病历及法律/医疗关键结论只能提议修改；
- 知识卡片和主题页可以由 AI 起草，关键判断先进入 `00-Inbox/AI待确认.md`；
- 索引、状态统计、断链检查和 `知识库变更日志.md` 可按规则自动维护。

## 已启用模块规则

- 知识库地图、判例库和选题库均已在本库启用；
- 选题 Skill 只能提出候选，用户确认后才能正式入库；
- 不宣称知识卡片会自动生成正式选题；
- 主题地图可以记录内容应用方向，但不能自动创建选题卡。

## 原则

1. 标准定义内容角色，不强制复杂目录；
2. 原始材料、知识资产、项目材料和操作流程分开；
3. 不把单案结论外推成普遍承诺；
4. 不上传私人资料；
5. 不静默删除或批量迁移。

## 完成检查

- 写入范围与用户请求一致，原始表达和来源仍可追溯；
- 新建正式知识至少包含 `title`、`type`，关键入口优先有 `description`；
- 关系通过“明确、有用、可追溯”三问法，不为消除孤立页制造链接；
- 运行任务匹配的验证脚本，并复核结果；
- 重要系统变化写入 `知识库变更日志.md`，一般编辑不制造流水；
- 对外发布必须另行精确筛选，私人知识库的 Git 状态不等于公开发布范围。
