# 知识库协作 Skill

> 基于 Karpathy 三层架构 + Zettelkasten 卡片笔记法 + 概念思维 + OKF/GBrain 养护理念。
> 本 skill 只规定 HOW，不储存证据或知识观点。
> **本知识库不是普通资料库，而是一套「知识资产生产系统」——原材料加工成资产，资产按需组装产出。**

---

## 核心架构：两套分层说明

本知识库有两套分层体系，互为经纬：

| 分层体系 | 视角 | 用途 |
|---------|------|------|
| **五层生产链路**（本节） | 横向——生产流程阶段 | 理解知识从原材料到成品的完整流转路径 |
| **四层架构分工**（后文） | 纵向——知识性质层次 | 理解每个文件属于"证据/解释/表达/流程"哪个层，该写什么不该写什么 |

**五层回答"怎么流转"，四层回答"怎么归类"。两者不冲突，共同定义知识库的骨架。**

### 五层生产链路

本知识库按 **输入层 → 资产层 → 项目层 → 输出层 → 反馈层** 五层组织，每层只做一件事：

```
输入层 (00-Inbox / 03-Resources)
    ↓ 提炼、归类、结构化
资产层 (02-Areas)
    ├── 知识卡片/    ← 原子化判断框架（可复用认知）
    ├── 判例库/      ← 事实证据（不写情绪）
    └── 主题聚合/    ← 长期记忆（见下方 GBrain 说明）
    ↓ 按需组装
项目层 (01-Projects)
    ├── 自媒体运营/  ← 内容生产项目
    ├── 寿险代理人理赔实战手册/
    ├── 理赔工具/
    ├── 知识库产品化/
    └── ...
    ↓ 调用资产 + skill
输出层 (生产管线产出)
    ├── 公众号文章
    ├── 小红书笔记
    ├── 短视频脚本
    └── content-brief.json（唯一生产交接单）
    ↓ 复盘反哺
反馈层 (复盘与反哺)
    ├── 内容复盘库
    ├── feedback-writeback（失败追溯）
    └── 反哺 → 更新资产层 / 更新 skill
```

**五层原则：**
- 每一层只做自己层的事，不越界
- 输出层不保存原始素材，资产层不写传播包装
- 反馈层的复盘必须回到资产层或生产层，不能只在输出层"修这一次"

## 输入层 → 资产层的提炼路径

```
00-Inbox/ 或 03-Resources/ 中的原始材料
    │
    ├── 能跨案件/险种/项目复用的抽象框架
    │   → 提炼为 02-Areas/知识卡片/（原子化卡片）
    │   → 追加 [[双向链接]] 关联已有卡片
    │
    ├── 只服务某一类案件的事实
    │   → 归入 02-Areas/判例库/（按三视图归档）
    │
    ├── 围绕某个问题/主题的认知集合
    │   → 组织成 02-Areas/主题聚合/（见下方 GBrain 说明）
    │
    ├── 有苗头但尚不成熟的创作方向
    │   → 进入 06-选题库/待挖掘方向.md
    │
    └── 完整的书籍/课程笔记
        → 归档到 03-Resources/读书笔记/
```

## GBrain 思想：主题聚合是长期记忆

`02-Areas/主题聚合/` 不只是一个"链接索引"——它是知识库的**长期记忆**。

**主题聚合 ≠ 资料列表。** 每个主题地图应该：

- **回答「当前我对某个主题的认知是什么」**——不是罗列资料，而是整理当前版本的判断
- **包含核心判断**——围绕这个主题，我现在相信什么、为什么这么相信
- **标记不确定性**——哪些判断还不够牢靠、需要更多证据
- **连接卡片和判例**——哪些知识卡片和判例支撑了当前判断
- **支持回溯和更新**——有版本记录，知道什么时候改过什么

**养护方式：**
- 当某一类案件反复出现并产生稳定打法时，更新对应主题地图中的「已验证方法」
- 当发现已有判断被新判例推翻时，更新「当前核心判断」并标注反例
- 主题地图不是一次写完就丢的——它要随着知识库的增长而被持续养护

## OKF-like 原则：尽量带 YAML 元数据

新增的**知识卡片、主题地图、选题卡**都应尽量包含 YAML 元数据，方便自动扫描和索引。

### 知识卡片 YAML
```yaml
---
title: 概念名
tags: [领域, 知识卡片]
links: [[关联卡片]]
created: 2026-07-09
updated: 2026-07-09
source: 来源（判例/书籍/经验/外部资料）
---
```

### 主题地图 YAML
```yaml
---
title: 主题名
type: theme_map
tags: [主题地图]
created: 2026-07-09
updated: 2026-07-09
related_cards:
  - [[卡片1]]
  - [[卡片2]]
related_cases:
  - 案号1
  - 案号2
---
```

### 选题卡 YAML
```yaml
---
title: 选题名
type: topic_card
status: draft | active | published | archived
source_cards:
  - [[卡片1]]
source_cases:
  - 案号1
related_theme:
  - [[主题地图]]
content_stage: idea | brief | writing | done
created: 2026-07-09
---
```

> YAML 自动索引让 knowledge-map-update 可以精确跟踪资产状态。新卡片建议带 YAML，已有卡片逐步补充。

| 写作/选题/运营 | 02-Areas/知识卡片/创作/ |

| 围绕问题的综合认知 | 02-Areas/主题聚合/ |

### 跨层判断标准

| 素材特征 | 目标层 | 判断逻辑 |
|---------|--------|---------|
| 只服务某一类案件的具体事实 | 判例库（证据层） | 事实冷静，不写包装 |
| 能跨案件/险种/项目/内容复用的抽象框架 | 知识卡片（解释层） | 回答"这是什么"，不回答"具体哪个案子" |
| 围绕一个问题的多张卡片+判例+方法的综合认知 | 主题聚合（长期记忆） | 回答"当前对X主题的认知是什么" |
| 选题入口（给谁看、痛点、冲突、可调用资源） | 选题库（选题资产） | 具备生产价值 |
| 完整书籍笔记 | 03-Resources/ | 外部参考材料 |

## 四层架构分工

| 层 | 位置 | 职责 | 不做什么 |
|----|------|------|---------|
| 证据层 | 02-Areas/判例库/ | 存事实、裁判观点、攻击角度、证据 | 不写情绪、标题、传播包装 |
| 解释层 | 02-Areas/知识卡片/ | 存框架、判断标准、方法论 | 不存单案事实 |
| 表达约束层 | 02-Areas/知识卡片/个人IP/ | 决定立场、口吻、CTA风格、禁忌 | 不提供事实 |
| 生产层 | .claude/skills/ | 存流程、规则、检查清单 | 不堆判例、观点、素材 |

**一句话：判例库=证据 / 知识卡片=解释 / IP=表达 / Skill=流程。中间用文章 YAML 头部做交接。**

## 素材自动识别

对话中自动识别有料瞬间并记录到 00-Inbox/灵感随手记/：

| 类型 | 识别信号 |
|-----|---------|
| 踩坑翻车 | 预期≠结果、报错、折腾半天 |
| 意外发现 | "没想到"、"原来可以" |
| 迭代打磨 | 改了多版、从复杂到简单 |
| 突破时刻 | 卡了很久终于解决 |
| 方法沉淀 | 可复用的技巧/心法 |

识别后在段落尾标注（📝素材+1），写入 00-Inbox/灵感随手记/。

## 内容发布后闭环

每次公众号文章/短视频脚本发布后，执行以下闭环：

### 1. 识别可沉淀的认知

从发布的内容中提取：
- **新方法/原则**：有没有可复用的操作步骤或判断标准？
- **新案例**：有没有值得单独记录的案例及其背后的规律？
- **新连接**：有没有发现两个已有概念之间的新关联？

### 2. 建议卡片

对每个可沉淀的认知：
- 建议卡片标题
- 建议归属领域（按归类规则）
- 写一句"判断维度"——这张卡片在什么场景下能帮到什么决策？

### 3. 写入确认

询问用户确认后，写入 02-Areas/知识卡片/ 对应目录。更新已有卡片的 [[链接]]。

### 4. 更新主题地图

如果新卡片或新案例修正了某个主题的核心判断，同步更新 `02-Areas/主题聚合/` 中对应的主题地图。

### 5. 更新效果日志

引导记录到 `01-Projects/自媒体运营/自我成长/内容复盘库.md`。

## 原则

1. **原子化** —— 一张卡片只讲一个概念。
2. **双向链接** —— 用 `[[卡片名]]` 建立关联。
3. **概念为本** —— 卡片回答"这是什么"，不回答"具体哪个案子"。
4. **你存我整** —— Claude Code 归类，用户确认。
5. **输出反哺** —— 每次生产结束，要点反哺到卡片/判例/skill/主题地图。
6. **五层不越界** —— 每层只做自己层的事。
7. **主题聚合要养护** —— 长期记忆需要持续更新，不是一次写完就丢。
8. **YAML 优先** —— 新资产尽量带元数据，方便自动索引。
9. **不要把判例库写成文章库** —— 情绪、标题、包装不进判例文件。
10. **不要让 skill 变成资料库** —— 避免"库里一套、skill 里又一套"。
11. **不要跳过来源记录** —— 每篇内容头部 YAML 记录调用卡片、调用判例、content-brief 路径。
12. **不要把个案结论写成普遍承诺** —— "这个案子说明" ≠ "一定能赔"。
