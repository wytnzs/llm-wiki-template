# 知识库协作 Skill

## 入口

先阅读根目录 `README.md`，获取知识库目的、实际目录映射和已启用模块。README 是唯一必需入口。

不要假设用户一定使用某套目录；推荐路径只在 README 没有说明时作为默认值。

## 基本流程

```text
输入 -> 整理 -> 连接 -> 使用 -> 检查 -> 复盘
```

目录只表达内容角色，不表达全部主题、来源和项目关系。一份知识保留一个稳定位置，通过来源、主题页和项目关系进入不同使用场景。

## 统一术语

- 主题：需要持续回答的问题；
- 聚合：围绕问题筛选资料的动作，不单独创建知识类型；
- 主题页：保存当前综合判断、证据、方法、反例和边界；
- 主题中心：自然语言入口和主题页的统一导航；
- 知识库地图：动态状态报告，不承担主题综合。

## 五种内容角色

| 角色 | 判断标准 | 默认位置 |
|---|---|---|
| 输入 | 尚未处理、需要保留原貌 | `00-Inbox/` |
| 项目 | 只服务当前目标 | `01-Projects/` |
| 资产 | 能跨项目复用 | `02-Areas/` |
| 资源 | 外部材料，不代表用户判断 | `03-Resources/` |
| 归档 | 已完成或暂时不用 | `04-Archive/` |

目录可以改名或合并，以 README 的映射为准。

## 触发流程

### 记一下

保存用户原始表达，不擅自扩写，不强制立即制成知识卡片。

### 整理收件箱

逐项判断：保留原始输入、进入项目、提炼资产、转为外部资源或归档。没有明确价值时先报告，不静默删除。

批量处理时每批默认不超过 10 条；未经确认不移动、覆盖或删除原文件。

### 提炼知识卡片

只有内容能够跨项目复用并帮助判断、决策或行动时才建卡。新卡至少包含 `title` 和 `type`，优先补充 `description`、可核验来源、状态和适用边界。来源不确定时保持空缺或进入待确认，不编造“综合研究”。

### 查一下

先读 README 确定资产位置，再按“主题页 -> 知识卡片或案例 -> 原始材料”下钻。回答时区分用户判断、外部资料和案例事实，并说明适用边界。

### 复盘

追溯问题来自输入、知识、项目、模板还是 Skill；把修正写回源头。

## 最小关系

日常优先维护：

1. `source`：从哪里来；
2. `related_theme_maps`：直接帮助回答哪个持续问题；
3. `related_projects`：在哪里真实使用过。

支持、反例和例子按需使用，不是必填项。新增关系前确认：关系能否一句话说清、沿过去是否有用、能否回到可靠依据。只是同领域或关键词相同，保留标签即可。

完整标准见 `05-Skills/知识关系与索引规则.md`；在模板仓库中对应 `docs/知识关系与索引规则.md`。

## AI 权限

- 原始材料只读，AI 可以分类和建立来源，不改写正文；
- 知识卡片和主题页可以起草，涉及用户判断、关键证据或普遍结论时先请求确认；
- 案例事实、合同、病历及法律或医疗结论只能提出建议；
- 索引、状态统计、断链检查和变更日志可以按既定口径维护；
- 删除、覆盖、批量迁移、提交、推送和对外发送需要明确授权。

## 可选模块

- README 未声明知识库地图时，不调用地图 Skill；
- README 未声明案例库时，不创建案例目录；
- README 未声明选题库时，不创建选题卡或 `06-选题库/`；
- 选题库已启用时，也必须由用户确认候选选题后再入库。

## 原则

1. 标准定义角色，不强制复杂目录；
2. 原始材料、知识资产、项目成品和操作流程分开；
3. Skill 只保存方法，不保存业务知识；
4. 不把个案结论写成普遍承诺；
5. 不为消灭孤立页或填满字段制造关系；
6. 不上传私人资料，不静默删除或批量迁移。
