# 内容选题库 Topic Library

> 选题不是一次性灵感，而是可以沉淀、复用、迭代的内容资产。
> **正式选题必须有资产支撑**——没有来源资产的选题，只能进入待挖掘方向，不能进入正式选题库。

---

## 一、核心定位

本 Skill 负责管理 `06-选题库/`。回答四个问题：

1. **值不值得写？** —— Scale 评分
2. **有没有重复？** —— 查重规则
3. **资产溯源是否完整？** —— 每个正式选题必须可追溯到资产源头
4. **如何交给生产管线？** —— content-brief 交接

### 选题决策权

- **用户负责决定写什么。** 用户的原始判断、真实表达欲和当前内容计划优先于任何自动评分。
- **本 Skill 不负责主动找题。** 不从知识库、判例库或内容缺口中自动生成候选选题，也不推荐“本次应该拍哪几条”。
- **AI只做结构工作。** 包括查重、归类、素材匹配、冲突提炼、风险提示、卡片规范化和生产交接。
- 用户尚未确认的想法只能暂存，不得因为评分高而自动升级为正式选题。

### 与内容生产的关系

| Skill | 职责 |
|-------|------|
| 用户 | 提出并确认选题，决定当前内容计划 |
| `topic-library` | 把用户确认的选题入库、查重、评分、溯源验证、输出 |
| 生产管线（gzh/xhs/short-video） | 消费 content-brief 写最终文案 |

## 四、好选题评分 Scale

每个候选选题按 **7 个维度评分**，每项 1-5 分：

| 维度 | 判断问题 |
|---|---|
| **反差** | 有没有"用户以为 X，但其实 Y"？ |
| **用户疼痛** | 目标用户是否真的在意？能否解决具体焦虑？ |
| **证据支撑** | 是否有判例、知识卡片、经验或行业事实支撑？ |
| **情绪爆点** | 讲述者是否有真实表达欲？能不能讲出情绪？ |
| **可迁移性** | 是否能从个案提炼出方法或判断框架？ |
| **人设匹配** | 是否强化"懂理赔、懂判例、懂代理人处境"的定位？ |
| **平台适配** | 是否明确适合短视频、公众号或小红书？ |

### 入库门槛

- 总分低于 24 分 → 不入库
- 缺少具体目标用户 → 不入库
- 缺少反差或用户疼痛 → 不入库
- **无来源资产**（未关联任何知识卡片/判例/个人经历/外部资料）→ **不入正式库**，仅进入待挖掘方向
- 有证据、有反差、有明确平台 → 生成正式选题卡片

## 五、资产溯源规则

**核心原则：每个正式选题必须能回答"这个选题的底气来自哪里？"**

### 可接受的来源资产类型

| 来源类型 | 说明 | 在选题卡 YAML 中的字段 |
|---------|------|----------------------|
| 知识卡片 | 从 `02-Areas/知识卡片/` 提取的判断框架 | `source_cards` |
| 判例/案例 | 从 `02-Areas/判例库/` 或实际办案经历 | `source_cases` |
| 主题地图 | 从 `02-Areas/主题中心/主题页/` 获取的当前认知 | `related_theme` |
| 个人经历 | 自己的从业经验、踩坑、突破 | 在 `source_cards` 中以 `[[个人经历：XXX]]` 形式标注 |
| 外部资料 | 书籍、课程、行业报告（来自 03-Resources/） | 在 YAML 描述中注明 |

### 溯源检查流程

1. **选题候选到来时**，先扫描知识库查找可关联的资产
2. 如果查到关联资产 → 在选题卡 YAML 中用字段记录
3. 如果没有查到任何关联资产 → 选题不能进入 `cards/`，只能写入 `待挖掘方向.md`
4. 如果查到的关联资产不充分（如只有间接关联）→ 可以入库但标记 `source_confidence: low`，并在 content-brief 阶段提示需补充素材

### 无来源资产的隔离

`06-选题库/待挖掘方向.md` 中的条目应当遵守：

- 只记录"有苗头但缺资产支撑"的方向，不设具体的选题卡
- 格式：| 方向 | 缺什么资产 | 建议来源类型 | 选题树 | 触发来源 | 挖掘建议 | 日期 |
- 当某方向的来源资产积累到足够时（至少关联 1 张卡片或 1 个判例），可提升为正式选题

## 六、查重规则

生成新卡片前，必须按顺序检查：

1. 标题相似度 —— 是否与已有卡片相似？
2. 核心冲突相似度 —— 冲突公式是否相同？
3. 素材依赖重复 —— 判例或卡片已被同树选题使用？
4. 开头钩子雷同 —— 已有文案开头类似？
5. 换壳检测 —— 只是换了说法？

如果相似 → 不直接入库，给出已有路径，建议换角度。

允许多角度：同一判例不同选题树、不同切入角度、同一经历不同侧面 → 不重复。

## 七、选题卡片结构

保存到 `06-选题库/cards/<选题树>/<标题>.md`。

### YAML 头部（必填）

```yaml
---
title: 选题标题
status: draft | active | published | archived
source_cards:        # 关联的知识卡片，如 [[未如实告知边界]]，[[等待期争议]]
  - 
source_cases:        # 关联的判例案号，如 2023-京0104民初XXXX
  - 
related_theme:       # 关联的主题地图，如 [[保险理赔主题地图]]
  - 
content_stage:       # 内容生产阶段：idea | brief | writing | done
  idea
created: 2026-07-09
priority: P0 | P1 | P2
topic_tree:           # 所属选题树
target_reader:        # 目标读者
confidence: high | medium | low   # 来源置信度
---
```

> `source_cards`、`source_cases`、`related_theme`、`content_stage` 是 v2 新增字段。选题卡应逐步补充这些字段。

### 正文结构

- **基础信息**：状态 / 优先级 / 选题树 / 平台 / 目标读者 / 生成日期
- **选题判断**：核心问题 / 核心冲突 / 反差公式 / 难度 / 时效
- **Scale 评分表**：7 维度 × 1-5 分 + 理由 + 总分
- **素材来源**：source_cards / source_cases / related_theme（溯源展示）
- **内容方向**：开头钩子 / 核心观点 / 可展开角度
- **风险提醒**：不能说满的地方 / 个案外推 / 需补充材料
- **下游记录**：content-brief 状态 / 稿件 / 发布记录

## 八、选题总索引写入规则

每张新卡片同步更新 `06-选题库/选题总索引.md`。

格式：| 状态 | 优先级 | 选题 | 选题树 | 目标读者 | 核心冲突 | 适合平台 | 来源资产 | 卡片路径 |

> `来源资产` 列为新增——简短展示该选题关联的知识卡片数和判例数。

## 九、待挖掘方向规则

写入 `06-选题库/待挖掘方向.md`。

格式：| 方向 | 缺什么资产 | 建议来源类型 | 选题树 | 触发来源 | 挖掘建议 | 日期 |

**待挖掘方向 ≠ 正式选题**。差异如下：

| 维度 | 正式选题 cards/ | 待挖掘方向 |
|------|----------------|-----------|
| 资产溯源 | 必须关联 ≥1 个资产 | 无需资产支撑 |
| 结构化 | 完整选题卡 + YAML | 简洁表格行 |
| 可生产性 | 可直接生成 content-brief | 尚不能生产 |
| 升降级 | 可降级至待挖掘 | 资产到位后可升级 |

## 十、与 content-brief 的关系

选题卡片是上游资产，content-brief.json 是进入生产前的交接单。

**触发词**："选这个" / "生成 content-brief" / "基于这个选题" / "开始写"

**路径**：`.claude/drafts/content-brief.json`

**字段**：topic / platform / target_person / reader_problem / core_conflict / core_viewpoint / why_i_want_to_say_this / selected_cards / selected_cases / related_theme / case_takeaways / claim_level / forbidden / output_plan / missing_materials

生成 content-brief 时，自动从选题卡 YAML 中读取 `source_cards`、`source_cases`、`related_theme`，分别填充到 `selected_cards`、`selected_cases`、`related_theme` 字段，不把主题地图混入素材字段。

## 十一、典型工作流

### 工作流 1：保存用户给的选题

保留用户原始表达 → 确认用户已决定采用 → 查重 → 溯源检查（能否关联到知识卡片/判例等？）→ 补全信息 → Scale 评分 → 生成卡片 → 更新索引 → 不能溯源则标记缺失材料

### 工作流 2：审查用户给出的候选清单

读取用户候选 → 与已发布内容和选题卡双重查重 → 标注素材充足度与重复风险 → 给出结构化审查结果 → 等待用户决定保留、合并或放弃 → 仅将用户确认项入库

### 工作流 3：从选题卡进入生产

读取卡片 YAML 中的 source_cards / source_cases → 生成 content-brief.json（自动填充素材来源）→ 更新下游记录 → 提示调用生产 skill

### 工作流 4：选题库健康检查

统计覆盖 → 缺口分析 → 陈旧检查 → 资产溯源完整性检查（哪些选题缺 source_cards？哪些缺 source_cases？）→ 输出健康报告

---

## 十二、硬约束

1. 不把已有文案换个说法当新选题
2. **没有来源资产的选题不能进入正式选题库**——只进待挖掘方向
3. 不把个案结论包装成普遍规则
4. 不承诺一定胜诉/赔付
5. 不保存无目标读者、无冲突、无证据的空泛选题
6. 不直接写最终文案，止步于 content-brief
7. 优先沉淀可复用判断框架
8. 历史运营看板只读不写。新选题只写到 06-选题库/
9. **溯源字段必须写**——source_cards / source_cases / related_theme / content_stage 是必填项（可空数组但不能缺字段）
10. **不主动生成、补齐或推荐选题**——知识库缺口只作为架构健康信息，不转化为内容建议
11. **评分没有决策权**——Scale 只用于整理和排序用户已确认的题，不能替代用户判断
12. **必须保留用户原始表达**——结构化时不得用通用标题覆盖用户最初的表达意图
