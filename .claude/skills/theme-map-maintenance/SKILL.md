# 主题地图维护 Skill

> 主题页不回答“这个主题有什么资料”，而回答“当前对这个问题的综合判断是什么，为什么相信，什么情况下可能失效”。

---

## 核心定位

先读取根目录 README。如果 README 没有其他映射，新知识库默认使用 `02-Areas/主题中心/主题页/`；已有知识库保持实际路径。聚合只是围绕问题筛选资料的动作，不单独形成目录或知识类型。

| 维度 | 知识卡片 | 案例库 | 主题地图 |
|------|---------|-------|---------|
| 粒度 | 原子概念 | 单案事实 | 综合认知 |
| 回答 | 这是什么 | 这个案子怎么了 | 当前我相信什么 |
| 维护方式 | 炼新卡片 | 录入新案例 | 认知升级时同步更新 |

---

## 标准结构

```yaml
---
title: 主题名
type: theme-map
description: 一句话说明主题页回答的问题
core_question: 需要持续回答什么问题？
status: active
tags: [主题地图]
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_cards: []
related_cases: []
---
```

正文模块：
1. 当前核心判断
2. 关键概念（关联卡片）
3. 证据来源
4. 已验证方法
5. 适用边界
6. 反例与风险
7. 最近更新记录
8. 待补问题

---

## 创建/更新时机

创建：至少 3 张卡片围绕同一问题、至少 5 个案例形成同类模式，或用户明确要求综合当前认知。

更新：新证据推翻或巩固判断、稳定方法形成、边界变化、项目反馈暴露问题，或主题长期未审阅。

---

## 工作流

### 创建
1. 明确核心问题 → 2. 收集具体证据 → 3. 提炼当前判断和边界 → 4. 按标准结构写入 → 5. 建立精选关系

### 更新
1. 读取已有地图 → 2. 确定触发原因 → 3. 只更新变化模块 → 4. 更新记录

### 健康检查
检查核心问题是否具体、证据能否下钻到文件或原文、关系是否精选、边界是否明确。目录级概述不能替代具体证据。

## 关系准入

`related_cards` 和 `related_cases` 只保留直接回答核心问题，或提供证据、方法、反例、边界和代表性验证的入口，不追求覆盖全部同领域资料。

新增关系前确认：

1. 关系能否一句话说清；
2. 沿过去是否帮助判断或行动；
3. 能否继续回到可靠依据。

三项都是“是”才建立关系。批量变更先干跑并抽查高相关、边界相关和明显不相关样本。完整标准见 `05-Skills/知识关系与索引规则.md`；在模板仓库中对应 `docs/知识关系与索引规则.md`。

当 AI 发现来源不足、可能误连、结论冲突或需要改变用户核心判断时，不直接改写，先请求用户确认。

主题页可以记录内容应用方向，但不能自动创建正式选题。

---

## 原则

1. 不是资料列表，是认知快照
2. 只更新变化的部分
3. 标注确定性程度
4. 保持双向链接有效
5. 不把目录、无出处数字或单个案例当作充分证据
6. 不为更新而更新，不为内容生产强行制造选题
