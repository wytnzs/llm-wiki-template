# 主题地图维护 Skill

> 基于 GBrain 思想：**主题聚合是长期记忆，不只是链接索引。**
> 主题地图不回答「这个主题有什么资料」——它回答「当前我对这个主题的认知是什么，为什么这么相信，什么情况下可能错」。

---

## 一、核心定位

`02-Areas/主题聚合/` 是知识库的**长期记忆层**。它位于知识卡片（原子概念）和判例库（单案事实）的**上层**，把分散的原子知识组织成**对一个问题的完整认知**。

| 维度 | 知识卡片 | 判例库 | 主题地图 |
|------|---------|-------|---------|
| 粒度 | 原子概念 | 单案事实 | 综合认知 |
| 回答 | "这是什么" | "这个案子怎么了" | "当前我相信什么" |
| 变化频率 | 低 | 中（持续录入） | **中（随认知迭代更新）** |
| 维护方式 | 炼新卡片 | 录入新判例 | **认知升级时同步更新** |

### 主题地图 vs 索引/导航

| 不是 | 是 |
|------|-----|
| ❌ 目录导航 | ✅ 当前核心判断 |
| ❌ 资料文件列表 | ✅ 关键概念+支撑证据 |
| ❌ 静态存档 | ✅ 活的认知——可更新、可反驳、可迭代 |
| ❌ 所有资料面面俱到 | ✅ 精选已验证的方法和边界 |

## 三、什么时候创建/更新主题地图

### 创建时机

当出现以下情况时，考虑创建新主题地图：

- 已经有 ≥3 张知识卡片围绕同一个问题
- 已经有 ≥5 个判例同属一个主题
- 已经形成了稳定的打法/判断，但尚未被组织成系统认知
- 用户明确说"总结一下我对XXX的认知"

### 更新时机

当出现以下情况时，更新已有主题地图：

| 触发条件 | 更新模块 | 更新内容 |
|---------|---------|---------|
| 新判例推翻了既有判断 | 当前核心判断 + 反例/风险 + 更新记录 | 修正认知，记录反例 |
| 新判例巩固了已有判断 | 证据来源 + 已验证方法 + 更新记录 | 补充证据密度和案例数 |
| 形成了新的稳定打法 | 已验证方法 + 更新记录 | 新增方法条目 |
| 发现边界条件变化 | 适用边界 + 更新记录 | 补充适用范围/例外 |
| 出现了可转化的新选题 | 可转化选题 + 更新记录 | 新增选题方向 |
| 用户反馈某个主题认知不准 | 全面审阅 → 逐项评估 → 更新 | 根据反馈修正 |
| 超过 30 天未更新 | 全部模块 → 判断是否需要更新 | 保持认知活跃 |

### 不更新（保持原样）的情况

- 新判例只是重复已有模式，未提供新认知 → 不更新主题地图，只更新判例库
- 新知识卡片只是已有框架的延伸 → 不更新主题地图，只关联 [[链接]]
- 没有新的认知突破或修正需要记录 → 等待下一次触发

## 四、工作流

### 工作流 1：创建新主题地图

1. 收集该主题下的已有知识卡片（扫描 `02-Areas/知识卡片/`）
2. 收集该主题下的已有判例（扫描 `02-Areas/判例库/`）
3. 提炼当前核心判断
4. 按标准结构写入 `02-Areas/主题聚合/{主题名}.md`
5. 在已有知识卡片和判例中增加 `[[主题名]]` 双向链接
6. 更新 `知识库地图.md`

### 工作流 2：更新已有主题地图

1. 读取 `02-Areas/主题聚合/{主题名}.md`
2. 确定触发更新的原因
3. 只更新需要更新的模块，不全部重写
4. 更新文件头部 `updated` 字段
5. 更新「最近更新记录」
6. 检查「待补问题」是否有可关闭的

### 工作流 3：主题地图健康检查

1. 遍历 `02-Areas/主题聚合/` 下所有主题地图
2. 逐项检查：

| 检查项 | 标准 | 不通过时 |
|-------|------|---------|
| 更新时效 | 最近 30 天内有更新 | 建议刷新核心判断 |
| 当前核心判断 | 有，且清晰 | 补充核心判断 |
| 关键概念 | 有 [[链接]] 关联 | 补充概念链接 |
| 证据来源 | 指向具体路径 | 补充证据引用 |
| 已验证方法 | 有（或合理空缺） | 根据判例补充 |
| 适用边界 | 有 | 补充边界说明 |
| 反例/风险 | 有（或合理空缺） | 分析并补充 |
| 可转化选题 | 有（或合理空缺） | 基于认知生成选题 |
| 待补问题 | 有持续追踪 | 新增问题或关闭已解决 |

3. 输出健康报告：

```
主题地图健康报告 — {日期}

🟢 保险理赔主题地图 — 良好，最近更新于XX
🟡 内容生产主题地图 — 注意：已更新卡可选转换题
🔴 XXX主题地图 — 过期：超过 60 天未更新
```

### 工作流 4：从 case-law-entry 触发更新

当 `case-law-entry` 的"层判断"步骤建议更新主题地图时：

1. 接收建议信息（哪个判例、什么打法稳定了）
2. 读取对应的主题地图
3. 更新「已验证方法」或「当前核心判断」
4. 在「最近更新记录」中注明 "由判例 {案号} 触发"
5. 向用户汇报更新内容

## 五、养护哲学

主题地图不是一次写完就丢的。它是你**当前认知的锚点**：

- **当新知识到来**，判断它是否修正了已有认知 → 修正就更新，否则保持
- **当认知稳定后**，主题地图变成"可教授给别人的体系"
- **当发现判断漏洞**，它不是失败，而是认知升级的机会——记录下来
- **定期回顾频率**：至少每月一次全面回顾，及时发现过时判断

**养护不是做加法。** 如果某个主题你已经不再关注，可以：
1. 在「最近更新记录」中注明"主题已暂停追踪，认知截至 XXXX"
2. 不移除，保留最后一次的有效判断
3. 待重新关注时恢复更新

| 上游 | 关系 |
|------|------|
| `02-Areas/知识卡片/` | 主题地图引用卡片作为关键概念的基础 |

|------|------|

## 七、原则

1. **主题地图不是资料列表**——它是你当前对一个问题的认知快照
2. **只更新变化的部分**——不需要每次全部重写
3. **标注确定性程度**——明确的判断和有待验证的猜想要区分
4. **保持[[链接]]活跃**——主题地图与卡片/判例的双向链接要持续维护
5. **记录认知变迁**——最近更新记录不只是日志，更是思维进化史
6. **不夸大适用范围**——在适用边界里清楚写"什么情况下这个判断无效"
7. **定期养护，但不频繁扰动**——有新认知才更新，不为更新而更新
