# 反馈回写 Skill

> 核心原则：**每次失败都要让系统变得更强，而不是只修这一次的结果。**
> 每一次失败都有一条根因链条——不追到最上游的源头，同一问题会在不同选题上反复出现。
> 灵感来源：GBrain 的持续养护理念 + OKF 的根因追溯思维。

---

## 什么时候触发

| 触发场景 | 说明 |
|---------|------|
| 写出来的内容感觉不对 | 口吻不对、结构不对、用户说"不是这个意思" |
| 选题方向偏差 | 用户说"不想写这个"、"这个角度不对" |
| 判例调用错误 | 用错了案例，或判例结论与文章观点不匹配 |
| 知识卡片调用不足 | 内容单薄，缺少框架支撑 |
| 平台适配失败 | 公众号的内容直接当短视频脚本用 |
| 法律边界说满 | 文案中出现"一定能赔"等夸大承诺 |
| content-brief 不完整 | 生产管线缺材料、缺目标读者、缺禁忌 |

## 根因分类体系

失败不能只停在"这个稿子写得不好"——要问：**是哪个上游环节没有准备好？**

以下 8 种根因分类，按从上游到下游排列。追溯时优先往上游查：

| # | 根因 | 表现 | 建议更新对象 |
|---|------|------|-------------|
| 1 | **知识卡片不足** | 内容单薄、缺少框架支撑、判断模糊 | `02-Areas/知识卡片/` 对应领域——补充新卡片或更新已有卡片 |
| 2 | **主题地图不清晰** | 同一主题下多次出现相似失误、整体认知不统一 | `02-Areas/主题聚合/` 对应主题地图——更新核心判断/适用边界/已验证方法 |
| 3 | **选题卡不完整** | 选题没有核心冲突、没有目标读者、没有来源支撑 | `06-选题库/cards/` 对应选题卡——补充 missing 字段，或移至待挖掘方向 |
| 4 | **content-brief 缺字段** | 生产时缺 target_person / forbidden / core_viewpoint | `.claude/drafts/content-brief.template.json`——补充缺失字段 |
| 5 | **个人IP约束不准** | 口吻不对、CTA 偏离人设、风格不对 | `02-Areas/知识卡片/个人IP/`——更新风格约束/CTA规则/禁忌 |
| 6 | **平台 skill 规则缺失** | 公众号内容当短视频用、平台调性适配不到位 | `.claude/skills/gzh-pipeline/SKILL.md` 或 `xhs-pipeline/SKILL.md` 或 `short-video-script/SKILL.md`——补充平台适配规则 |
| 7 | **判例/证据边界不足** | 判例调用错误、判例标签不准确、个案结论被推广 | `02-Areas/判例库/` 对应判例——修正标签/补充边界/补充适用条件 |
| 8 | **生产 skill 流程缺陷** | 漏了步骤、顺序错了、检查项不够 | `.claude/skills/` 对应 SKILL.md——补充流程步骤或检查项 |

### 追溯方法

每次失败，按以下顺序问三组问题：

**第一组：问题出在哪个层？**
- 是没素材 → 知识卡片不足 or 判例/证据边界不足
- 是选题错了 → 选题卡不完整 or 主题地图不清晰
- 是写坏了 → content-brief 缺字段 or 个人IP约束不准 or 平台 skill 规则缺失 or 生产 skill 流程缺陷

**第二组：根因是单点的还是系统的？**
- 单点：只在一个内容上出了问题 → 补那个内容的对应文件即可
- 系统：两个以上选题都出现同类问题 → 问题在上游（主题地图/知识卡片/skill）

**第三组：这次修正够不够？**
- 修改完后问自己：同样的错误下次还会再犯吗？
- 如果不会 → 修对了
- 如果还会 → 回到第一组，根因没找对

## 反馈回写流程

### 第一步：记录失败

在 `01-Projects/自媒体运营/自我成长/内容复盘库.md` 的"AI输出失败复盘"表中追加一行：

| 日期 | 任务 | 使用 skill | 问题类型 | 具体表现 | 根因分类 | 上游文件 | 修正动作 | 是否需更新 skill |
|------|------|-----------|---------|---------|---------|---------|---------|----------------|

> **上游文件**列是新增字段——记录根因对应的具体文件路径，方便后续追溯和批量修复。

### 第二步：追溯根因

使用上述根因分类体系和追溯方法，找到真正的根因。

### 第三步：更新源头

根据根因，更新对应的源头：

| 根因分类 | 更新对象 | 建议操作 |
|---------|---------|---------|
| 知识卡片不足 | `02-Areas/知识卡片/` | 补充新卡片 / 更新已有卡片 / 补 [[链接]] |
| 主题地图不清晰 | `02-Areas/主题聚合/` | 更新核心判断 / 补充适用边界 / 更新已验证方法 |
| 选题卡不完整 | `06-选题库/cards/` | 补充 missing 字段 / 移至待挖掘方向 |
| content-brief 缺字段 | `.claude/drafts/content-brief.template.json` | 补充 target_person / forbidden 等字段 |
| 个人IP约束不准 | `02-Areas/知识卡片/个人IP/` | 更新风格约束 / CTA 规则 / 表达禁忌 |
| 平台 skill 规则缺失 | 对应平台 skill 的 SKILL.md | 补充适配规则 / 检查清单 |
| 判例/证据边界不足 | 判例库对应判例 | 修正标签 / 补充边界说明 / 补充适用条件 |
| 生产 skill 流程缺陷 | 对应 skill 的 SKILL.md | 补充流程步骤 / 检查项 / 顺序修正 |

### 第四步：验证

修改完源头后，运行一次**根因验证**：

- 如果再出现同一问题 → 根因定位有误，回到第二步重新追溯
- 如果不再出现 → 确认修好
- 如果改用别的 skill 时出现类似问题 → 根因可能是通用规则缺失，考虑更新 `knowledge-base-collab` 或 CLAUDE.md

## 常见根因对照表

| 问题类型 | 最可能的根因 | 首次修正位置 |
|---------|------------|------------|
| 选题不清 | 选题卡不完整 | `06-选题库/cards/` 对应卡片 |
| 判例调用错误 | 判例/证据边界不足 | `02-Areas/判例库/` 对应判例 |
| 口吻不对 | 个人IP约束不准 | `02-Areas/知识卡片/个人IP/` |
| 结构像文章不像视频 | 平台 skill 规则缺失 | `short-video-script/SKILL.md` |
| 法律边界说满 | 判例/证据边界不足 + content-brief 缺字段 | 判例库 + content-brief.template |
| 缺少用户痛点 | 知识卡片不足 | `02-Areas/知识卡片/创作/` 或 `个人IP/` |
| 平台适配失败 | 平台 skill 规则缺失 | 对应平台 skill |
| 同一主题反复出问题 | 主题地图不清晰 | `02-Areas/主题聚合/` 对应主题地图 |
| content-brief 频繁缺材料 | content-brief 缺字段 | `.claude/drafts/content-brief.template.json` |
| 生产流程遗漏步骤 | 生产 skill 流程缺陷 | 对应 SKILL.md |
| 多次不同选题同类错误 | 系统性问题，上游资产整体不足 | 知识卡片/主题地图/选题库 整体评估 |

## 复盘建议更新报告

每次完成根因修复后，生成一份简短的建议更新报告：

```
【反馈回写报告】
失败任务：XXX
根因分类：[根因分类名]
根因文件：[文件路径]
修正动作：[做了什么]
上游影响：修正后，[X]个相关选题也会受益
下次类似任务启动时：应自动加载修正后的源文件
```

> 这份报告可在下次启动同类任务时自动读取，避免重复踩坑。
