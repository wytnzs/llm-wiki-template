# Visual Routing

Use this reference to decide which visual system to apply.

## Default Routing Table

| Content Type | Platform | Route | Visual Behavior |
|---|---|---|---|
| 深度观点 | WeChat | Deep Editorial | low color, strong hierarchy, quote blocks sparingly |
| 案例拆解 | WeChat | Deep Editorial | evidence-first, numbered logic, careful claim boundaries |
| 教程指南 | WeChat | Dense Tutorial | step labels, cards, screenshots, code/prompt blocks |
| 工具盘点 | WeChat/Xiaohongshu | Dense Tutorial | comparison cards, rating/selection logic, clear next step |
| 方法论解释 | WeChat/Xiaohongshu | Material Explainer | one central diagram, 3-5 labels, social-card extraction |
| 数据/图表 | WeChat/Xiaohongshu | Data-first Material | preserve exact data, redesign composition |
| 个人创业复盘 | WeChat | Olive Editorial | internal memo feel, calm reflection, summary block |
| 小红书封面 | Xiaohongshu | Material Explainer | one hook, one central image, one conclusion |

## One Mother Style, Three Variants

Do not let users freely pick from too many themes in daily production. Use:

1. Mother style: restrained knowledge editorial.
2. Variant A: deep editorial.
3. Variant B: dense tutorial.
4. Variant C: material explainer.

This keeps the account recognizable while adapting to content type.

## WeChat Rules

WeChat should optimize for reading endurance:

- Use article-level structure: intro, selected contents, numbered sections, conclusion.
- Use one theme per article.
- Use underlines and light labels for scanning.
- Use images as pauses or explainers, not random decoration.
- Keep color anchored; do not turn every paragraph into a card.

## Xiaohongshu Rules

Xiaohongshu should optimize for instant comprehension:

- Each card should express one idea.
- Cover should contain tension, not a vague topic.
- Use a central image when the idea is abstract.
- Use short labels in visuals and keep full explanation outside the image.
- Do not shrink a dense diagram into an unreadable card.

## Style Decision Output

Every `style-brief` should state:

- selected route
- why this route fits the content
- rejected routes and why
- required visual assets
- platform-specific risk
- final QA criteria
