# Claude Code 协作规则

Claude Code 是本知识库的主要执行者，负责在用户授权范围内完成需求判断、直接执行、检查和复盘，而不是只给操作建议。

开始任务前先阅读根目录 `README.md`。README 是知识库目的、实际目录职责和已启用模块的唯一说明入口；不要在本文件重复维护目录树和项目清单。当前磁盘文件与历史对话冲突时，以当前磁盘文件和 README 为准。

## 启动顺序

1. 读取根目录 `README.md`，确认内容角色、实际路径、已启用模块和安全边界；
2. 查看工作区已有修改，识别用户或其他执行者正在进行的工作，不回滚、不覆盖、不全量暂存；
3. 判断任务属于输入、项目、资产、来源、系统维护还是内容生产；
4. 根据下表调用匹配的 Skill，只加载完成当前任务所需的文件；
5. 批量迁移、删除、覆盖、改变关键判断或公开发布前，先说明范围、风险和验收方式。

`AGENTS.md` 是 Codex 的身份与协作入口，不替代本文件。两者共享的知识库结构和业务规则统一从 README 与 `.claude/skills/` 读取。

## 任务路由

| 用户要做什么 | Claude Code 使用的入口 |
|---|---|
| 记录、整理、提炼、查询或判断内容角色 | `knowledge-base-collab` |
| 处理 `00-Inbox/` | `inbox-processing` |
| 创建、更新或审查主题页与主题关系 | `theme-map-maintenance` |
| 体检知识库、更新地图或核对动态统计 | `knowledge-map-update` |
| 录入或维护拒赔判例 | `case-law-entry` |
| 挖掘候选选题或保存已确认选题 | `topic-selection` / `topic-library` |
| 生产公众号、小红书或短视频内容 | 先用 `content-briefing`，再进入对应平台 Skill |
| 输出失败、用户否定结果或发现系统性问题 | `feedback-writeback` |
| 执行周期维护 | 读取 `05-Skills/知识库定期维护指令.md`，再串联相应 Skill |

## 基本流程

```text
输入 -> 整理 -> 连接 -> 使用 -> 检查 -> 复盘
```

## 执行规则

1. 先判断内容角色：输入、项目、资产或来源，再按 README 中的实际映射处理；来源再区分外部资料、个人输入和系统历史，04不再作为归档目标。
2. 保留用户原始表达；不是所有输入都值得立即制成知识卡片。
3. 只有可跨项目复用并帮助判断、决策或行动的内容才进入资产层。
4. 区分用户判断、外部资料和判例事实，记录来源与适用边界。
5. 内容生产前读取当前项目状态、目标受众、证据和表达约束，不直接从单张卡片跳到成稿。
6. 输出失败时追溯到输入、知识、项目定义、模板或 Skill，并把修正写回源头。
7. 正式知识文件至少包含 `title` 和 `type`；关键入口优先补充 `description`。

## 完成标准

1. 修改范围与用户目标一致，原始输入和依据仍可追溯；
2. 涉及法律、医疗、合同、判例或客户信息时，区分事实、推断和待核实项；
3. 运行任务对应的检查脚本；脚本通过不能替代内容和边界复核；
4. 只有影响结构、规则、正式判断或能力的变化才更新 `知识库变更日志.md`；
5. 最终报告已完成、未处理、需要确认和验证结果，不用“已经执行”代替证据。

需要确认 Claude Code 接管是否完整时运行：

```text
python .claude/skills/knowledge-base-collab/scripts/check_claude_code_setup.py
```

## 已启用模块

- 知识库地图：已启用；按 `knowledge-map-update` 维护。
- 判例库：已启用；按 `case-law-entry` 录入和维护。
- 选题库：已启用；候选选题必须经用户确认后才进入正式选题库。

## 禁止事项

- 不静默删除、覆盖或批量迁移；
- 不改写 `00-Inbox/` 和 `03-Resources/` 中的原始材料；
- 不把 Skill 变成业务资料库；
- 不把个案结论写成普遍承诺；
- 不上传真实判例、客户资料、个人知识和内容成品；
- 不自动提交、推送或公开发布；用户明确要求发布时，仍需精确筛选公开范围；
- 不把 Claude Code 改写成 Agent、Codex 或其他工具名称。
