# Claude Code 初始化规则

本文档供 Claude Code 执行。目标是建立一套足够简单、能持续使用的知识库，而不是机械复制目录。

## 一、先了解用户

在创建或修改文件前，一次性询问：

1. 新建知识库，还是整理已有知识库；
2. 主要输入是什么；
3. 主要用来做什么；
4. 希望得到什么输出；
5. 是否采用推荐目录，还是保留现有目录；
6. 是否需要知识库地图、案例库、选题库。

未得到明确回答时，不安装可选模块。选题库默认不安装。

## 二、建立最小核心

新建知识库时，可使用以下推荐结构：

```text
00-Inbox/       输入
01-Projects/    当前工作
02-Areas/       长期资产
  主题中心/     自然语言索引和主题页
03-Resources/   统一来源：外部资料、个人输入和系统历史
.claude/skills/ Claude Code 执行流程
```

已有知识库不强制改名。应先识别现有目录分别承担什么职责，再把对应关系写入根目录 `README.md`。归档不是必需的顶层目录；已处理但仍需追溯的材料保留在来源层，用状态、标签和连接表达处理进度。

新建知识库时创建一个主题中心，不再分别创建“主题聚合”“关系导航”和主题索引。主题中心 `README.md` 负责自然语言入口，`主题页/` 保存每个持续问题的当前综合判断。

## 三、生成协作入口

### README.md

必须生成，内容保持简短：

- 知识库目的；
- 主要区域及职责；
- 基本工作流；
- 常用指令；
- 已启用的可选模块。

### CLAUDE.md

从 `templates/CLAUDE.template.md` 生成，写入实际目录映射和已启用模块。不要把 Claude Code 改写成 Agent 或 Codex。

### AI 待确认与变更日志

- 从模板生成 `00-Inbox/AI待确认.md`，承接不能自动写入正式知识的来源缺口、冲突、关系建议和关键判断变更；
- 从模板生成根目录 `知识库变更日志.md`，只追加影响结构、规则、正式判断或能力的重要变化；
- 两者是轻量治理入口，不是新的知识层级。

## 四、安装 Skill

始终安装：

- `knowledge-base-collab`
- `feedback-writeback`
- `theme-map-maintenance`

按需安装：

- 知识库地图：`modules/knowledge-map/`
- 案例库：`modules/case-library/`
- 选题库：`modules/topic-library/`

没有安装某模块时，其他 Skill 必须跳过对应路径和检查项，不能报错或擅自创建目录。

## 五、最小文件规范

知识文件至少包含：

```yaml
---
title: 文件标题
type: 文件类型
---
```

推荐增加：`description`、`status`、`tags`、`created`、`updated`、`source`。

知识资产日常只优先维护 `source`、`related_theme_maps` 和 `related_projects`。主题关系必须直接帮助回答主题页的核心问题；只是同领域或关键词相同时只保留标签。

`SKILL.md` 使用 Claude Code 自身的 `name` 和 `description` 字段，不套用知识文件模板。

## 六、保护已有数据

- 不读取任务无关的私人文件；
- 不上传用户资料；
- 不因为不符合推荐目录就批量移动；
- 删除、覆盖和大规模迁移前先报告影响；
- 不引入数据库、RAG 或 MCP，除非用户另行要求；
- 原始资料保持只读；案例、法律和医疗关键结论先进入 AI 待确认队列。

## 七、关系与检索

- 标签用于稳定分类，不表达推理关系；
- 双向链接只连接两份存在明确语义关系的具体内容；
- 主题页围绕一个持续问题保存当前判断、证据、方法、反例和边界；
- 聚合是围绕问题筛选资料的动作，不单独创建目录或知识类型；
- 知识库地图只记录动态状态，不替代主题页或检索入口；
- 日常优先维护 `source`、`related_theme_maps` 和真实发生过的 `related_projects`；
- 不为消灭孤立页面、增加链接数量或填满字段制造关系。

新增正式关系前必须确认：关系能否一句话说清、沿过去是否有用、能否回到可靠依据。详细规则见 `docs/知识关系与索引规则.md`。

## 八、AI 修改权限

| 内容 | 默认权限 |
|---|---|
| 原始输入与03来源材料 | 读取、分类建议、建立来源；不改写正文 |
| 知识卡片和主题页 | 可以起草；涉及用户判断、关键证据或普遍结论时先请求确认 |
| 案例事实、合同、病历、法律或医疗结论 | 只提出建议，确认后写入 |
| 索引、状态统计、断链检查和变更日志 | 可按既定规则维护 |
| 删除、覆盖、批量迁移、提交、推送和对外发送 | 必须获得对应授权 |

## 九、协作和发布边界

- 不让两个 Agent 同时修改同一成品；
- 工作区已有修改默认属于用户或其他协作者，不回滚、不覆盖；
- 公开仓库只发布规则、模板、示例、Skill 和通用脚本；
- 真实卡片、案例、客户资料、项目材料、内容成品、凭证和个人配置不进入公开提交；
- 推送前逐项暂存，并运行 `scripts/check-public-safety.ps1`；工作区混杂时不得使用 `git add -A`。

完整边界见 `docs/知识库运作系统与协作边界.md`。
