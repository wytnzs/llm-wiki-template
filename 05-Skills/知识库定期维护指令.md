# 知识库定期维护指令

> 这是 Claude Code、Codex 和人工维护共用的运行单。它负责发现、分流和记录问题，不以“全部变绿”为目标，不用猜测来源、虚假双链或批量改写原始资料消灭告警。

## 一、运行频率

| 频率 | 目的 | 默认动作 |
|---|---|---|
| 每周 | 发现新故障 | 只读审计、新增文件抽查、Inbox 时效检查 |
| 每两周 | 完成一次维护闭环 | 来源复核、主题证据抽查、Inbox 路由、地图和日志更新 |
| 每月 | 检查系统是否真的有用 | 抽查检索下钻、项目调用、长期未用资产和公开边界 |

出现批量建链、目录调整、模板升级、Skill 改动或大量资料导入时，不等固定日期，立即执行一次完整检查。

## 二、执行前边界

1. 先读根目录 `README.md`、本指令和相应 Skill；Claude Code 读取 `.claude/CLAUDE.md`，Codex 读取 `AGENTS.md`，不要求同时加载两个身份入口；
2. 查看工作区现有修改，不回滚、不覆盖用户或其他 Agent 的改动；
3. 不让两个 Agent 同时修改同一成品；
4. `00-Inbox/`、`03-Resources/` 中的原始材料只读，除非有可靠原文可修复明确的收容错误；`04-Archive/` 已退役，只检查是否仍仅保留说明文件；
5. 判例、客户、合同、医疗和个人资料不自动对外发送；
6. 定期维护不自动提交或推送 GitHub。公开发布必须单独筛选规则、模板、示例和工具，并再次确认范围。

## 三、每周快速检查

按顺序执行：

1. 运行只读审计：

   ```text
   python .claude/skills/knowledge-map-update/scripts/audit_knowledge_base.py --details
   ```

2. 检查最近一周新增或修改的正式文件是否包含 `title`、`type`，标签和关系字段是否为正确列表；
3. 检查是否出现乱码、真实断链、同名歧义、旧路径和主题双向不一致；
4. 检查 Inbox 是否存在超过两周未路由的输入；
5. 新问题写入 `00-Inbox/AI待确认.md`，但不重复登记已拒绝、延期或已处理事项；
6. 如果没有结构变化，不为了“看起来维护过”改写地图和日志。

## 四、每两周完整维护

### 1. 来源与状态

- 对缺少 `source` 的知识卡片回查原始材料、项目记录和用户确认；
- 找不到可靠来源时保持空缺，并将 `status`、`evidence_status` 和正文提示调整到与证据强度一致；
- 具体数字、比例、市场案例和法律/医疗结论没有原始依据时，不进入稳定结论或主题强证据。

### 2. 主题中心

- 每次抽查 1 至 2 张近期实际调用的主题页；
- 核心判断必须回答 `core_question`，并区分事实、工作判断和待验证项；
- “证据来源”优先使用具体卡片、判例、原始文件、章节或项目记录，目录只能作为导航；
- 新增或移除 `related_cards` 时，同步卡片的 `related_theme_maps`；
- 普通同领域内容只保留标签，不为增加链接数量建立关系；
- 主题页的内容方向不是正式选题，用户确认后才能进入 `06-选题库/`。

### 3. Inbox

- 按 `inbox-processing` Skill 分批处理，每批不超过 10 条；
- 先输出“保留输入 / 进入项目 / 提炼资产 / 进入来源层（外部资料、个人输入或系统历史）/ 待清理隔离”的路由建议；
- 未获得确认时不移动、不覆盖、不删除原文件；
- 图片与文本属于同一输入时作为一组处理，不重复计算资产。

### 4. 地图与日志

- 重新扫描后更新 `知识库地图.md` 的规模、健康问题和 3 至 5 项下一步；
- 已解决事项从健康问题移出，并在 `00-Inbox/AI待确认.md` 写明处理结果；
- 只有影响结构、规则、正式判断或能力的变化才追加到 `知识库变更日志.md`；
- 地图写当前状态，日志写重要变化，README 只解释系统，三者不互相复制。

## 五、每月有效性抽查

从最近真实任务中选择 3 个问题，按以下路径实际下钻：

```text
自然语言问题
-> 主题中心
-> 主题页
-> 知识卡片或判例
-> 原始依据
-> 输出或项目使用
```

每个问题记录：是否命中正确主题、是否能在 1 至 2 跳内找到可用内容、是否能回到来源、是否出现明显误连、最终是否帮助了判断或行动。链接数量不作为质量指标。

再抽查：

- 近期项目实际调用了哪些卡片；
- 哪些资产长期没有被查询、项目或内容使用；
- 哪些主题判断已经被新证据推翻或需要缩小边界；
- 哪些 Skill 失败应回写上游，而不是只修改一次结果。

## 六、可直接交给 Claude Code 的循环提示词

```text
请执行一次知识库定期维护。先读取根目录 README.md、.claude/CLAUDE.md、05-Skills/知识库定期维护指令.md，以及任务匹配的 Skill。先查看工作区已有修改，不回滚或覆盖他人改动。

依次完成：
1. 运行全库只读审计；
2. 复核新增正式文件的 YAML、标签、双链和来源；
3. 检查缺来源卡片与 evidence_status 是否一致，不猜来源；
4. 抽查近期使用的主题页，把模糊证据收敛到具体文件并保持双向一致；
5. 只列出最多 10 条 Inbox 路由建议，未经确认不移动、不删除；
6. 只有状态确实发生变化时，才更新 AI待确认、知识库地图和重要变更日志；没有变化时保持文件不动；
7. 再运行审计、元数据干跑和 git diff --check。

请输出：本次发现、已安全修复、需要我确认、未处理原因和验收结果。不要自动提交或推送 GitHub，不要上传判例、客户、个人知识和内容成品。
```

如果使用 Claude Code 的循环或定时功能，把上面整段作为任务正文；频率建议每两周一次。循环只负责检查和安全修复，遇到隐私、来源冲突、主题核心判断变化、文件移动、删除、提交或推送时暂停等待确认。

## 七、完成标准

- 审计结果与实际文件一致，没有为归零制造假数据；
- 原始输入仍可追溯，关键结论能回到依据；
- 主题关系明确、有用、可追溯且双向一致；
- 待确认、地图和日志的状态一致；
- 所有未处理事项都说明原因、影响和下一步；
- 最终执行：

  ```text
  python .claude/skills/knowledge-map-update/scripts/audit_knowledge_base.py --details
  python .claude/skills/knowledge-base-collab/scripts/enrich_metadata.py
  git diff --check
  ```
