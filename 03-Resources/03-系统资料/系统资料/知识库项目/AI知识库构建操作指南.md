# AI 知识库构建操作指南

## 一、这套系统是什么

这套系统不是普通笔记库，也不是单纯的 AI 写作模板。

它是一套基于 **Obsidian + Markdown + Claude Code + Skill + YAML 元数据** 的轻量级知识资产生产系统。

它的目标是：

> 把碎片信息、个人经验、行业资料和日常思考，持续加工成可以复用、可以输出、可以决策、可以变现的知识资产。

普通知识库解决的是：

```text
我把资料放在哪里？
```

这套系统解决的是：

```text
资料如何变成判断？
经验如何变成方法？
方法如何变成内容？
内容如何反哺系统？
Claude Code 如何长期理解我？
```

最终形成的生产链路是：

```text
输入 → 提炼 → 连接 → 输出 → 复盘
```

对应到目录和流程上，就是：

```text
00-Inbox 原始输入
→ 02-Areas 知识资产
→ 02-Areas/主题聚合 长期记忆
→ 06-选题库 内容入口
→ content-brief 内容交接单
→ 公众号 / 小红书 / 短视频 / 咨询 / 决策
→ 复盘回写 skill 和知识库
```

## 二、为什么不是普通知识库

普通知识库的逻辑通常是：

```text
收集 → 分类 → 保存 → 搜索
```

这套系统的逻辑是：

```text
输入 → 提炼 → 连接 → 输出 → 复盘
```

区别在于：

| 普通知识库 | 本系统 |
|-----------|--------|
| 关心存了什么 | 关心能产出什么 |
| 以文件夹为中心 | 以知识资产为中心 |
| AI 临时理解资料 | Claude Code 长期按规则维护 |
| 内容输出靠临时提示词 | 内容输出靠卡片、主题、选题、brief 和 skill |
| 写完一次就结束 | 写完后复盘，反哺系统 |

所以，这套系统的核心不是“更会分类”，而是：

> 把一个人的碎片经验、专业资料和日常思考，持续加工成可输出、可复用、可售卖的知识资产。

## 三、技术基础

这套系统采用轻量组合：

```text
Obsidian + Markdown + Claude Code + Skill + YAML 元数据
```

### 1. Obsidian：负责本地知识管理

Obsidian 的优势是本地文件、Markdown 格式、双向链接、可长期保存。

每一条笔记，本质上都是一个 `.md` 文件。以后即使不用 Obsidian，这些文件仍然可以被其他工具读取。

### 2. Markdown：负责内容承载

Markdown 的优势是：

```text
人能读
AI 能读
GitHub 能展示
Obsidian 能管理
Claude Code 能修改
长期迁移成本低
```

对个人和小团队来说，Markdown 已经足够支撑第一版知识资产系统。

### 3. Claude Code：负责执行

Claude Code 不是简单的写作工具，而是这套知识库的操作员。

它负责：

```text
整理 Inbox
提炼知识卡片
更新主题地图
生成选题
生成 content-brief
写公众号 / 小红书 / 短视频
检查乱码、断链、旧路径
复盘内容质量
更新 skill 规则
```

### 4. Skill：负责流程标准化

如果每次都临时告诉 Claude Code 怎么做，系统就不稳定。

Skill 的作用是把常用流程写成固定规则，比如：

```text
如何提炼知识卡片
如何维护主题地图
如何生成选题
如何写短视频脚本
如何复盘失败内容
如何更新知识地图
```

### 5. YAML 元数据：负责结构化识别

知识库里的文件不能只靠标题和正文判断身份，还需要元数据。

例如知识卡片：

```yaml
---
type: project_note
asset_type: concept
status: seed
source: personal
related_topics: []
updated: 2026-07-09
---
```

这样 Claude Code 能知道：这是知识卡片，不是项目文档，也不是草稿。

这吸收了 Gemini OKF 的思想：用普通 Markdown 文件，加结构化元数据，让知识既能给人看，也能给 AI 用。

## 四、适合哪些人

这套系统尤其适合需要长期输出、长期积累、长期复用知识的人。

| 人群 | 典型需求 |
|------|----------|
| 专业服务者 | 把案例、法规、经验变成判断框架和客户解释 |
| 自媒体创作者 | 把观点、故事、案例变成选题、脚本、文章 |
| 咨询顾问 / 培训老师 | 把项目经验沉淀成课程、方法论和交付物 |
| 创业者 / 管理者 | 把信息、反馈、会议变成决策和复盘 |
| 学生 / 研究者 | 把资料、论文、书籍变成概念、论证和写作 |
| 销售 / 私域运营 | 把客户信息、痛点、异议变成方案和话术 |

关键不是他们使用什么目录，而是：

> 他们要把什么变成什么。

## 六、最快使用方法：把 GitHub 地址发给 Claude Code

这套系统未来不建议让用户手动一步一步搭。

更好的方式是：

```text
文章负责讲明白
GitHub 负责交付模板
Claude Code 负责自动初始化
```

用户只需要做 5 步。

### 第一步：打开 GitHub 模板仓库

复制模板仓库地址：

```text
https://github.com/你的用户名/llm-wiki-template
```

正式发布后，将上面的地址替换成真实 GitHub 地址。

### 第二步：把初始化指令发给 Claude Code

把下面这段话发给 Claude Code：

```text
请打开这个 GitHub 仓库：
https://github.com/你的用户名/llm-wiki-template

请先阅读 README.md、QUICKSTART.md 和 AGENT_SETUP.md，
然后帮我在本地初始化一套 AI 知识资产生产系统。

请先问我属于哪类人群：
1. 专业服务型
2. 内容生产型
3. 企业管理型
4. 学习研究型
5. 销售私域型

然后根据我的选择：
- 创建知识库目录；
- 复制模板；
- 安装 Claude Code skills；
- 创建 CLAUDE.md；
- 生成我的使用说明；
- 告诉我第一步应该往 00-Inbox 放什么。

注意：
- 不要读取、上传或暴露我的隐私资料；
- 不要上传我的本地知识库；
- 不要删除我的文件；
- 所有涉及删除、覆盖、上传的操作，都必须先让我确认。
```

### 第三步：选择你的知识库类型

Claude Code 会先询问你属于哪类人群。

你可以直接回答：

```text
我是专业服务型，主要做保险理赔内容和咨询。
```

或：

```text
我是内容生产型，主要做短视频、小红书和公众号。
```

### 第四步：把资料放进 00-Inbox

初始化完成后，不需要急着分类。

先把这些资料放进 `00-Inbox`：

```text
随手想法
读书笔记
课程笔记
网页剪藏
案例材料
语音转文字
工作复盘
客户常见问题
```

### 第五步：让 Claude Code 开始整理

发给 Claude Code：

```text
请整理 00-Inbox，把里面的内容分类处理：

1. 能沉淀的，提炼成知识卡片；
2. 能形成主题的，更新主题地图；
3. 能做内容的，生成选题卡；
4. 暂时不成熟的，放入待挖掘方向；
5. 无用内容先列出建议，不要直接删除。

完成后请输出整理报告：
- 新增了哪些知识卡片；
- 更新了哪些主题地图；
- 生成了哪些选题；
- 哪些内容还需要我补充信息；
- 下一步建议我做什么。
```

这就是用户最快上手路径。

## 七、初始化后的日常使用流程

系统搭好后，日常使用只需要围绕 6 个动作。

### 1. 记录

把所有新内容先放进 `00-Inbox`。

可以发：

```text
记一下：我今天发现，普通人做知识库失败，不是因为不会分类，而是因为没有输出闭环。
```

### 2. 整理

定期发：

```text
请整理收件箱，把 00-Inbox 里的内容提炼成知识卡片、主题地图或选题卡。
```

### 3. 聚合

当某个方向积累多了，发：

```text
请更新主题地图，看看这些卡片能不能形成一个稳定判断。
```

### 4. 选题

想做内容时，发：

```text
请基于主题地图和知识卡片，生成 10 个可用于短视频/公众号/小红书的选题。
```

### 5. 出稿

选定一个选题后，先生成 content-brief：

```text
请基于这个选题生成 content-brief，明确目标读者、核心观点、调用卡片、证据来源、表达边界和输出计划。
```

然后再写稿：

```text
请读取 content-brief，生成一条 300-500 字的短视频口播脚本。
```

### 6. 复盘

内容完成后，发：

```text
请对这次内容生产进行复盘，并回写知识库：
- 哪些观点可以沉淀为知识卡片；
- 哪些选题判断需要修改；
- 哪些主题地图需要更新；
- 哪些 skill 规则需要补充；
- 下次同类内容怎么做得更好。
```

## 八、最小闭环

如果今天就要开始，不要想太复杂，只跑最小闭环：

```text
1. 把所有新材料先放进 00-Inbox
2. 每周清理一次 Inbox
3. 从 Inbox 提炼 3 张知识卡片
4. 把相关卡片连接到一个主题地图
5. 从主题地图生成 3 个选题
6. 选一个选题生成 content-brief
7. 输出一篇文章或一条短视频脚本
8. 做复盘，并回写知识库
```

这条线跑起来以后，知识库就会开始产生复利。

## 九、GitHub 仓库应该承担什么

GitHub 仓库不是你的真实知识库。

它应该是公开交付包，包含：

```text
README.md                 产品首页
QUICKSTART.md             5 分钟上手
AGENT_SETUP.md            给 Claude Code 看的自动初始化说明
docs/                     完整教程和方法论
templates/                知识卡片、主题地图、选题卡、brief 模板
examples/                 脱敏 demo 链路
.claude/skills/           公开版通用 skill
scripts/                  初始化和安全检查脚本
```

不要上传：

```text
真实 00-Inbox
真实判例库
真实知识卡片
真实自媒体内容
真实客户信息
真实个人 IP 细节
03-Resources 外部资料
.obsidian
.codex
settings.local.json
content-brief.json
API key 或 token
```

一句话：

> 文章负责转化，GitHub 负责交付，README 负责入口，AGENT_SETUP 负责自动化。

## 十、常见问题

### 1. 我还需要手动建目录吗？

正式使用 GitHub 模板后，不需要。

你只需要把 GitHub 地址发给 Claude Code，让它读取 `AGENT_SETUP.md` 自动初始化。

### 2. 我需要懂 GitHub 吗？

不需要深入懂。

你只要会复制仓库地址，并把地址发给 Claude Code。

### 3. 这是不是 RAG？

不是。

这套系统暂时不依赖数据库、向量检索或复杂 RAG。它先用 Markdown、YAML、目录、双向链接和 skill，把知识资产生产流程跑通。

未来可以接入更复杂能力，但第一阶段重点是低成本、可复制、可执行。

### 4. 我的隐私资料会不会上传？

不会，前提是严格遵守公开仓库规则。

公开 GitHub 仓库只放模板、方法论、示例和通用 skill，不放真实判例、真实客户、真实项目、真实知识资产。

### 5. 为什么还要保留手动指令？

因为有些用户不一定用 GitHub 自动初始化，或者初始化后需要手动微调。

所以本文附录保留完整手动指令库，作为备用方案。

## 十一、附录：手动操作指令库

以下内容是备用指令。正常情况下，用户不需要逐条执行，而是由 GitHub 仓库里的 `AGENT_SETUP.md` 和初始化脚本自动完成。

### 附录 1：手动创建基础目录

```text
请帮我在当前目录创建一个 AI 知识库基础结构。

目录如下：

00-Inbox        原始输入，放还没处理的想法、摘录、网页剪藏、语音转文字

02-Areas        长期知识资产，包括知识卡片、主题地图、案例库、判断框架
03-Resources    外部参考资料，包括书籍、课程、PDF、文章、行业资料
04-Archive      归档，放已经完成或暂时不用的内容
05-Skills       给人看的流程、模板、检查清单
06-选题库        可转化为文章、短视频、小红书、课程的选题资产
.claude/skills  Claude Code 实际执行的技能说明

请为每个目录创建一个 README.md，说明这个目录的用途、放什么、不放什么。
不要创建无关文件，不要引入数据库。
```

### 附录 2：手动创建知识库总规则

```text
请在根目录创建或更新 CLAUDE.md，作为 Claude Code 操作这个知识库时必须遵守的总规则。

请写清楚：

1. 这个知识库的目标不是存资料，而是把资料加工成知识资产。
2. 目录职责：

   - 01-Projects 是当前目标；
   - 02-Areas 是长期资产；
   - 03-Resources 是外部资料；
   - 04-Archive 是归档；
   - 05-Skills 是给人看的流程模板；
   - .claude/skills 是 Claude Code 执行技能。
3. 所有新内容优先进入 Inbox，再加工成卡片、主题或选题。
4. 每次输出内容后，都要复盘并反哺知识库。
5. 不要把 Claude Code 改写成 Agent 或 Codex。
6. 不要把 skill 当资料库。
7. 不要把原始资料直接当成知识资产。
```

## 十二、结语

未来的知识库，不应该只是第二大脑。

它更应该是第二生产系统。

真正好的 AI 知识库，不是让你收藏更多，而是让你更快形成判断，更稳定输出内容，更持续积累资产。

所以，这套系统真正要交付的不是一堆文件夹，而是一条可以运行的生产线：

```text
输入 → 提炼 → 连接 → 输出 → 复盘
```

当这条线跑起来，知识就不再是散落的资料，而会逐渐变成可以持续调用、持续输出、持续变现的长期资产。
