# 三套知识库思想整合规划

## 一、文档目的

本文用于整理当前知识库系统的下一阶段建设方向，方便后续调阅、复盘，并交给 Claude Code 执行具体修改。

当前已经学习和参考的三套知识库构建思想包括：

| 来源 | 核心价值 | 在本知识库中的吸收方式 |
|------|----------|------------------------|
| LLM Wiki | 输入、资产、项目、输出、反馈的知识生产线 | 保留并强化当前目录结构和内容管线 |
| GBrain | 长期记忆、主题聚合、定期维护、健康检查 | 强化主题地图、反馈回写和健康检查机制 |
| Gemini OKF | Markdown + YAML + index + log 的通用格式 | 为卡片、主题、选题、项目建立标准元数据 |

本系统的目标不是做一个普通笔记库，而是做一套轻量级知识资产生产系统。

## 二、系统定位

建议将当前知识库定位为：

> 轻量级 LLM Wiki：用 Obsidian + Claude Code + Markdown，把碎片信息加工成可复用知识资产，并持续产出内容、课程、案例、产品。

一句话说明：

> 本系统不是帮人存资料，而是帮人把资料变成判断、方法、选题、内容和可交付产品。

核心生产链路：

```text
资料进入系统
→ 提炼成概念
→ 连接成判断
→ 组织成方法
→ 变成选题
→ 生成内容
→ 复盘反哺
```

## 三、三套思想如何合并

### 1. LLM Wiki：负责生产线

LLM Wiki 思想适合作为当前知识库的主结构：

```text
00-Inbox        原始输入
01-Projects     当前项目
02-Areas        长期资产
03-Resources    外部资料
04-Archive      归档
05-Skills       人看的流程和模板
06-选题库        选题资产
.claude/skills  Claude Code 实际执行技能
```

需要坚持的原则：

- 目录按行为目标分，不按资料类别分。
- 02-Areas 是资产层，不是资料仓库。
- 项目结束后，要把可复用经验沉淀回资产层。
- 内容输出后，要通过复盘反哺知识库和 skill。

### 2. GBrain：负责长期记忆和维护

GBrain 的价值不在于是否使用数据库，而在于它强调：

- 碎片信息要定期聚合成长期记忆；
- 检索前应先有主题级理解；
- 系统需要健康检查；
- 失败经验要回写到规则中。

在当前知识库中，对应落点是：

```text
02-Areas/主题聚合/
.claude/skills/feedback-writeback/
知识库健康检查清单
content-brief 调用记录
```

尤其要强化 `02-Areas/主题聚合/`。主题地图不能只是索引，而应该成为“某个主题当前版本的长期判断”。

建议每个主题地图统一包含：

```text
1. 当前核心判断
2. 关键概念
3. 证据来源
4. 已验证的方法
5. 适用边界
6. 反例和风险
7. 可转化选题
8. 最近更新记录
9. 待补问题
```

### 3. Gemini OKF：负责通用格式

OKF 的价值是让知识库更标准、更可迁移、更容易被 AI 理解。

当前不需要把整个知识库改成 OKF，但应吸收它的格式思想：

- Markdown 文件承载正文；
- YAML frontmatter 承载结构化元数据；
- index.md 说明目录；
- log.md 记录变化；
- 文件之间用 Markdown 链接表达关系。

优先标准化四类文件：

```text
知识卡片
主题地图
选题卡
项目进度
```

## 四、当前知识库状态判断

当前结构已经比较完整：

| 层级 | 当前目录 | 状态 |
|------|----------|------|
| 输入层 | 00-Inbox | 已建立 |
| 项目层 | 01-Projects | 已建立 |
| 资产层 | 02-Areas | 已建立，需继续强化 |
| 资源层 | 03-Resources | 已建立 |
| 归档层 | 04-Archive | 已建立 |
| 人类流程层 | 05-Skills | 需明确用途 |
| 选题层 | 06-选题库 | 已建立，数量偏少 |
| Claude Code 执行层 | .claude/skills | 已建立 |

当前优势：

- 已经形成输入、资产、项目、输出、反馈的基础闭环。
- 判例库、知识卡片、主题聚合已经分层。
- `content-brief` 可以作为内容生产的交接单。
- 已有内容生产、选题、反馈回写等 Claude Code skill。

当前短板：

- `06-选题库` 数量仍偏少，支撑持续内容生产不够。
- `02-Areas/主题聚合` 主题地图数量偏少，长期记忆层还不够厚。
- OKF-like YAML 元数据尚未完全统一。
- 缺少一条完整的产品化 demo 链路。
- 需要建立固定健康检查机制。

## 五、下一步优化计划

### 阶段一：冻结主架构

短期内不要继续大改目录，先稳定当前结构。

需要明确：

```text
05-Skills = 给人看的流程、模板、清单
.claude/skills = Claude Code 实际执行技能
```

避免出现两个问题：

- 把真实资料塞进 skill；
- 把流程规则散落到知识卡片里。

### 阶段二：引入 OKF-like 元数据

先从模板开始，不要求一次性改完全库。

知识卡片模板建议：

```yaml
---
type: project_note
asset_type: concept
status: seed
source: personal
related_topics: []
related_projects: []
updated: 2026-07-09
---
```

主题地图模板建议：

```yaml
---
type: project_note
status: active
core_question:
source_cards: []
source_cases: []
updated: 2026-07-09
---
```

选题卡模板建议：

```yaml
---
type: project_note
platform: [short_video, xhs, gzh]
source_cards: []
source_cases: []
content_stage: idea
updated: 2026-07-09
---
```

项目进度模板建议：

```yaml
---
type: project_note
status: active
current_goal:
next_action:
updated: 2026-07-09
---
```

### 阶段三：强化主题聚合

优先补强或新增以下主题地图：

```text
02-Areas/主题聚合/知识管理主题地图.md
02-Areas/主题聚合/内容生产主题地图.md
02-Areas/主题聚合/保险理赔主题地图.md
02-Areas/主题聚合/个人IP主题地图.md
02-Areas/主题聚合/知识库产品化主题地图.md
```

主题地图的作用不是收集链接，而是形成当前版本判断。

示例问题：

- 这个主题现在最核心的判断是什么？
- 哪些知识卡片支撑这个判断？
- 哪些判例或事实可以作为证据？
- 这个判断适用于什么场景？
- 它在哪些情况下会失效？
- 它可以转化成哪些选题？

### 阶段四：打通完整内容生产 demo

为产品化和直播售卖准备一条完整演示链路。

建议放置路径：

```text
01-Projects/知识库产品化/demo/
```

建议 demo 主题：

```text
普通人为什么做知识库总失败
```

建议产物：

```text
demo-inbox.md
demo-knowledge-card.md
demo-theme-map.md
demo-topic-card.md
demo-content-brief.json
demo-short-video-script.md
demo-review.md
```

这条链路用于证明系统不是“目录搭建”，而是完整的知识资产生产线。

### 阶段五：建立健康检查机制

建议每周运行一次知识库健康检查。

检查内容：

```text
1. 是否有乱码
2. 是否有旧路径
3. 是否有断链
4. Inbox 是否堆积
5. 知识卡片是否缺 YAML
6. 选题是否缺来源卡片
7. content-brief 是否缺调用记录
8. 成品内容是否缺复盘
9. skill 是否和当前目录结构一致
```

健康检查的目的不是追求完美，而是防止系统长期运行后变脏、变乱、变难用。

## 七、近期优先级

建议按以下顺序推进：

1. 先修正地图和核心说明文件，让 Claude Code 读库时不迷路。
2. 再统一模板元数据，让新增内容从一开始就标准化。
3. 然后补强主题地图，把长期记忆层做厚。
4. 最后做完整 demo 链路，用于直播展示和模板交付。

当前最重要的不是继续增加复杂度，而是让系统更可演示、更可复制、更可交付。
