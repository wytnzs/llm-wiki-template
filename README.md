# AI 知识资产生产系统

> 用 Markdown 保存可迁移的知识资产，用 Claude Code 完成整理、连接、调用和复盘。

本库服务两个核心目标：沉淀保险理赔与个人认知资产；把经过验证的资产用于项目、内容生产和产品化。目录用于区分内容角色，不要求每份资料都机械套用同一结构。

## 从这里开始

日常使用不需要先理解全部目录和关系，只按任务进入：

| 现在要做什么 | 入口 |
|---|---|
| 处理新输入 | `00-Inbox/` |
| 推进当前目标 | `01-Projects/` |
| 先了解某个问题的当前判断 | `02-Areas/主题中心/README.md` |
| 核对具体方法、判例或原始依据 | `02-Areas/知识卡片/`、`02-Areas/判例库/`、`03-Resources/` |
| 查看 AI 提出的待确认修改 | `00-Inbox/AI待确认.md` |
| 查看知识库规模、健康问题和下一步 | `知识库地图.md` |
| 查看最近发生的重要变化 | `知识库变更日志.md` |

## 统一术语

- **主题中心**：自然语言索引、主题入口和主题页的统一模块；
- **主题**：需要持续回答的一个问题；
- **聚合**：围绕问题寻找和筛选资料的动作，不单独形成目录或知识类型；
- **主题页**：保存围绕主题形成的当前判断、证据、方法、反例和边界；
- **知识库地图**：全库状态报告，不承担主题综合和内容索引。

目录只回答“内容现在承担什么角色”，不表达全部主题关系。一份知识只保留一个稳定文件位置，通过来源、主题页和项目关系进入不同使用场景。

## 基本闭环

```text
输入 -> 整理 -> 连接 -> 使用 -> 检查 -> 复盘
```

- 输入：保留原始表达和外部材料；
- 整理：判断内容继续保留为输入，还是进入项目、资产或统一来源层；
- 连接：用来源、适用边界、主题地图和必要链接建立上下文；
- 使用：为项目决策、理赔实务和内容生产提供依据；
- 检查：核对内容、来源、边界、链接和交付结果；
- 复盘：定位问题来自输入、知识、项目定义、模板还是 Skill，并写回源头。

来源材料的稳定流转为：

```text
00-Inbox 保留尚未路由的输入
-> 03-Resources（外部资料、个人原始输入和系统历史的统一来源层）
-> 02-Areas 提炼可复用知识和当前主题判断
-> 01-Projects 调用
-> 反馈写回 02-Areas 或对应 Skill
```

来源材料进入 `03-Resources/` 后原则上不改写正文；正式知识通过具体来源路径、链接、判例、页码或章节回到依据。资料是否已经提炼通过集合导读和使用关系表达，不再通过搬往另一目录表达。

## 实际目录

| 区域 | 职责 |
|---|---|
| `00-Inbox/` | 尚未处理的得到笔记、灵感、剪藏和临时材料 |
| `01-Projects/` | 当前要完成的项目目标、状态和工作材料 |
| `02-Areas/知识卡片/` | 可跨项目复用的概念、判断、方法和行动 |
| `02-Areas/判例库/` | 保险判例证据；按险种为主库，另有拒赔理由和疾病视图 |
| `02-Areas/主题中心/` | 统一检索入口；`主题页/` 保存围绕问题形成的当前判断、证据、方法和边界 |
| `02-Areas/自媒体运营/` | 已生产或可复用的公众号、小红书、短视频等内容资产 |
| `03-Resources/` | 统一来源层：外部书籍、课程、文章、行业资料、个人原始输入和系统历史 |
| `04-Archive/` | 已退役，仅保留迁移说明，不再接收新内容 |
| `05-Skills/` | 给人阅读的操作手册、模板和检查清单 |
| `.claude/skills/` | Claude Code 执行流程，不存放业务知识 |

当前项目以 `01-Projects/` 中实际存在的五个项目目录为准。自媒体成品暂保留在 `02-Areas/自媒体运营/`，作为可复用的已产出资产；具体内容任务仍应先在项目或 content brief 中说明目标、受众和当前状态。

## 最小关系规则

日常维护一份知识资产只先回答三个问题：

1. **从哪里来？** 使用 `source` 回到原始材料、判例、项目记录或用户确认；
2. **直接帮助回答哪个问题？** 确实能提供判断、证据、方法、反例或边界时，连接对应主题页；
3. **在哪里真实使用过？** 只有项目实际调用后才写入 `related_projects`。

`supports`、`contrasts`、`examples` 等语义关系按需使用，不是每张卡片的必填项。普通关系不打分，不为消除孤立页制造链接。

## AI 修改边界

| 内容 | AI 权限 |
|---|---|
| `00-Inbox/`、`03-Resources/` 中的原始材料 | 读取、分类建议和建立来源链接，不改写原文 |
| 判例事实、合同条款、病历和法律/医疗关键结论 | 只能提出修改建议，确认后写入 |
| 知识卡片和主题页 | 可以起草；涉及用户判断、关键证据或普遍性结论时先进入 `00-Inbox/AI待确认.md` |
| 索引、状态统计、断链检查和变更日志 | 可按既定规则自动维护 |

AI 待确认项被接受、拒绝或延期时记录处理结果，避免同一建议在后续会话中反复出现。

## 最小文件规范

正式知识文件至少包含：

```yaml
---
title: 文件标题
type: 文件类型
---
```

推荐按需要补充 `description`、`status`、`tags`、`source`、`created`、`updated`。新建或实际更新的正式知识卡片应优先补齐 `description` 和可核验的 `source`；不确定来源时进入待确认清单，不编造来源。`SKILL.md` 使用 Claude Code 的 `name` 和 `description`，不套用知识文件字段。

## 常用指令

```text
记一下：……
整理收件箱
提炼知识卡片
查一下：……
复盘这次结果
更新主题地图
更新知识地图
执行知识库体检
查看 AI 待确认
检查 Claude Code 接管
```

内容生产前先明确项目状态、受众、目标和证据来源；不能只给一个主题就直接产出。输出不好时，不只改稿，还要记录根因并修正对应知识、项目定义、模板或 Skill。

## 安全边界

- 不把真实判例、客户信息、个人卡片和内容成品上传到公开模板仓库；
- 不静默删除、覆盖或批量迁移现有数据；
- 不把单案结果写成普遍承诺；
- 不把 Claude Code 文档改写成其他 Agent 的说明。
